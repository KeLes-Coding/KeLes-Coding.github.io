# 2. 分层模块设计

## 2.1 Router / Controller 层

对应代码：

- [router/AI.go](/I:/Study/26_shixi/golang/GoMind_org/router/AI.go)
- [controller/session/session.go](/I:/Study/26_shixi/golang/GoMind_org/controller/session/session.go)

这两层负责：

- 定义路由。
- 解析请求参数。
- 校验 `modelType` / `chatMode`。
- 给请求挂超时控制。
- 给流式接口设置 SSE 响应头。
- 把错误码转成 JSON 或 SSE 事件输出。

当前聊天相关接口如下：

- `GET /api/v1/AI/chat/sessions`：获取当前用户的会话列表。
- `GET /api/v1/AI/chat/session/:id`：获取某个会话的基础信息，包括当前绑定配置和聊天模式。
- `GET /api/v1/AI/chat/session-tree`：获取会话树结构，用于前端分组显示。
- `GET /api/v1/AI/chat/observability`：获取聊天链路观测指标快照。
- `POST /api/v1/AI/chat/send-new-session`：新建会话，并同步返回首轮回答。
- `POST /api/v1/AI/chat/send`：在已有会话中继续聊天，并同步返回回答。
- `POST /api/v1/AI/chat/history`：获取某个会话的历史消息。
- `POST /api/v1/AI/chat/stop`：停止当前流式回答。
- `POST /api/v1/AI/chat/resume-stream`：根据 `streamId + lastSeq` 恢复中断的流式回答。
- `POST /api/v1/AI/chat/send-stream-new-session`：新建会话，并立即开始流式输出首轮回答。
- `POST /api/v1/AI/chat/send-stream`：在已有会话中继续聊天，并以流式方式输出回答。

可以按用途分成三组：

1. **同步聊天**
   - `send-new-session`
   - `send`

2. **流式聊天**
   - `send-stream-new-session`
   - `send-stream`
   - `stop`
   - `resume-stream`

3. **会话查询**
   - `sessions`
   - `session/:id`
   - `session-tree`
   - `history`
   - `observability`

Controller 层 `ChatSend` 代码如下：

```go
func ChatSend(c *gin.Context) {

```

每一步的作用分别是：

- `ShouldBindJSON`：反序列化请求体。
- `validateModelType`：校验模型类型是否合法。
- `validateChatMode`：校验聊天模式是否合法。
- `buildChatTimeoutContext`：挂同步接口超时。
- `ChatSendWithControl`：进入 service 主链路。

## 2.3 Session Service 层

对应代码：

- [service/session/session.go](/I:/Study/26_shixi/golang/GoMind_org/service/session/session.go)
- [service/session/chat_controlled.go](/I:/Study/26_shixi/golang/GoMind_org/service/session/chat_controlled.go)

这一层负责：

- 组织同步聊天与流式聊天主流程。
- 串联 `AIHelper`、Redis、MySQL、通知。
- 推进 user message / assistant message。
- 推进 `session.version / persisted_version`。

核心函数与作用如下：

- `CreateSessionAndSendMessage`：新建会话并同步返回首轮回答。
- `ChatSend`：已有会话继续聊天并同步返回回答。
- `CreateStreamSessionOnly`：只创建流式会话，不启动模型生成。
- `CreateStreamSessionAndSendMessage`：新建会话并开始流式输出。
- `StreamMessageToExistingSessionWithControl`：流式主链路入口。
- `GetChatHistory`：读取会话历史。
- `ensureOwnedSession`：校验会话存在且归当前用户所有。
- `persistMessageSync`：同步写入一条正式消息。
- `persistSessionProgressWithPersistedVersion`：推进 session 的正式版本。
- `commitHelperHotState`：提交 Redis 热状态。
- `finalizeAssistantTurn`：assistant 终态统一收敛。

这一层把一轮对话拆成了几个提交点：

1. 先把 `user message` 写进 `AIHelper`，并同步提交 Redis 热状态。
2. 再把 `user message` 正式落 MySQL。
3. 然后调用模型生成 assistant。
4. 最后把 assistant 终态统一做“落库 + version 推进 + 热状态提交 + 通知”。

以用户消息进入上下文为例，对应代码如下：

```go
func appendUserMessageAndCommitHotState(ctx context.Context, helper *aihelper.AIHelper, userName string, userQuestion string) (*model.Message, code.Code) {
    if helper == nil {
        return nil, code.CodeInvalidParams
    }
    message := helper.AddMessageWithStatus(userQuestion, userName, true, false, model.MessageStatusCompleted)
    if code_ := commitHelperHotState(ctx, helper); code_ != code.CodeSuccess {
        return nil, code_
    }
    return message, code.CodeSuccess
}
```

这段代码的含义是：

- `helper.AddMessageWithStatus(...)` 先把用户问题加到当前运行时上下文里。
- 这条消息的 `SessionVersion` 会自动挂到 `helper.GetVersion() + 1`，表示它属于“下一轮推进”。
- 紧接着立刻 `commitHelperHotState`，这样即使请求后面断开，其他实例也能从 Redis 看到“这轮用户消息已经进入上下文”。

正式写 MySQL 时，主链路走的是同步落库：

```go
func persistMessageSync(ctx context.Context, message *model.Message) code.Code {
    if message == nil {
        return code.CodeInvalidParams
    }
    if code_ := ensureSessionWriteOwnership(ctx, message.SessionID); code_ != code.CodeSuccess {
        return code_
    }
    if _, err := messageDAO.CreateMessage(message); err != nil {
        observability.RecordDBPersistFail()
        log.Println("persistMessageSync CreateMessage error:", err)
        return code.CodeServerBusy
    }
    return code.CodeSuccess
}
```

这里要注意两个点：

- 先做 `ensureSessionWriteOwnership`，避免旧 owner 在 lease 已失效后继续尾写。
- 主消息落库不再依赖异步 outbox/MQ，采用同步写入，保证 `user/assistant` 核心消息是正式历史的一部分。

assistant 终态的统一收敛逻辑如下：

```go
func finalizeAssistantTurn(ctx context.Context, sess *model.Session, sessionID string, helper *aihelper.AIHelper, message *model.Message) code.Code {
    if sess == nil || helper == nil || message == nil {
        return code.CodeInvalidParams
    }
    if code_ := persistMessageSync(ctx, message); code_ != code.CodeSuccess {
        savePendingPersistHotStateBestEffort(ctx, helper)
        return code_
    }
    if code_ := persistSessionProgressWithPersistedVersion(ctx, sessionID, helper); code_ != code.CodeSuccess {
        savePendingPersistHotStateBestEffort(ctx, helper)
        return code_
    }
    if code_ := commitHelperHotState(ctx, helper); code_ != code.CodeSuccess {
        enqueueHotStateRebuildRepairBestEffort(helper)
        return code_
    }
    publishAssistantReadyNotificationBestEffort(ctx, sess, message)
    return code.CodeSuccess
}
```

这段代码可以理解成 assistant 终态的四步提交：

- `persistMessageSync`：assistant 正式入库。
- `persistSessionProgressWithPersistedVersion`：把 `session.version` 和 `session.persisted_version` 推到下一轮。
- `commitHelperHotState`：把 Redis 热状态也推进到同一轮。
- `publishAssistantReadyNotificationBestEffort`：通知作为旁路能力，失败不回滚主链路。

这一层的作用是“把一轮对话从运行时状态收敛成正式状态”。

## 2.4 执行保护层

对应代码：

- [service/session/guard.go](/I:/Study/26_shixi/golang/GoMind_org/service/session/guard.go)

这一层负责：

- 限流。
- 单实例内同 session 串行执行。
- 多实例下 owner 选举与串行推进。

核心函数：

- `allowChatRateLimit`
- `withSessionExecutionGuard`
- `withSessionOwnerGuard`
- `sessionOwnerGuardFromContext`

这一层最关键的是 `withSessionExecutionGuard`，因为它决定了同一个 `session` 能不能被安全串行推进。

核心代码如下：

```go
func withSessionExecutionGuard(ctx context.Context, sessionID string, fn func(context.Context) codeExecutorResult) codeExecutorResult {
    localLock := globalSessionLocalLockManager.getLock(sessionID)
    waitStart := time.Now()
    localLock.Lock()
    defer localLock.Unlock()

    execCtx, cancel := context.WithCancel(ctx)
    defer cancel()

    if myredis.IsAvailable() && rt.IsOwnerEligible() {
        ownerLease, ownerState, err := myredis.AcquireOrRefreshSessionOwnerLease(execCtx, sessionID, currentInstanceID)
        if err == nil && ownerLease != nil {
            execCtx = withSessionOwnerGuard(execCtx, &sessionOwnerGuard{
                SessionID:  sessionID,
                OwnerID:    ownerLease.OwnerID,
                FenceToken: ownerLease.FenceToken,
            })
            ownerLease.StartSessionOwnerWatchdog(execCtx, func() { cancel() })
        }
    }

    distributedLock, err := myredis.AcquireSessionDistributedLock(execCtx, sessionID)
    if err != nil {
        return fn(execCtx)
    }
    if distributedLock == nil && myredis.IsAvailable() {
        return codeExecutorResult{busy: true, code: code.CodeTooManyRequests}
    }
    if distributedLock != nil {
        distributedLock.StartWatchdog(execCtx, func() { cancel() })
        defer func() {
            _ = myredis.ReleaseSessionDistributedLock(execCtx, distributedLock)
        }()
    }

    return fn(execCtx)
}
```

它的执行顺序是：

1. 先拿本地锁。
2. 再尝试 owner lease。
3. 最后再拿 distributed lock。

这样设计的原因是：

- **本地锁最便宜**，先把同实例内并发挡住，避免同机多个 goroutine 一起去抢 Redis。
- **owner lease 用来解决“谁应该执行”**，它表达的是实例级所有权，不只是互斥。
- **distributed lock 用来解决“当前这次推进能不能进入临界区”**，它是最终的跨实例串行保护。

换句话说，`owner lease` 解决的是“谁是当前 owner”，`distributed lock` 解决的是“当前 owner 能不能安全进入本轮写路径”。

“多实例下 owner 选举与串行推进”这一段，核心代码其实是下面这部分：

```go
if myredis.IsAvailable() && rt.IsOwnerEligible() {
    preferredOwner, activeInstances, err := myredis.ResolvePreferredSessionOwner(execCtx, sessionID)
    if err != nil {
        observability.RecordSessionRedisLockDegrade()
    } else {
        currentInstanceID := rt.CurrentInstanceID()
        currentLease, ownerErr := myredis.GetSessionOwnerLeaseDetail(execCtx, sessionID)
        if ownerErr != nil {
            observability.RecordSessionRedisLockDegrade()
        }

        currentOwner := ""
        if currentLease != nil {
            currentOwner = currentLease.OwnerID
        }

        if preferredOwner != "" && preferredOwner != currentInstanceID && currentOwner != currentInstanceID {
            if currentOwner != "" {
                return codeExecutorResult{
                    busy:         true,
                    code:         code.CodeOwnerMismatch,
                    retryAfterMs: ownerMismatchRetryAfter,
                }
            }
        }

        ownerLease, ownerState, err := myredis.AcquireOrRefreshSessionOwnerLease(execCtx, sessionID, currentInstanceID)
        if err != nil {
            observability.RecordSessionRedisLockDegrade()
        } else if ownerLease == nil {
            return codeExecutorResult{
                busy:         true,
                code:         code.CodeOwnerMismatch,
                retryAfterMs: ownerMismatchRetryAfter,
            }
        } else {
            execCtx = withSessionOwnerGuard(execCtx, &sessionOwnerGuard{
                SessionID:  sessionID,
                OwnerID:    ownerLease.OwnerID,
                FenceToken: ownerLease.FenceToken,
            })
            ownerLease.StartSessionOwnerWatchdog(execCtx, func() {
                cancel()
            })
        }
    }
}
```

这段逻辑可以拆成 4 步：

1. `ResolvePreferredSessionOwner`
   先根据当前活跃实例和路由策略，算出“理论上更适合处理这个 session 的实例”。

2. `GetSessionOwnerLeaseDetail`
   再看看这个 `session` 现在是否已经有实例真正持有 owner lease。

3. `AcquireOrRefreshSessionOwnerLease`
   如果 lease 为空，当前实例可以尝试抢占；如果 lease 已经属于自己，就做续租；如果 lease 属于别人，则当前请求被拒绝。

4. `StartSessionOwnerWatchdog`
   一旦后续 watchdog 发现 owner lease 丢了，直接 `cancel()` 当前执行上下文，阻止旧 owner 继续推进。

这里有一个很关键的细节：`preferredOwner` 只是建议路由，不是最终裁决。

也就是说：

- 如果 hash/路由规则认为应该由 A 实例处理，但当前根本还没有任何实例拿到 lease，
  那么 B 实例只要真实收到了请求，仍然可以尝试成为 owner。
- 真正决定执行资格的，不是 `preferredOwner` 这个建议值，而是 `AcquireOrRefreshSessionOwnerLease` 这次 CAS 抢占是否成功。

这样做的原因是要避免“理论 owner 已经挂了，但 Redis 里短时间还保留着旧路由印象”，导致所有新请求都被误判成 busy。

owner lease 成功后，还会把 `OwnerID + FenceToken` 注入上下文：

```go
execCtx = withSessionOwnerGuard(execCtx, &sessionOwnerGuard{
    SessionID:  sessionID,
    OwnerID:    ownerLease.OwnerID,
    FenceToken: ownerLease.FenceToken,
})
```

后面所有关键写路径都会再校验一次这个 fence：

- 写 MySQL 前校验
- 写 Redis 热状态前校验
- 推进 session version 前校验

因此多实例串行推进的完整语义其实是：

1. 先通过 `preferredOwner` 做建议路由，减少跨实例抖动。
2. 再通过 `owner lease` 做真正的 owner 选举。
3. 然后通过 `distributed lock` 保证当前 owner 在这一次推进里独占临界区。
4. 最后通过 `fence token` 防止旧 owner 在失去资格后继续尾写。

配套的 owner fence 校验也很关键：

```go
func ensureSessionWriteOwnership(ctx context.Context, sessionID string) code.Code {
    guard := sessionOwnerGuardFromContext(ctx)
    if guard == nil || guard.SessionID != sessionID {
        return code.CodeSuccess
    }
    valid, err := myredis.ValidateSessionOwnerFence(ctx, sessionID, guard.OwnerID, guard.FenceToken)
    if err != nil {
        return code.CodeServerBusy
    }
    if !valid {
        return code.AIModelCancelled
    }
    return code.CodeSuccess
}
```

它的意义是：即使旧请求还没退出，只要 lease 已经被新实例接管，旧请求后续的 DB/Redis 写入也会被拒绝。

## 2.5 会话运行时层：AIHelper

对应代码：

- [common/aihelper/aihelper.go](/I:/Study/26_shixi/golang/GoMind_org/common/aihelper/aihelper.go)
- [common/aihelper/manager.go](/I:/Study/26_shixi/golang/GoMind_org/common/aihelper/manager.go)

`AIHelper` 是 **会话运行时对象**，在实例本地运行，不负责真相来源。

它负责：

- 保存最近消息窗口。
- 保存历史摘要。
- 保存当前会话版本和已持久化版本。
- 管理底层模型对象。
- 导入 / 导出 Redis 热状态。
- 拼接发给模型的上下文。

核心函数：

- `LoadHotState`
- `ExportHotState`
- `LoadMessages`
- `ReconcileMessages`
- `ensureContextSummary`
- `buildModelMessages`
- `GenerateResponseForPreparedUserMessage`
- `StreamResponseForPreparedUserMessage`
- `StreamResponseWithExistingAssistantForPreparedUserMessage`

`AIHelper` 这一层最重要的点是：它不是数据库模型或 Redis 快照，充当的角色是当前会话在本实例里的运行时执行器。

它的核心状态定义如下：

```go
type AIHelper struct {
    model                AIModel
    fallbackModel        AIModel
    messages             []*model.Message
    SessionID            string
    contextSummary       string
    summaryMessageCount  int
    messageWindowStart   int
    version              int64
    persistedVersion     int64
    selectionSignature   string
    recoveryMode         RecoveryMode
}
```

几个字段需要结合起来理解：

- `messages`：当前内存里保留的消息窗口。
- `contextSummary + summaryMessageCount`：较早历史已经被压缩成摘要，并且记录“摘要覆盖到了第几条消息”。
- `messageWindowStart`：当前 `messages` 这段切片在完整历史里的起点位置。
- `version`：当前 helper 认为会话已经推进到哪一轮。
- `persistedVersion`：当前 helper 认为 MySQL 已经正式追平到哪一轮。

Redis 热恢复对应的导入代码如下：

```go
func (a *AIHelper) LoadHotState(state *model.SessionHotState) {
    if state == nil {
        return
    }
    a.messages = make([]*model.Message, 0, len(state.RecentMessages))
    for i := range state.RecentMessages {
        msg := state.RecentMessages[i]
        modelMsg := &model.Message{
            MessageKey:     msg.MessageKey,
            SessionID:      msg.SessionID,
            SessionVersion: msg.SessionVersion,
            UserName:       msg.UserName,
            Content:        msg.Content,
            IsUser:         msg.IsUser,
            Status:         model.MessageStatus(msg.Status),
            CreatedAt:      msg.CreatedAt,
        }
        a.messages = append(a.messages, modelMsg)
    }
    a.contextSummary = state.ContextSummary
    a.summaryMessageCount = state.SummaryMessageCount
    a.messageWindowStart = state.RecentMessagesStart
    a.version = state.Version
    a.persistedVersion = state.PersistedVersion
    a.recoveryMode = RecoveryModeWarm
}
```

这里的关键点是：`LoadHotState` 恢复的是“可继续服务的上下文”，不是数据库真相。它把 helper 标成 `RecoveryModeWarm`，告诉后续逻辑当前是热恢复态。

真正发给模型前，会先保证摘要覆盖足够，再拼装上下文：

```go
func (a *AIHelper) ensureContextSummary(ctx context.Context) error {
    targetSummaryCount := a.messageWindowStart + len(a.messages) - maxContextMessages
    if targetSummaryCount <= a.summaryMessageCount {
        return nil
    }
    ...
    newSummary, err := a.model.GenerateSummary(ctx, currentSummary, utils.ConvertToSchemaMessages(messagesToSummarize))
    if err != nil {
        return nil
    }
    a.contextSummary = newSummary
    a.summaryMessageCount = targetSummaryCount
    return nil
}

func (a *AIHelper) buildModelMessages() []*schema.Message {
    start := a.summaryMessageCount - a.messageWindowStart
    if start < 0 {
        start = 0
    }
    if len(a.messages)-start > maxContextMessages {
        start = len(a.messages) - maxContextMessages
    }

    schemaMessages := make([]*schema.Message, 0, 1+len(a.messages[start:]))
    if a.contextSummary != "" {
        schemaMessages = append(schemaMessages, &schema.Message{
            Role: schema.System,
            Content: "以下是当前会话较早历史的摘要，请在回答时延续这些上下文信息：\n" + a.contextSummary,
        })
    }
    schemaMessages = append(schemaMessages, utils.ConvertToSchemaMessages(a.messages[start:])...)
    return schemaMessages
}
```

这两段代码说明了它的上下文策略：

- 更早历史不直接全量发送，而是压成摘要。
- 最近窗口消息保留原文。
- 最终给模型的是“摘要 + 最近 N 条”。

这也是为什么当前文档一直强调它是“运行时层”：它负责的是上下文组织，而不是消息保存。

## 2.6 能力编排与模型工厂层

对应代码：

- [common/aihelper/factory.go](/I:/Study/26_shixi/golang/GoMind_org/common/aihelper/factory.go)
- [common/aihelper/capability.go](/I:/Study/26_shixi/golang/GoMind_org/common/aihelper/capability.go)

这一层负责把“底层模型调用”和“上层聊天模式”分开。

对应关系：

- Provider：底层 LLM 调用
- Capability：聊天模式增强

当前支持的能力模式：

- `PlainChatCapability`
- `RAGChatCapability`
- `MCPChatCapability`
- `RAGMCPChatCapability`

这样设计后：

- 同一个 Provider 可以被不同聊天模式复用。
- RAG / MCP 不需要各自复制一套模型接入逻辑。

这个分层把“底层模型”和“聊天增强逻辑”拆开。

工厂创建逻辑如下：

```go
func (f *AIModelFactory) CreateAIHelper(ctx context.Context, modelType string, sessionID string, config RuntimeConfig) (*AIHelper, error) {
    provider, err := f.CreateProvider(ctx, modelType, config)
    if err != nil {
        return nil, err
    }

    capability, err := f.CreateCapability(config)
    if err != nil {
        return nil, err
    }

    model := NewCapabilityModel(provider, capability)
    helper := NewAIHelper(model, sessionID, config.SelectionSignature(modelType))
    return helper, nil
}
```

这里的读法是：

- `CreateProvider` 只关心怎么连到底层 LLM。
- `CreateCapability` 只关心当前聊天模式要不要加 RAG / MCP。
- 最后通过 `NewCapabilityModel(provider, capability)` 把两层组合起来。

`Capability` 本身也很直观，例如普通聊天和 RAG 的区别就是：

```go
func (c *PlainChatCapability) GenerateResponse(ctx context.Context, provider ChatProvider, messages []*schema.Message) (*schema.Message, error) {
    return provider.Generate(ctx, messages)
}

func (c *RAGChatCapability) GenerateResponse(ctx context.Context, provider ChatProvider, messages []*schema.Message) (*schema.Message, error) {
    ragMessages, err := c.buildRAGMessages(ctx, messages)
    if err != nil {
        return provider.Generate(ctx, messages)
    }
    return provider.Generate(ctx, ragMessages)
}
```

这说明：

- `PlainChatCapability` 什么都不做，直接透传。
- `RAGChatCapability` 先把最后一条用户问题改写成“带检索结果的 prompt”，再交给同一个 Provider。

因此这一层的作用也比较明确，即**聊天模式变化，不需要改 Provider；Provider 变化，也不需要重写 RAG/MCP 逻辑。**

## 2.7 数据模型层

对应代码：

- [model/session.go](/I:/Study/26_shixi/golang/GoMind_org/model/session.go)
- [model/message.go](/I:/Study/26_shixi/golang/GoMind_org/model/message.go)
- [model/session_state.go](/I:/Study/26_shixi/golang/GoMind_org/model/session_state.go)
- [model/stream_state.go](/I:/Study/26_shixi/golang/GoMind_org/model/stream_state.go)

关键字段说明：

- `Session.Version`：当前会话已经推进到第几轮。
- `Session.PersistedVersion`：MySQL 正式历史已经追平到第几轮。
- `Message.SessionVersion`：这条消息属于哪一轮推进。
- `Message.Status`：消息最终状态，如 `completed / cancelled / timeout / failed / partial`。
- `SessionHotState.SelectionSignature`：当前热状态绑定的模型选择签名。
- `SessionHotState.PendingPersist`：Redis 已推进，但 MySQL 未完全追平。
- `SessionHotState.HotStateDirty`：MySQL 成功了，但 Redis 热状态可能未完全追平。

结合结构体进行解析：

`Session` 里的正式版本字段如下：

```go
type Session struct {
    ID                  string
    ContextSummary      string
    SummaryMessageCount int
    Version             int64
    PersistedVersion    int64
}
```

这里有两个版本号：

- `Version`：会话逻辑上已经推进到哪一轮。
- `PersistedVersion`：MySQL 正式消息已经追平到哪一轮。

之所以要拆开，是因为链路里允许出现“Redis 热状态已经更先进，但 MySQL 还没完全收敛”的短暂窗口。

Redis 热状态对应的数据结构如下：

```go
type SessionHotState struct {
    SessionID           string
    SelectionSignature  string
    OwnerID             string
    FenceToken          int64
    Version             int64
    PersistedVersion    int64
    PendingPersist      bool
    HotStateDirty       bool
    ContextSummary      string
    SummaryMessageCount int
    RecentMessagesStart int
    RecentMessages      []SessionHotMessage
}
```

这个结构表达了三层信息：

- **恢复上下文**：`ContextSummary`、`SummaryMessageCount`、`RecentMessagesStart`、`RecentMessages`
- **一致性水位**：`Version`、`PersistedVersion`
- **分布式写保护**：`OwnerID`、`FenceToken`

而 `PendingPersist` / `HotStateDirty` 是补偿层用来识别“不一致已经出现在哪一侧”的标记：

- `PendingPersist=true`：Redis 这边已经推进了，但 MySQL 还可能没写完整。
- `HotStateDirty=true`：MySQL 已经成功了，但 Redis 热状态可能没及时追平。

流式恢复也有单独的数据模型：

```go
type StreamResumeMeta struct {
    StreamID         string
    SessionID        string
    MessageID        string
    UserName         string
    Status           StreamRuntimeStatus
    NextSeq          int64
    ResumeDeadlineAt *time.Time
    OwnerID          string
    FenceToken       int64
}
```

它和 `Message.Status` 不完全一样：

- `Message.Status` 描述一条 assistant 消息最终是 `completed/cancelled/partial/...`
- `StreamResumeMeta.Status` 描述的是流任务当前运行态，比如 `streaming`、`detached`

所以数据模型层真正做的事情，是把“正式历史状态”“共享热状态”“流式恢复状态”明确拆开。

## 2.8 恢复与补偿层

对应代码：

- [service/session/recovery.go](/I:/Study/26_shixi/golang/GoMind_org/service/session/recovery.go)
- [service/session/repair.go](/I:/Study/26_shixi/golang/GoMind_org/service/session/repair.go)

这一层负责两件事：

1. 当前请求进来时，恢复会话上下文。
2. 上一轮失败后，补偿未完全收敛的状态。

核心函数：

- `getReusableExecutionHelper`
- `BuildEphemeralHelperFromHotState`
- `BuildEphemeralHelperFromDB`
- `savePendingPersistHotStateBestEffort`
- `enqueueHotStateRebuildRepairBestEffort`
- `RepairPendingSessionPersistenceFromHotState`
- `RebuildSessionHotStateFromDatabase`

恢复层的核心目标不是“尽量复用缓存”，而是先判断当前该信任哪种状态来源。

进程内 helper 复用逻辑如下：

```go
func getReusableExecutionHelper(userName string, sessionID string, selectionSignature string) (*aihelper.AIHelper, bool) {
    manager := aihelper.GetGlobalManager()
    helper, exists := manager.GetAIHelper(userName, sessionID)
    if !exists {
        return nil, false
    }
    if !helper.MatchesSelection(selectionSignature) {
        manager.RemoveAIHelper(userName, sessionID)
        return nil, false
    }
    return helper, true
}
```

目的是“执行对象复用”，不是恢复真相源。因为即使命中了本地 helper，后续仍然可能用 Redis 或 DB 去对齐状态。

注：当前设置本地的 helper 的声明周期为 90s。

真正基于 Redis 热状态重建 helper 的代码如下：

```go
func BuildEphemeralHelperFromHotState(ctx context.Context, userName string, sess *model.Session, resolved *resolvedChatRequest, hotState *model.SessionHotState) (*aihelper.AIHelper, bool, error) {
    helper, reused, err := buildExecutionHelper(ctx, userName, sess.ID, resolved)
    if err != nil {
        return nil, false, err
    }

    helper.LoadHotState(hotState)
    applySessionMetadataToHelper(sess, helper)
    return helper, reused, nil
}
```

这里的顺序很重要：

- 先准备执行 helper。
- 再 `LoadHotState` 恢复最近窗口、摘要、版本水位。
- 最后用 `applySessionMetadataToHelper` 再拿 session 表里的正式元信息覆盖一次，避免 helper 持有过旧的 session 元数据。

如果 Redis 热状态不可信，就直接回退数据库重建：

```go
func BuildEphemeralHelperFromDB(ctx context.Context, userName string, sess *model.Session, resolved *resolvedChatRequest) (*aihelper.AIHelper, bool, codePathResult) {
    helper, reused, err := buildExecutionHelper(ctx, userName, sess.ID, resolved)
    if err != nil {
        return nil, false, codePathResult{ok: false, code: code.AIModelFail}
    }

    code_ := fullReconcileHelperWithDatabase(sess, helper)
    if code_ != code.CodeSuccess {
        return nil, reused, codePathResult{ok: false, code: code_}
    }
    return helper, reused, codePathResult{ok: true}
}
```

补偿层则专门处理“两边没有完全同时成功”的情况。典型例子是 Redis 已推进、MySQL 失败：

```go
func savePendingPersistHotStateBestEffort(ctx context.Context, helper *aihelper.AIHelper) {
    saveRepairHotStateBestEffort(ctx, helper, true, false)
}

func saveRepairHotStateBestEffort(ctx context.Context, helper *aihelper.AIHelper, pendingPersist bool, hotStateDirty bool) {
    hotState := helper.ExportHotState()
    if pendingPersist {
        hotState.PendingPersist = true
    }
    if hotStateDirty {
        hotState.HotStateDirty = true
    }
    _, _ = myredis.SaveSessionHotState(ctx, hotState)
}
```

也就是说，主链路失败时不会在原地无限重试，而是尽量把“需要修”的语义写回热状态。

之后 repair worker 会尝试把完整的一轮消息回放到 MySQL：

```go
func RepairPendingSessionPersistenceFromHotState(ctx context.Context, hotState *model.SessionHotState) (bool, error) {
    if hotState == nil || !hotState.PendingPersist {
        return false, nil
    }
    ...
    for _, version := range versions {
        var repairMessages []model.SessionHotMessage
        hasUser := false
        hasAssistant := false
        ...
        if !hasUser || !hasAssistant {
            continue
        }
        for _, hotMessage := range repairMessages {
            _, err := messageDAO.CreateMessage(&model.Message{
                MessageKey:     hotMessage.MessageKey,
                SessionID:      hotMessage.SessionID,
                SessionVersion: hotMessage.SessionVersion,
                UserName:       hotMessage.UserName,
                Content:        hotMessage.Content,
                IsUser:         hotMessage.IsUser,
                Status:         model.MessageStatus(hotMessage.Status),
            })
            if err != nil {
                return false, err
            }
        }
    }
    ...
    _, err := myredis.SaveSessionHotState(ctx, hotState)
    return true, err
}
```

这里最关键的保护条件是：

- 只修复 `PendingPersist=true` 的热状态。
- 只修复“既有 user 又有 assistant”的完整轮次。
- 修完 MySQL 后，再反过来把 Redis 热状态更新为已收敛。

所以恢复与补偿层的本质是：**主链路优先可用，最终一致性靠 Redis 热状态标记和 repair 任务补齐。**
