# 8. DeepAgent Chat Enhancement 设计：Planning / Search / Thought / Answer

在 LangGraph Chat Runtime 落地之后，Orbit 终于具备了一套足够稳定的聊天执行容器。普通 `chat` 已经不再依赖 service 内部的扁平循环，而是进入了 graph runtime。

这一步完成后，下一个问题自然就变成了：

```text
如果普通聊天已经可以跑在 graph 里
那么更复杂的 agent 能力，应该怎么接进现有会话体系？
```

这里有两种完全不同的方向。

一种做法是把 Agent 做成一套独立产品链路：

```text
Chat 一套运行时
Agent 另一套运行时
```

另一种做法是把 Agent 作为 Chat 的增强分支：

```text
同一套 Conversation / Message / Stream 产品模型
同一条 ChatGraph
只是在 graph 内部区分 normal_chat 和 agentic_chat
```

Orbit 选择的是第二种[^phase2-plan]。

因此，本轮 Phase 2 的核心是：

```text
让 DeepAgent 作为 Chat 的可选增强执行分支进入现有会话体系
```

本文讨论的，就是这一阶段的设计与落地。

## 1. 功能介绍

Phase 2 不是为了追求“什么都能做的超级代理”，而是先验证一条最小可行闭环：

```text
用户发送 agent 模式消息
  -> 进入 agentic_chat
  -> 生成 planning
  -> 执行 search / fetch 工具循环
  -> 生成每轮 summary
  -> 输出 reasoning
  -> 输出 final answer
  -> 前端展示 Thought Block + 正文
```

这一阶段主要解决三类问题。

### 1.1 Agent 模式如何进入现有 ChatGraph

在 LangGraph 阶段之前，`agent` 只是 `chat_mode` 的一个取值，但并没有真正进入 graph runtime。

当时系统的状态更像：

```text
chat  -> 新 LangGraph 路径
agent -> 旧 legacy 路径
tool  -> 旧 legacy 路径
```

这意味着 `agent` 还没有真正享受到 graph runtime 的好处：

- 没有统一的 route_execution
- 没有统一的 state
- 没有 graph 内部的事件分层
- 没有自然的 Thought Block 落点

Phase 2 首先要做的，就是让 `agent` 模式真正进入同一条 ChatGraph[^langgraph-runtime-post]。

### 1.2 Agent 的中间过程如何变成用户可理解的过程展示

DeepAgent 的真实执行过程往往包含：

- planning
- 多轮工具调用
- 中间推理
- 最终总结

如果把这些原始 trace 全量直接暴露出来，通常会有两个问题：

```text
信息太碎
协议太脏
```

Orbit 不希望 Thought Block 只是“原始工具日志回放”，而是希望它是：

```text
用户能读懂的任务过程摘要
```

因此 Phase 2 不是把 raw trace 原封不动搬出来，而是先把过程整理成结构化阶段：

```text
planning
loop(tool + summary)
reason
content
```

### 1.3 工具与工作区如何收敛到安全边界内

这一阶段不会直接开放：

- shell
- git
- 浏览器自动化
- 本地文件系统
- memory / HITL / subagent

因为这些能力一旦全部混进第一版 Agent，很难快速判断问题出在哪一层。

因此，Phase 2 只选择最小工具集：

- `websearch`
- `webfetch`
- 一个隔离的内存工作区

其目的不是做“全能代理”，而是验证：

```text
DeepAgent 能否在 Orbit 的 ChatGraph 中完成一次最小研究任务闭环
```

## 2. 现状与问题

在这一阶段之前，Orbit 已经有两层成熟基础：

### 2.1 会话产品模型已经稳定

Orbit 当前已经具备：

- `Conversation`
- `Message`
- message tree
- branch / fork / edit / regenerate
- replay / resume
- active stream
- assistant placeholder message

这意味着 Agent 能力不需要重新设计一套 Agent 会话模型。

### 2.2 LangGraph Chat Runtime 已经就位

在 Phase 1 中，普通 `chat` 已经进入：

```text
prepare_context
  -> call_model
  -> finalize_message
```

这一步的价值，在于 Orbit 已经有了一个可以继续长出分支的执行骨架。

### 2.3 当前仍然存在的问题

即使有了上述基础，在 Phase 2 开始前，系统仍然有几个明显缺口。

#### 2.3.1 `agent` 还不是 graph 内的一等执行分支

如果 `agent` 继续保留在 legacy 路径，那么 Orbit 最终会出现：

```text
普通 chat 一套运行时
Agent 一套运行时
```

这会让：

- 事件协议
- 状态模型
- 排障路径
- 未来 checkpoint / resume 能力

全部分裂成两套体系。

#### 2.3.2 Thought Block 还没有结构化来源

如果没有明确的 thought 结构，前端最终只能显示：

- reasoning token
- tool call delta
- tool result

但这更像调试视图，不像最终产品能力。

Orbit 需要的不是：

```text
“模型和工具到底吐了什么原始块”
```

而是：

```text
“这轮 Agent 做了什么、搜到了什么、现在做到哪一步了”
```

#### 2.3.3 工具循环还没有安全边界

如果不先给 Phase 2 划定边界，第一版 Agent 很容易直接失控成：

```text
任意搜索
任意抓取
任意写文件
任意执行本地命令
```

但在 Orbit 当前阶段，这种扩张速度会明显超过系统的可控性。

因此必须先确定一个更克制的最小版本：

```text
搜索
抓取
摘要
隔离工作区
硬预算
```

## 3. DeepAgent Chat Enhancement 设计

### 3.1 总体架构

Phase 2 之后，ChatGraph 从一个线性图扩展成了带路由的图：

```text
prepare_context
  ↓
route_execution
  ├─ normal_chat
  └─ agentic_chat
  ↓
finalize_message
```

其中：

- `chat_mode="chat"` -> `normal_chat`
- `chat_mode="agent"` -> `agentic_chat`
- `chat_mode="tool"` 仍暂时走 legacy 路径 （已废除）

这里最重要的设计判断是：

```text
mode 是 message level 的
不是 conversation level 的固定模式
```

这意味着：

- 同一个 conversation 的不同消息可以走不同模式
- 不同 branch 的不同叶子也可以走不同模式
- runtime 选择必须以当前这一次 assistant 生成对应的 message 为准

### 3.2 `agentic_chat` 的职责

`agentic_chat` 是 graph 内部一个明确的执行节点。

它主要负责三件事：

#### 3.2.1 创建 Agent 运行环境

包括：

- 组装 agent 工具集
- 创建隔离工作区
- 初始化 budget
- 准备 llm invoke 闭包

#### 3.2.2 把用户问题交给 DeepAgent

它会从 `input_messages` 中提取最后一条用户问题，并调用：

```text
DeepAgent.run(user_query, history_messages)
```

这意味着 DeepAgent 不是孤立运行，而是建立在现有会话上下文之上的。

#### 3.2.3 把 AgentResult 写回 ChatState

`DeepAgent` 返回的不只是正文，而是：

```text
planning_text
loop_summaries
reasoning_text
final_content
thought_events
workspace_files
token_usage
error
```

`agentic_chat` 再把其中和聊天运行时直接相关的部分写回 `ChatState`。

### 3.3 DeepAgent 的内部三阶段

DeepAgent 这一轮并没有实现一个极度通用的“任意多阶段代理框架”，而是先收敛成了三阶段结构：

```text
Phase 1: planning
Phase 2: tool loop + summary
Phase 3: final generation
```

这种拆法的目的，是把用户可见正文从工具循环中分离出来。

#### 3.3.1 Planning

Planning 阶段使用一次不带工具的 LLM 调用，生成简短的执行计划。

它承担两个作用：

- 给 Thought Block 一个稳定的起始段
- 给后续 Agent tool loop 提供显式执行目标

这里的 planning 不是为了做学术意义上的“计划器”，而是为了让 Agent 的过程展示不从一堆零散工具调用开始。

#### 3.3.2 Tool Loop + Summary

第二阶段是真正的 Agent 执行层。

它通过 `LLMClient.stream_with_messages(..., enable_tools=True)` 驱动工具调用。当前工具集主要包括：

- `websearch`
- `webfetch`
- `write_file`
- `read_file`
- `list_files`

但这一阶段并不直接把原始工具输出全部暴露给用户，而是再补一层每轮 summary：

```text
一轮工具调用结果
  -> 再调用一次轻量 LLM
  -> 产出 2-3 句阶段总结
```

这样 Thought Block 展示的主体就不再是原始日志，而是：

```text
工具做了什么
这一轮得到了什么
这些信息对最终回答有什么帮助
```

#### 3.3.3 Final Generation

这一阶段是整个设计里最关键的一个判断。

Orbit 没有让 Agent 在工具循环期间就直接开始输出最终回答，而是专门保留了独立的第三阶段：

```text
收集完工具信息
  -> 单独一次无工具 LLM 调用
  -> 统一生成 reasoning + final answer
```

这样做有两个非常重要的好处。

第一，避免协议冲突。

如果 Agent 一边输出正文，一边又继续触发工具调用，那么前端几乎无法稳定理解：

```text
现在这段是最终回答
还是中途的临时输出
```

第二，保证最终正文有清晰来源。

最终用户看到的正文，全部来自最后一次明确的总结阶段，而不是工具循环过程中的残留输出。

### 3.4 Thought Block 结构

这一阶段的 Thought Block 不是 raw trace，而是结构化阶段流。

当前结构为：

```text
planning
loop
  ├─ thought.tool
  └─ thought.summary
reason
content
```

对应的事件类型为：

- `thought.planning`
- `thought.tool`
- `thought.summary`
- `thought.reason`

这里有几个关键设计点。

#### 3.4.1 `tool` 和 `summary` 分开

如果只有 tool，没有 summary，用户看到的就只是：

```text
调用 websearch
调用 webfetch
调用 webfetch
```

这种信息非常像调试面板，而不像过程理解面板。

`summary` 的价值，在于给每轮执行加一个阶段归纳，让用户知道：

```text
这一轮到底得到了什么
```

#### 3.4.2 `reason` 继续保留为模型原生 thinking

Orbit 没有完全把模型 reasoning 抹掉，而是把它放在：

```text
planning / tool loop 之后
final answer 之前
```

这样既保留了模型思考痕迹，也不会让整个 Thought Block 只有 reasoning token。

#### 3.4.3 事件既用于 streaming，也用于 completed 后恢复

Thought 事件不是只存在于流式传输过程中。

Orbit 还把它们持久化到了消息读取链路里。也就是说：

```text
streaming 时
  -> 实时显示 Thought Block

completed 后刷新页面
  -> 仍然能恢复 Thought Block
```

这一步非常重要，因为它让 thought 不再只是一次性流事件，而开始成为消息结果的一部分。

### 3.5 工作区设计

这一阶段的工作区没有开放真实本地文件系统，而是选择了一个更安全的隔离版本：

```text
AgentWorkspace
```

它只允许操作白名单文件：

- `plan.md`
- `notes.md`
- `final.md`

这层设计的意义有三点。

#### 3.5.1 允许 Agent 有最小 artifact 能力

Agent 不再只是：

```text
一边搜
一边忘
```

它可以把：

- planning
- 中间笔记
- 最终答案

分别写入工作区文件。

#### 3.5.2 不直接暴露真实文件系统

Phase 2 的目标是验证：

```text
DeepAgent 是否适合 Orbit
```

而不是立刻开放系统级文件能力。

这一步能显著降低：

- 安全风险
- 调试复杂度
- 产品边界不清的问题

#### 3.5.3 为后续升级留下替换点

后续如果要把工作区从内存结构扩展到真正的文件系统，`AgentWorkspace` 本身就是天然的替换点。

### 3.6 预算与超时

Agent 的第一版必须具备硬预算，不然它很容易从“增强聊天”演变为“不可控长任务”。

当前引入的预算包括：

```text
max_rounds
max_tool_calls
max_search_calls_per_round
timeout_seconds
```

这几个预算的意义分别是：

- `max_rounds`：限制工具循环轮数
- `max_tool_calls`：限制全局工具调用总量
- `max_search_calls_per_round`：避免单轮无限搜索
- `timeout_seconds`：避免任务长期占用运行时

这一阶段预算不是前端可调参数，而是后端固定护栏。

这说明 Orbit 当前对 Agent 的定位仍然很克制：

```text
先证明闭环可控
再讨论更开放的能力
```

## 4. 实现思路

### 4.1 为什么 `agent` 必须先并入 ChatGraph

如果 Agent 继续独立存在，系统最终会演化出两套平行复杂度：

```text
Chat runtime 一套
Agent runtime 一套
```

这样会在后续逐渐制造更多重复设计：

- 一套事件协议
- 一套状态模型
- 一套 checkpoint 语义
- 一套消息收口逻辑

把 Agent 塞回 ChatGraph 的好处在于：

```text
产品模型继续统一
执行分支由 graph 控制
```

未来即使继续新增：

- `rag_chat`
- `tool_chat`
- `background_research`

也仍然可以围绕同一套 graph runtime 扩展。

### 4.2 为什么 Thought Block 不能直接暴露 raw trace

很多 Agent 系统在第一版都容易犯一个错误：

```text
“把所有内部 trace 直接显示出来”
```

这看起来信息很多，但实际产品效果往往不好：

- 太碎
- 太难读
- 太像调试界面
- 不适合作为稳定协议长期维护

Orbit 在这一阶段更偏向一个中间层思路：

```text
内部 trace 可以复杂
对外展示必须收敛成稳定阶段
```

所以当前 thought 流不是：

```text
raw tool trace
```

而是：

```text
planning / tool / summary / reason
```

### 4.3 为什么最终正文必须统一在最后阶段生成

这一点是整个 Phase 2 设计里最关键的工程取舍。

如果让 Agent 在工具循环中一边思考、一边出正文，那么会出现非常难处理的情况：

```text
正文已经开始输出
但后面又继续调工具
甚至又推翻前面的结论
```

这会直接冲击：

- 前端协议稳定性
- assistant message 的持久化语义
- 用户对“最终回答”的理解

因此 Orbit 选择：

```text
中间过程只输出 thought
最终正文只由最后阶段统一生成
```

这让：

- 过程
- reasoning
- final content

三者在协议上变得更清晰。

### 4.4 为什么第一版只开放搜索、抓取和内存工作区

在 Orbit 当前阶段，如果一开始就把：

- shell
- git
- browser
- 本地文件系统
- subagent

全部混进来，会让每次问题定位都变得异常困难。

而搜索 / 抓取 / 内存工作区这三者，已经足够验证：

```text
Agent 是否适合 Orbit 当前聊天产品形态————适合
Thought Block 是否真的有产品价值————有，优化前端推理、工具调用链显示，利于后续上下文的管理，特别是拼接与解压
message-level mode 是否成立——成立，提供了 message level 的灵活性。
```

## 5. 关键代码

### 5.1 `chat_runtime.py`：`route_execution` 如何把 `agent` 接回 ChatGraph

Phase 2 最核心的结构变化，仍然发生在：

```text
backend/app/services/langgraph_runtime/chat_runtime.py
```

这一阶段不再让 `agent` 留在 legacy 路径，而是把 graph 从线性结构扩成了带路由的结构：

```python
# backend/app/services/langgraph_runtime/chat_runtime.py
builder.add_node("prepare_context", self._prepare_context)
builder.add_node("route_execution", self._route_execution)
builder.add_node("normal_chat", self._normal_chat)
builder.add_node("agentic_chat", self._agentic_chat)
builder.add_node("finalize_message", self._finalize_message)

builder.add_conditional_edges(
    "route_execution",
    self._route_decision,
    {
        "normal_chat": "normal_chat",
        "agentic_chat": "agentic_chat",
    },
)
```

真正决定执行分支的逻辑非常简单：

```python
def _route_execution(self, state: ChatState) -> dict:
    chat_mode = state.get("chat_mode", "chat")
    execution_mode = "agentic_chat" if chat_mode == "agent" else "normal_chat"
    return {"execution_mode": execution_mode}
```

这段代码之所以重要，不是因为判断逻辑复杂，而是因为它第一次让：

```text
message level chat_mode
```

正式进入 runtime 决策层。

也就是说，`chat_mode` 不再只是 API 请求中的一个产品字段，而开始真正改变 graph 内部的执行路径。

### 5.2 `stream_run.py`：外层 producer 如何给 DeepAgent 准备运行环境

虽然 DeepAgent 真正运行在 `chat_runtime.py` 里，但外层 `stream_run.py` 负责把它接进现有会话体系。

关键代码如下：

```python
# backend/app/services/conversations/stream_run.py
effective_chat_mode = (
    getattr(assistant_message, "chat_mode", None) or conversation.chat_mode
)
if effective_chat_mode == "tool":
    await self._produce_stream_legacy(...)
    return

initial_state = self._build_langgraph_state(
    ...,
    chat_mode=effective_chat_mode,
)
```

并且，它为 `agentic_chat` 专门准备了一个 LLM 调用闭包：

```python
async def _agent_llm_invoke(
    messages,
    system_prompt,
    enable_tools,
    tools,
    tool_runtime,
    max_tool_rounds,
):
    async for chunk in _llm_client.stream_with_messages(
        config=_llm_config,
        messages=messages,
        model=_model,
        enable_tools=enable_tools,
        system_prompt=system_prompt,
        tools=tools,
        tool_runtime=tool_runtime,
        max_tool_rounds=max_tool_rounds,
    ):
        yield chunk
```

这层闭包有两个作用。

第一，它让 DeepAgent 可以多次调用 LLM：

- planning
- tool loop
- final generation

第二，它仍然继续复用 Orbit 现有的 `LLMClient.stream_with_messages()` 能力，而不是让 DeepAgent 直接接 provider SDK。

也就是说，DeepAgent 虽然是 agent harness，但底层模型调用仍然走 Orbit 已有的 provider 归一化层。

### 5.3 `chat_runtime.py`：`agentic_chat` 节点具体做了什么

`agentic_chat` 的核心职责，不是自己实现搜索逻辑，而是把 DeepAgent 组织成一个 graph 节点。

对应代码如下：

```python
# backend/app/services/langgraph_runtime/chat_runtime.py
async def _agentic_chat(self, state: ChatState) -> dict:
    if self._llm_invoke is None:
        return {"error": "agentic_chat 不可用：缺少 llm_invoke"}

    writer = get_stream_writer()

    def on_event(event: dict[str, Any]) -> None:
        writer(dict(event))

    external_tools = self._build_agent_tools()
    workspace = AgentWorkspace(run_id=state.get("assistant_message_id", "default"))

    agent = DeepAgent(
        external_tools=external_tools,
        tool_runtime=self._tool_runtime,
        llm_invoke=self._llm_invoke,
        budget=AgentBudget(
            max_rounds=3,
            max_tool_calls=6,
            max_search_calls_per_round=2,
            timeout_seconds=120,
        ),
        on_event=on_event,
        workspace=workspace,
    )
```

然后它从当前 `input_messages` 中提取最后一条用户问题：

```python
user_query = ""
for msg in reversed(state.get("input_messages", [])):
    if isinstance(msg, HumanMessage):
        user_query = str(msg.content)
        break
```

最后执行：

```python
result = await agent.run(
    user_query=user_query,
    history_messages=state.get("input_messages", []),
)
```

并把结果映射回 `ChatState`：

```python
return_updates = {
    "response_text": result.final_content,
    "reasoning_text": result.reasoning_text,
    "thought_events": result.thought_events,
    "workspace_files": result.workspace_files,
}
```

这段代码说明了一件非常重要的事：

```text
DeepAgent 不是独立产品运行时
而是 ChatGraph 内部的一个执行节点
```

也正因为如此，Orbit 才能继续沿用现有 Conversation / Message / Stream 模型，而不用另起一套 Agent 对象体系。

### 5.4 `deep_agent.py`：Planning / Tool Loop / Final Generation 如何拆开

DeepAgent 的主体位于：

```text
backend/app/services/langgraph_runtime/deep_agent.py
```

它的 `run()` 最关键的结构不是“能调工具”，而是三阶段拆分：

```python
# backend/app/services/langgraph_runtime/deep_agent.py
async def _run_internal(
    self,
    user_query: str,
    history_messages: list[BaseMessage],
) -> AgentResult:
    ws_tools = self._build_workspace_tools()
    self._tool_runtime.register_tools(ws_tools)
    all_tools = list(self._external_tools) + ws_tools

    self._planning_text = await self._phase_planning(user_query)

    if self._planning_text:
        await self._workspace.write_file("plan.md", self._planning_text)

    await self._phase_agent_loop(user_query, history_messages, all_tools)

    tool_error = self._error
    self._error = None
    self._final_content = ""
    self._reasoning_text = ""
    await self._phase_final_generation(user_query, history_messages)
```

这段代码里最值得注意的是最后三行。

Orbit 在这一阶段刻意把：

```text
工具循环
```

和：

```text
最终用户可见正文
```

拆开了。

也就是说，tool loop 不直接向用户流式输出最终回答，最终正文一定来自最后一次明确的 `final generation`。

这正是 Phase 2 里最关键的协议取舍之一。

### 5.5 `deep_agent.py`：工具循环为什么既发 `tool` 又发 `summary`

工具循环里最核心的逻辑位于 `_phase_agent_loop()`：

```python
if chunk.tool_results:
    tool_round += 1
    results = chunk.tool_results
    total_tool_calls += len(results)

    if total_tool_calls > self._budget.max_tool_calls:
        raise RuntimeError(...)

    search_calls_this_round = sum(
        1 for r in results if str(r.get("name") or "").strip() == "websearch"
    )
    if search_calls_this_round > self._budget.max_search_calls_per_round:
        raise RuntimeError(...)

    for tr in results:
        self._emit(
            "thought.tool",
            "loop",
            f"调用工具：{tr.get('name', 'unknown')}",
            {"tool": tr.get("name", "unknown"), "args": tr.get("args", {})},
        )

    summary_text = await self._summarize_tool_round(...)
    self._emit("thought.summary", "loop", summary_text, {...})
```

这里有三层设计含义。

第一，预算控制不只是配置对象存在，而是真的在循环里被执行。

第二，`thought.tool` 和 `thought.summary` 是分开的。Orbit 并没有把 Thought Block 直接设计成原始工具日志，而是额外补了一层“这一轮得到了什么”的总结语义。

第三，工具结果还会继续写入 `notes.md`：

```python
existing += f"\n## 第 {tool_round} 轮\n{summary_text}\n"
for tr in results:
    existing += f"\n### {tr.get('name', 'unknown')}\n{tr.get('output', '')}\n"
await self._workspace.write_file("notes.md", existing)
```

这意味着工作区不是摆设，而是 Agent 中间资料的真实容器。

### 5.6 `deep_agent.py`：为什么最终正文一定来自最后阶段

Phase 3 的代码如下：

```python
async def _phase_final_generation(
    self,
    user_query: str,
    history_messages: list[BaseMessage],
) -> None:
    notes_content = self._workspace.get_content("notes.md") or ""
    system_text = (
        "你是一个研究助手。以下是你通过搜索和浏览收集到的信息：\n\n"
    )
    if notes_content:
        system_text += notes_content
    system_text += (
        f"\n\n用户原始问题：{user_query}\n\n"
        "请基于以上收集的信息，给出详细、有依据的中文回答。"
    )
```

然后执行一次不带工具的 LLM 调用：

```python
async for chunk in self._llm_invoke(
    messages,
    None,
    False,
    None,
    None,
    None,
):
    if chunk.reasoning_delta:
        self._emit("thought.reason", "reason", chunk.reasoning_delta)
        self._emit_raw({"type": "reasoning_delta", "delta": chunk.reasoning_delta})

    if chunk.content_delta:
        content_parts.append(chunk.content_delta)
        self._emit_raw({"type": "content_delta", "delta": chunk.content_delta})
```

这里最关键的不是“又调了一次模型”，而是：

```text
最终正文的协议来源非常明确
```

也就是说，前端看到的最终 `message.delta`，不是工具循环过程中某个中间输出残留下来的结果，而是明确来自最后一次总结生成阶段。

这让：

- Thought Block
- reasoning
- final answer

三者在协议层面能够稳定分层。

### 5.7 `agent_workspace.py` 与 `stream_adapter.py`：最小工作区和 Thought 事件如何打到前端

`AgentWorkspace` 位于：

```text
backend/app/services/langgraph_runtime/agent_workspace.py
```

它的边界非常明确：

```python
_ALLOWED_FILES = frozenset({"plan.md", "notes.md", "final.md"})

def _resolve(self, path: str) -> str:
    normalized = PurePosixPath(path).name
    if normalized not in _ALLOWED_FILES:
        raise WorkspaceSecurityError(...)
    return normalized
```

也就是说，Agent 虽然拥有“写文件”能力，但并不能越过这三个白名单文件。

与此同时，`StreamAdapter` 负责把后端 thought 事件变成前端消息事件：

```python
# backend/app/services/langgraph_runtime/stream_adapter.py
elif event_type.startswith("thought."):
    await conversation_stream_store.append_event(
        self._stream_id,
        event="message.thought",
        data={
            "message_id": str(self._message_id),
            "type": event_type,
            "phase": event.get("phase", ""),
            "text": event.get("text", ""),
            "meta": event.get("meta", {}),
        },
    )
```

后续 `useConversations.ts` 再把这些事件合并成：

```text
message.thought_events
```

最终由 `MessageBubble.tsx` 渲染为 Thought Block。

这说明 Phase 2 不是只做了“后端能调搜索”，而是已经把：

```text
planning / tool / summary / reason
```

这一套结构化过程真正打到了消息 UI 上[^phase2-report]。

## 6. 总结

DeepAgent 阶段完成的，不是“再造一个 Agent 产品”，而是把 Agent 能力嵌回了 Orbit 已有的聊天运行时。

它带来的最大变化是：

```text
Orbit 的 chat 不再只有“问一句，答一句”
而开始具备“为回答做准备、执行任务、展示过程、再生成答案”的能力
```

但与此同时，这一阶段仍然保持了足够克制的边界：

- Agent 是 Chat enhancement，不是独立产品模型
- Thought Block 是结构化阶段流，不是 raw trace 面板
- 工作区是隔离内存区，不是真实文件系统
- 工具是最小集合，不是全开放
- 预算是硬护栏，不是无限制自治

这也正是 Orbit 当前阶段更稳妥的一条路线：

```text
先把 Agent 增强能力嵌入现有聊天体系
先证明它在产品、协议和运行时层面都能闭环
再决定后续是否向更开放的代理系统继续扩展
```
