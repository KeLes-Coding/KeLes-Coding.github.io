# 5. 会话管理设计：Branch / Fork / active_child

本文记录 Orbit 下一阶段会话管理架构设计。

MVP 阶段的 `messages` 表将一个 conversation 视为一条线性消息列表：

```text
U1 -> A1 -> U2 -> A2 -> U3 -> A3
```

这个模型可以支撑最基础的多轮对话，但无法自然表达以下需求：

```text
重发某条 assistant 回复
编辑某条历史 user 消息
在多个回答版本之间切换
从历史节点继续探索
从某个历史点 fork 出新会话
```

因此，Orbit 的会话管理需要从线性消息列表升级为 **message tree**。

核心设计是：

```text
parent_message_id        负责形成树
active_child_message_id  负责记录每个分叉口当前选中的 child
active_leaf_message_id   负责缓存当前 visible path 的最终节点
```

一句话概括：

```text
parent 负责形成树；
active_child 负责记住每个分叉口选了哪条路；
active_leaf 负责缓存当前整条路径最终停在哪里；
branch 在同一棵树里切路径；
fork 把当前路径复制成一棵新的树。
```

## 1. 功能介绍

本阶段主要扩展 Orbit 的会话管理能力，使 conversation 不再只能线性追加消息，而是可以在同一个会话内保留多个版本路径。

### 1.1 Branch

Branch 表示同一个 conversation 内的路径切换。

它用于解决：

```text
同一个问题，我想保留多个版本。
```

典型场景包括：

```text
重发 assistant 回复
编辑 user 消息
切换某条消息的 sibling 版本
从历史节点继续探索
```

Branch 的特点是：

```text
conversation_id 不变
message tree 不变
只改变 active_child / active_leaf
当前 visible path 发生变化
```

### 1.2 Fork

Fork 表示从当前 conversation 的某个历史节点复制一段 visible path，创建一个新的 conversation。

它用于解决：

```text
这个分支我想单独展开成一个新的长期主题。
```

Fork 的特点是：

```text
创建 new_conversation
复制当前 visible path 中 root -> target message 的消息
新 conversation 拥有独立的 active_leaf_message_id
后续演进不影响原 conversation
```

第一阶段推荐使用 **Fork visible path only**：

```text
只复制当前可见路径中从 root 到目标 message 的消息。
```

暂不复制目标节点下方未选中的 sibling 子树。

### 1.3 Edit / Regenerate / Stop

Edit 和 Regenerate 不作为独立的底层结构存在，而是复用 message tree 的 sibling 机制。

```text
Regenerate assistant:
  在原 assistant 的 parent 下创建新的 assistant sibling。

Edit user:
  在原 user 的 parent 下创建新的 user sibling，并从新 user 继续生成 assistant。

Stop:
  终止当前 streaming run，并把当前 assistant message 标记为 partial 或 cancelled。
```

这样可以避免为每一种操作创建独立的数据模型。

## 2. 核心概念

### 2.1 Message-level parent

Orbit 推荐采用 **message-level parent**。

也就是说，每条消息都是树中的一个节点，`parent_message_id` 指向上一条可见消息。

例如：

```text
U1 -> A1 -> U2 -> A2
```

存储关系是：

```text
U1.parent = null
A1.parent = U1
U2.parent = A1
A2.parent = U2
```

其中：

```text
user 的 parent 通常是上一条 assistant；
assistant 的 parent 通常是触发它生成的 user。
```

不要让 `parent` 指向 "user-assistant 对"。

`turn` 是展示层或派生层概念，不建议作为底层树结构。

原因是：

```text
assistant 可能有多个重发版本
assistant 可能处于 streaming / partial / failed 状态
后续 Agent 场景会有 tool message
user-assistant pair 无法自然表达复杂分支
message 是 LLM 上下文线性化的最小单位
```

### 2.2 active_leaf

`conversations.active_leaf_message_id` 表示：

```text
当前会话正在展示的 active path 的最终消息。
```

它的主要作用是快速加载当前可见路径。

加载会话消息时，可以从 `active_leaf` 沿 `parent_message_id` 一路回溯到 root，再 reverse 得到当前 visible path：

```text
active_leaf -> parent -> parent -> ... -> root
reverse 后得到：
root -> ... -> active_leaf
```

### 2.3 active_child

仅有：

```text
conversation.active_leaf_message_id
message.parent_message_id
```

可以表达当前路径，但不足以恢复局部分支的选择状态。

例如：

```text
U1
└── A1
    └── U2
        ├── A2
        │   └── U3
        │       └── A3
        └── A2'
            ├── U3'
            │   └── A3'
            │       └── U4'
            │           └── A4'
            └── U3''
                └── A3''
```

如果用户当前在 `A2` 分支下，然后切换到 `A2'`，系统不能简单选择：

```text
A2' 本身
```

也不能选择：

```text
A2' 子树下最深的 leaf
```

因为这都不一定是用户上次在 `A2'` 子树里选择的路径。

更合理的规则是：

```text
每个节点记录自己当前选中的 child。
```

也就是：

```text
messages.active_child_message_id
```

例如：

```text
U2.active_child = A2'
A2'.active_child = U3''
U3''.active_child = A3''
```

那么从 root 一路沿着 `active_child_message_id` 向下走，就可以得到当前可见路径：

```text
U1 -> A1 -> U2 -> A2' -> U3'' -> A3''
```

### 2.4 active_leaf 与 active_child 的分工

推荐同时保留：

```text
conversations.active_leaf_message_id
messages.active_child_message_id
```

它们不是重复字段，而是分工不同。

| 字段 | 粒度 | 作用 |
| --- | --- | --- |
| `active_leaf_message_id` | conversation | 缓存当前 visible path 的最终节点 |
| `active_child_message_id` | message | 记录当前节点在 children 中选中了哪个 child |

二者配合后，可以同时获得：

```text
active_child_message_id:
  branch 选择的真实状态。
  负责恢复每个分叉点选中的 child。
  是 message tree 当前路径的 source of truth。

active_leaf_message_id:
  当前路径终点的缓存。
  负责让发送消息、加载上下文、恢复 streaming 状态更快。
  可以由 active_child 链重新计算出来。

```

### 2.5 为什么不让每条 message 记录 active_leaf

不建议设计成：

```text
messages.active_leaf_message_id
```

因为这会带来写放大。

例如当前路径：

```text
U1 -> A1 -> U2 -> A2' -> U3'' -> A3''
```

如果每个节点都记录最终 leaf：

```text
U1.active_leaf = A3''
A1.active_leaf = A3''
U2.active_leaf = A3''
A2'.active_leaf = A3''
U3''.active_leaf = A3''
```

当用户继续发送：

```text
U4 -> A4
```

就要把整条祖先链全部更新：

```text
U1.active_leaf = A4
A1.active_leaf = A4
U2.active_leaf = A4
...
```

这是 `O(depth)` 写入。

而 `active_child` 只需要更新直接父节点：

```text
A3''.active_child = U4
conversation.active_leaf = U4
```

再创建 assistant：

```text
U4.active_child = A4
conversation.active_leaf = A4
```

所以每次追加消息只需要：

```text
O(1) 写入
```

更适合长会话。

## 3. 数据模型设计

本阶段在 MVP 的 `conversations` 和 `messages` 表上增加树结构字段。

`users` 和 `llm_configs` 不需要调整。

核心关系从：

```text
Conversation 1 -> N Message
Message 按 sequence_no 线性排序
```

升级为：

```text
Conversation 1 -> N Message
Message 通过 parent_message_id 形成 tree
Conversation 通过 active_leaf_message_id 记录当前路径终点
Message 通过 active_child_message_id 记录局部分支选择
```

### 3.1 `conversations`

`conversations` 保存会话级别元信息。

本阶段新增字段：

| 字段 | 说明 |
| --- | --- |
| `active_leaf_message_id` | 当前 active path 的最终消息 |
| `forked_from_conversation_id` | 当前会话从哪个原会话 fork 而来 |
| `forked_from_message_id` | 当前会话从原会话的哪条消息 fork 而来 |
| `summary_leaf_message_id` | 当前摘要对应的 active path leaf |

推荐迁移：

```sql
alter table conversations
add column active_leaf_message_id uuid null;

alter table conversations
add column forked_from_conversation_id uuid null
  references conversations(id) on delete set null;

alter table conversations
add column forked_from_message_id uuid null
  references messages(id) on delete set null;

alter table conversations
add column summary_leaf_message_id uuid null
  references messages(id) on delete set null;

alter table conversations
add constraint fk_conversations_active_leaf_message
foreign key (active_leaf_message_id)
references messages(id)
on delete set null;
```

> [!NOTE]
> `active_leaf_message_id` 引用 `messages(id)` 时，迁移顺序需要注意循环依赖。
> 可以先新增 nullable 字段，再单独添加 foreign key constraint。

#### 设计说明

`active_leaf_message_id` 用于快速恢复当前 visible path。

会话初始化时：

```text
active_leaf_message_id = null
```

第一轮用户消息和 assistant 消息写入后：

```text
active_leaf_message_id = A1
```

`forked_from_conversation_id` 和 `forked_from_message_id` 用于记录 fork 来源。

这两个字段只记录来源，不影响新会话的后续演进。Fork 后的新会话拥有独立的消息树、标题、摘要、模型配置和 active path。

`summary_leaf_message_id` 用于标记当前摘要对应哪条 active path。

在 message tree 结构下，同一个 conversation 内不同 branch 的上下文可能不同，因此摘要不能只依赖消息数量。后续摘要策略需要同时记录：

```text
summary
summary_updated_at
summary_message_count
summary_leaf_message_id
```

### 3.2 `messages`

`messages` 仍然是会话内容的事实源。

本阶段新增字段：

| 字段 | 说明 |
| --- | --- |
| `parent_message_id` | 当前消息的父消息，用于组成 message tree |
| `active_child_message_id` | 当前消息在 children 中当前选中的 child |
| `depth` | 当前消息在树中的深度 |
| `source_message_id` | 当前消息来源于哪条旧消息 |
| `revision_type` | 当前消息的创建来源类型 |

推荐迁移：

```sql
alter table messages
add column parent_message_id uuid null
  references messages(id) on delete set null;

alter table messages
add column active_child_message_id uuid null
  references messages(id) on delete set null;

alter table messages
add column depth integer not null default 0;

alter table messages
add column source_message_id uuid null
  references messages(id) on delete set null;

alter table messages
add column revision_type varchar(30) null;
```

字段说明：

| 字段 | 说明 |
| --- | --- |
| `parent_message_id` | 当前消息接在哪一条历史消息之后 |
| `active_child_message_id` | 当前消息在自己的 children 中选中的 child |
| `depth` | root 为 0，child = parent.depth + 1 |
| `source_message_id` | edit / regenerate / fork copy 的来源消息 |
| `revision_type` | `normal`、`edit`、`regenerate`、`fork_copy` |

#### 设计说明

`parent_message_id` 是 message tree 的主结构字段。

读取某条 visible path 时，可以从 leaf 回溯：

```sql
with recursive path as (
  select *
  from messages
  where id = :active_leaf_message_id

  union all

  select parent.*
  from messages parent
  join path child on child.parent_message_id = parent.id
)
select *
from path
order by depth asc, sequence_no asc;
```

`active_child_message_id` 是局部分支选择字段。

切换 sibling 时，只需要更新目标 parent：

```text
parent.active_child_message_id = target_child.id
```

然后从 `target_child` 开始沿 active_child 链向下走，得到新的 active leaf。

`sequence_no` 继续保留，但语义从 UI 展示顺序调整为：

```text
conversation 内消息创建顺序。
```

Sibling 的稳定排序可以使用：

```text
同一 parent 下按 sequence_no asc 排序。
```

这样前端可以计算：

```text
当前 sibling index
sibling count
```

用于显示 `1 / 2`、`2 / 2` 等版本切换控件。

### 3.3 推荐索引

```sql
create index idx_messages_conversation_parent
on messages(conversation_id, parent_message_id);

create index idx_messages_conversation_active_child
on messages(conversation_id, active_child_message_id);

create index idx_messages_conversation_depth
on messages(conversation_id, depth);

create index idx_messages_source_message
on messages(source_message_id)
where source_message_id is not null;

create index idx_conversations_active_leaf
on conversations(active_leaf_message_id)
where active_leaf_message_id is not null;
```

应用层需要保证：

```text
active_child_message_id 指向的消息必须满足：

active_child.conversation_id = current_message.conversation_id
active_child.parent_message_id = current_message.id
```

数据库层可以通过 foreign key 保证消息存在，但跨字段一致性更适合放在 service 层校验。

## 4. 操作语义

本节说明普通发送、assistant 生成、重发、编辑、切换 branch 和 fork 的写入规则。

### 4.1 普通发送消息

当前 active leaf 是：

```text
A2
```

用户发送新消息 `U3`。

写入 user message：

```text
U3.parent = A2
U3.depth = A2.depth + 1
U3.revision_type = normal
A2.active_child = U3
conversation.active_leaf = U3
```

随后创建 assistant placeholder `A3`：

```text
A3.parent = U3
A3.depth = U3.depth + 1
A3.status = streaming
A3.revision_type = normal
U3.active_child = A3
conversation.active_leaf = A3
```

流式完成后：

```text
A3.status = completed
```

如果用户手动停止：

```text
A3.status = partial 或 cancelled
conversation.active_leaf = A3
```

空会话第一轮消息是特殊情况：

```text
U1.parent = null
U1.depth = 0
A1.parent = U1
A1.depth = 1
conversation.active_leaf = A1
```

### 4.2 Assistant 生成

Assistant 生成本质上是在某条 user message 后创建 assistant child。

第一阶段仍然沿用 MVP 的 placeholder 设计：

```text
创建 assistant message:
  role = assistant
  content = ''
  status = streaming

流式过程中:
  持续追加 content

生成完成:
  status = completed

生成失败:
  status = failed

用户停止:
  status = partial 或 cancelled
```

Assistant message 需要继续保存模型快照：

```text
llm_config_id
provider
model
token_usage
response_metadata
```

这样用户后续切换会话默认模型，不会影响历史 assistant 的审计和成本统计。

### 4.3 重发 assistant

原路径：

```text
U1 -> A1 -> U2 -> A2
```

用户重发 `A2`。

`A2` 的 parent 是 `U2`。

新建 `A2'`：

```text
A2'.parent = U2
A2'.depth = U2.depth + 1
A2'.source_message_id = A2
A2'.revision_type = regenerate
A2'.status = streaming
```

然后：

```text
U2.active_child = A2'
conversation.active_leaf = A2'
```

结构变为：

```text
U1
└── A1
    └── U2
        ├── A2
        └── A2'
```

原来的 `A2` 不删除，只是不在当前 active path 上。

### 4.4 编辑 user

原路径：

```text
U1 -> A1 -> U2 -> A2
```

用户编辑 `U2`，得到 `U2'`。

`U2` 的 parent 是 `A1`。

新建 `U2'`：

```text
U2'.parent = U2.parent = A1
U2'.depth = U2.depth
U2'.content = edited_content
U2'.source_message_id = U2
U2'.revision_type = edit
U2'.status = completed
```

然后：

```text
A1.active_child = U2'
conversation.active_leaf = U2'
```

接着自动生成 assistant placeholder `A2'`：

```text
A2'.parent = U2'
A2'.depth = U2'.depth + 1
A2'.status = streaming
U2'.active_child = A2'
conversation.active_leaf = A2'
```

结构变为：

```text
U1
└── A1
    ├── U2
    │   └── A2
    └── U2'
        └── A2'
```

原来的 `U2 -> A2` 仍然保留，可以切回。

### 4.5 切换 branch

假设结构是：

```text
U2
├── A2
│   └── U3
│       └── A3
└── A2'
    ├── U3'
    │   └── A3'
    │       └── U4'
    │           └── A4'
    └── U3''
        └── A3''
```

用户从 `A2` 切到 `A2'`。

正确逻辑：

```text
1. U2.active_child = A2'
2. 从 A2' 开始沿 active_child 向下走
3. 得到 A2' 子树中上次选中的 leaf
4. conversation.active_leaf = leaf
```

如果此前：

```text
A2'.active_child = U3''
U3''.active_child = A3''
```

那么切换后路径是：

```text
U1 -> A1 -> U2 -> A2' -> U3'' -> A3''
```

而不是：

```text
U1 -> A1 -> U2 -> A2'
```

也不是：

```text
A2' 子树下最深的 leaf
```

### 4.6 Fork

Fork 和 Branch 不同。

Branch 是：

```text
同一个 conversation 内切换路径。
```

Fork 是：

```text
从当前 conversation 的某个历史节点复制出一个新的 conversation。
```

第一版建议只做：

```text
Fork visible path only
```

也就是只复制当前 active path 中从 root 到 target message 的消息。

例如当前 visible path：

```text
U1 -> A1 -> U2' -> A2' -> U3'' -> A3''
```

从 `A2'` fork：

```text
new conversation:
U1_copy -> A1_copy -> U2'_copy -> A2'_copy
```

复制时重建 parent 和 active_child：

```text
U1_copy.active_child = A1_copy
A1_copy.active_child = U2'_copy
U2'_copy.active_child = A2'_copy
new_conversation.active_leaf = A2'_copy
```

每条 copy message 记录来源：

```text
copy.source_message_id = old_message.id
copy.revision_type = fork_copy
```

Fork 后的新会话独立拥有：

```text
conversation_id
thread_id
title
summary
active_leaf_message_id
messages
```

后续新会话继续对话、编辑、重发或归档，都不影响原 conversation。

## 5. 一致性约束

Message tree 结构需要维护几个重要的不变式。

### 5.1 active path 不变式

`conversation.active_leaf_message_id` 必须等于：

```text
从 root 开始沿 active_child_message_id 一路走到的最终节点。
```

反过来说：

```text
从 active_leaf 沿 parent 回溯得到 visible path；
这条 path 上每个 parent.active_child 都应该指向 path 中的下一个 message。
```

### 5.2 parent / child 一致性

如果：

```text
M.active_child_message_id = C.id
```

则必须满足：

```text
C.conversation_id = M.conversation_id
C.parent_message_id = M.id
```

### 5.3 conversation 归属一致性

如果：

```text
conversation.active_leaf_message_id = M.id
```

则必须满足：

```text
M.conversation_id = conversation.id
```

### 5.4 role 交替规则

聊天模式下，常规路径通常是：

```text
user -> assistant -> user -> assistant
```

但底层模型不应该强制只能交替。

原因是后续可能出现：

```text
system
tool
assistant tool call
tool result
agent event
```

因此，第一阶段可以在 service 层对普通 chat API 做轻约束：

```text
普通发送必须接在当前 active leaf 后
assistant 生成必须接在 user message 后
regenerate 目标必须是 assistant message
edit 目标必须是 user message
```

底层 `messages` 表保留更通用的 message tree 能力。

### 5.5 事务与并发

普通发送、重发、编辑、切换 branch 和 fork 都应该在事务内完成。

同一个 conversation 内的写操作建议先锁 conversation 行：

```sql
select *
from conversations
where id = :conversation_id
for update;
```

原因是这些操作都会修改：

```text
conversations.active_leaf_message_id
messages.active_child_message_id
messages.sequence_no
```

如果同一个 conversation 同时发生多个写操作，可能导致 active path 错乱或 sequence_no 冲突。

第一阶段可以采用 conversation 行锁保证同一会话内写入串行化。

前端不需要直接消费完整 message tree。

推荐 API 层仍然默认返回当前 visible path。

| 方法 | 路径 | 说明 |

如果 assistant 没有内容:

Stop 不删除 message。

## 7. 前端展示语义

前端默认只展示 visible path。

例如后端返回：

```text
U1 -> A1 -> U2 -> A2'
```

前端不需要知道完整树结构，也不需要自行计算 active path。

为了支持版本切换，`MessageRead` 可以增加 sibling 信息：

```ts
interface MessageBranchState {
  parent_message_id?: string | null
  active_child_message_id?: string | null
  sibling_index: number
  sibling_count: number
  previous_sibling_id?: string | null
  next_sibling_id?: string | null
}
```

其中：

```text
sibling_index:
  当前 message 在同 parent children 中的序号，从 1 开始。

sibling_count:
  同 parent children 的总数。

previous_sibling_id / next_sibling_id:
  用于左右切换版本。
```

Assistant 多版本展示：

```text
A2   1 / 2
A2'  2 / 2
```

User 编辑版本展示：

```text
U2   1 / 2
U2'  2 / 2
```

切换版本时，前端调用 switch API。

后端返回新的 visible path，前端整体替换当前消息数组即可。

## 8. 迁移策略

当前数据库中已有的线性消息需要迁移成 tree。

迁移规则：

```text
同一个 conversation 内按 sequence_no asc 排序。

第一条消息:
  parent_message_id = null
  depth = 0

后续消息:
  parent_message_id = 前一条消息 id
  depth = 前一条消息 depth + 1

每条非末尾消息:
  active_child_message_id = 下一条消息 id

conversation.active_leaf_message_id = 最后一条消息 id
```

如果 conversation 还没有消息：

```text
active_leaf_message_id = null
```

迁移后，旧的线性会话会变成一条没有分叉的 message tree。

```text
U1 -> A1 -> U2 -> A2
```

等价于：

```text
U1.active_child = A1
A1.active_child = U2
U2.active_child = A2
A2.active_child = null
conversation.active_leaf = A2
```

### 9.3 Repository 能力

`MessageRepository` 需要从线性读写升级为 tree 读写。

当前真正落地的方法主要有：

```text
list_visible_path(conversation)
list_path_to_message(conversation_id, message_id)
list_siblings(message)
create_user_message(...)
create_assistant_placeholder(...)
set_conversation_active_leaf(conversation, message)
resolve_active_leaf_from(message_id)
clone_message_to_conversation(source, target_conversation_id, parent_message)
```

#### 9.3.1 `list_visible_path()`：为什么前端仍然能收到线性消息数组

可见路径读取的核心代码是：

```python
async def list_visible_path(self, conversation: Conversation) -> list[Message]:
    if conversation.active_leaf_message_id is None:
        return []

    messages_by_id: dict[UUID, Message] = {}
    current_id: UUID | None = conversation.active_leaf_message_id
    visited: set[UUID] = set()
    while current_id is not None:
        if current_id in visited:
            break
        visited.add(current_id)
        message = await self.get_by_id(conversation_id=conversation.id, message_id=current_id)
        if message is None:
            break
        messages_by_id[message.id] = message
        current_id = message.parent_message_id

    path = list(messages_by_id.values())
    path.reverse()
    return path
```

这段代码说明，聊天 UI 并不需要直接消费整棵树。后端只需要：

```text
从 conversation.active_leaf_message_id 出发
  -> 沿 parent_message_id 回溯到 root
  -> reverse 成 root -> leaf
```

这样即使底层存储已经 tree 化，前端仍然可以继续使用：

```text
Message[]
```

这种最简单的线性数组结构。

#### 9.3.2 `create_user_message()` / `create_assistant_placeholder()`：为什么追加写入只需要 O(1)

当前普通发送消息时，Repository 的关键不是只创建 message，而是创建后立刻把父节点选中新 child。

例如 `create_user_message()`：

```python
if parent_message is not None:
    parent_message.active_child_message_id = message.id
    await self.session.flush()
```

`create_assistant_placeholder()` 里也一样：

```python
if parent_message is not None:
    parent_message.active_child_message_id = message.id
    await self.session.flush()
```

这就把前面设计里的规则真正落成了代码：

```text
新增 child 时
  -> 直接让 parent.active_child 指向这个 child
```

因此普通追加写入不需要更新整条祖先链，只需要：

```text
创建新 message
更新直接父节点 active_child
更新 conversation.active_leaf
```

这就是 `active_child + active_leaf` 组合能把写入复杂度维持在 `O(1)` 的原因。

#### 9.3.3 `resolve_active_leaf_from()`：Branch 为什么不是简单切到目标节点

分支切换最关键的恢复逻辑是：

```text
从指定 message 开始
如果当前 message.active_child_message_id 存在，则继续向下
直到遇到 active_child_message_id 为空的节点
返回最终 leaf
```

当前代码就是：

```python
async def resolve_active_leaf_from(self, *, conversation_id: UUID, message: Message) -> Message:
    current = message
    visited: set[UUID] = set()
    while current.active_child_message_id is not None:
        if current.id in visited:
            break
        visited.add(current.id)
        child = await self.get_by_id(
            conversation_id=conversation_id,
            message_id=current.active_child_message_id,
        )
        if child is None or child.parent_message_id != current.id:
            break
        current = child
    return current
```

这段代码对应的设计结论是：

```text
Branch 切换时
  -> 不是只切到目标 sibling 本身
  -> 也不是找它子树里最深的节点
  -> 而是沿目标 sibling 自己保存的 active_child 选择继续恢复
```

这才能保证用户切回某条老分支时，看到的是“上次停留的位置”。

#### 9.3.4 `clone_message_to_conversation()`：Fork v1 为什么只复制 visible path

Fork 的核心复制逻辑是：

```python
async def clone_message_to_conversation(
    self,
    *,
    source: Message,
    target_conversation_id: UUID,
    parent_message: Message | None,
) -> Message:
    clone = Message(
        conversation_id=target_conversation_id,
        parent_message_id=parent_message.id if parent_message else None,
        depth=(parent_message.depth + 1) if parent_message else 0,
        source_message_id=source.id,
        revision_type="fork_copy",
        ...
    )
    self.session.add(clone)
    await self.session.flush()
    if parent_message is not None:
        parent_message.active_child_message_id = clone.id
        await self.session.flush()
    await self.session.refresh(clone)
    return clone
```

这段实现有两个要点：

```text
1. 新消息保留 source_message_id，方便追溯 fork 来源
2. 复制时重新建立 parent / active_child，而不是复用原树节点
```

因此 Fork 后的新会话是：

```text
结构相似
来源可追溯
但后续独立演进
```

### 9.4 Service 能力

`ConversationService` 需要承担 active path 的业务一致性。

当前主要落在：

```text
backend/app/services/conversations/branch.py
```

已经真正实现的核心能力有：

```text
switch_branch
fork_conversation
```

#### 9.4.1 `switch_branch()`：为什么只改一个 parent 的 active_child

当前分支切换逻辑是：

```python
if target.parent_message_id is not None:
    parent = await self.messages.get_by_id(
        conversation_id=conversation_id,
        message_id=target.parent_message_id,
    )
    parent.active_child_message_id = target.id

leaf = await self.messages.resolve_active_leaf_from(
    conversation_id=conversation_id,
    message=target,
)
await self.messages.set_conversation_active_leaf(conversation=conversation, message=leaf)
```

这个实现非常贴合前面的设计：

```text
切 branch 时
  -> 只修改目标 sibling 的父节点 active_child
  -> 再从目标 sibling 沿 active_child 恢复整条子路径
  -> 最后更新 conversation.active_leaf
```

也就是说，Branch 不是“替换整条路径”，而是：

```text
只在分叉点改一次选择
剩下的路径由子树内部已有状态自己恢复
```

#### 9.4.2 `fork_conversation()`：为什么只能从当前 visible path fork

Fork 逻辑里最关键的一段是：

```python
visible_path = await self.messages.list_visible_path(conversation)
target_index = next(
    (index for index, message in enumerate(visible_path) if message.id == message_id), None
)
if target_index is None:
    raise HTTPException(
        status_code=status.HTTP_400_BAD_REQUEST, detail="只能从当前可见路径 fork"
    )
```

这说明当前 Fork v1 的边界非常明确：

```text
只允许从当前 visible path 上的节点 fork
不复制未选中的 sibling 子树
```

后面的复制循环也严格按：

```text
root -> target
```

顺序复制，逐条重建新会话里的 parent / active_child 链。

#### 9.4.3 为什么这些写操作都必须在事务内

无论是 `switch_branch()` 还是 `fork_conversation()`，都会同时修改：

```text
conversation.active_leaf_message_id
messages.active_child_message_id
messages.sequence_no
conversation.updated_at
```

所以当前代码里这些操作都围绕同一个会话事务执行，并在 service 收口阶段：

```python
await self.conversations.touch(conversation_id)
await self.session.commit()
```

这样前端看到的 visible path 才不会落在一个“active_child 已改、active_leaf 还没改”或者“新会话已建、消息还没复制完”的中间态。
