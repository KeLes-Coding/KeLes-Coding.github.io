# 6. 可恢复并行会话设计：Replay / Resume / Multi-branch / Idempotency

本文记录 Orbit 在现有消息树与流式消息能力基础上，如何进一步支持：

```text
1. SSE 断开后继续运行同一条 LLM 生成任务
2. 客户端按 last_seq replay 缺失事件并恢复 live 订阅
3. 同一 conversation 内多个 branch 同时运行
4. 普通发送、edit、regenerate 的显式 base message 语义
5. 重复请求幂等复用，不重复写树、不重复创建运行实例
```

这篇文档合并了原来的“断点续传实现计划”和“并行会话设计”两部分内容，统一描述 Orbit 当前阶段的运行时模型、数据模型和接口语义。

## 1. 功能介绍

本阶段主要补齐三类能力。

### 1.1 可恢复流式运行

前端 SSE 连接断开、页面刷新或同一用户多端查看同一会话时，后端仍应继续运行同一条 LLM 生成任务。客户端重新连上后，可以根据自己的 `last_seq` 追赶缺失事件，然后继续实时订阅。

这里的恢复语义应作出区分，不是：

```text
让模型从某个 token 位置继续生成 （模型侧的续传）
```

而是：

```text
让 Orbit 后台自己维护一条可 replay（用于追赶）、可 resume（用于实时流式传播） 的应用侧 stream
```

### 1.2 并行 branch 运行

单主线会话只能表达“当前唯一一条活跃回复”，但 Orbit 已经具备 `message tree`、`active_child`、`active_leaf`、`edit`、`regenerate` 和 `fork` 等能力。接下来需要允许：

```text
同一 conversation 内多个 branch 同时运行
普通发送显式挂到任意合法 base message 后
不同 branch 独立恢复各自对应的 active stream
```

### 1.3 幂等与并发控制

并行阶段一旦允许同一父节点下并发创建 sibling，就必须明确处理两类问题：

```text
重复请求
并发请求
```

重复请求需要通过 `idempotency_key` 复用既有 user / assistant / stream，而并发请求需要通过 base message 级锁和原子顺序号分配，保证消息树写入稳定。

## 2. 现状与问题

当前阶段，Orbit 已经具备以下基础能力：

```text
assistant streaming 占位消息
message.created / delta / completed / failed / cancelled 事件
message tree
branch switch
fork
edit / regenerate
```

但此前的单流时代仍然保留了三条关键限制：

```text
1. LLM 生成任务和当前 SSE 连接绑定在同一个协程中
2. 同一 conversation 只能存在一条 active stream
3. 普通发送默认隐式挂到 active_leaf 后面
```

这些限制会带来一组连锁问题：

```text
前端断开连接会影响生成任务生命周期
客户端没有稳定的 stream_id / seq，无法补发缺失事件
conversation 级唯一 active stream 无法表达多个 branch 同时 streaming
用户切 branch 后无法恢复该 branch 对应的运行实例
重复点击发送或重试时容易生成重复 sibling
max(sequence_no) + 1 在并发写入时容易冲突
```

因此，当前 Orbit 需要从“单会话、单主线、请求内直连生成”的模式，升级为“可恢复、可并行、可幂等”的运行时模型。

## 3. 可恢复并行会话设计

### 3.1 三层模型

本阶段需要把 Orbit 的能力拆成三层：

```text
Conversation = 聚合摘要层
Message      = 业务事实层
Run / Stream = 运行实例层
```

#### 3.1.1 Conversation

`Conversation` 负责：

```text
保存标题、默认模型、summary、thread_id
缓存当前可见路径末端 active_leaf_message_id
会话级状态信号 has_active_run （用于前端侧边栏非焦点标记（执行中/完成通知））
```

它服务的是：

```text
侧边栏会话列表
当前主路径快速加载
“这个会话当前是否还有后台运行”的轻量提示
```

#### 3.1.2 Message

`Message` 继续作为会话消息事实源，负责：

```text
保存消息正文与最终状态
表达 parent / child / sibling 关系
表达 edit / regenerate / fork_copy 来源
提供页面加载、历史查询、导出和审计依据
```

消息树的核心关系仍然由以下字段表达：

```text
parent_message_id
active_child_message_id
source_message_id
revision_type
status
sequence_no
```

#### 3.1.3 Run / Stream

`Run / Stream` 表示一次具体的流式运行实例，代表 “某条 assistant 生成任务在当前运行时中的事件序列”。

它负责：

```text
维护 stream_id
分配 seq
记录事件日志
支持 replay catch-up
支持 live fan-out
在 retention 窗口内提供 resume 能力
```

这三层拆开之后：

```text
Conversation 不再承担“唯一恢复入口”
Message 不再承担“运行时事件日志”
Run / Stream 专门承担“这次生成任务如何运行、如何恢复”
```

### 3.2 运行时分层

可恢复流式运行需要再把运行链路拆成三部分：

```text
1. Stream Producer
2. Stream Runtime Store
3. SSE Subscriber
```

#### 3.2.1 Stream Producer

`Stream Producer` 负责：

```text
创建 assistant streaming 占位消息
读取 llm_client.stream
把 provider chunk 转成 Orbit 应用级事件
更新 assistant runtime snapshot
结束时落库最终 completed / failed / cancelled 状态
```

它不直接关心某个前端连接是否还在线。

#### 3.2.2 Stream Runtime Store

`Runtime Store` 保存的是事件序列，例如：

```text
message.created
message.delta
message.reasoning_delta
message.completed
conversation.run_state_changed
```

它负责：

```text
create_stream
append_event
subscribe
get_stream
get_stream_by_message_id
complete_stream
cancel_stream
cleanup_expired_streams
```

#### 3.2.3 SSE Subscriber

订阅端统一做两段式输出：

```text
1. replay catch-up
2. live subscribe
```

客户端只要持有：

```text
stream_id
last_seq
```

就可以在刷新、闪断或切换页面后继续追平。

### 3.3 Runtime Store 与 Message Snapshot 的区别

为了支持恢复，Orbit 需要同时保留“过程”和“当前状态”。

`Runtime Store` 保存的是事件日志，例如：

```text
message.created
message.delta
message.reasoning_delta
message.completed
```

而 assistant runtime snapshot 保存的是当前累计状态，例如：

```text
content = "今天天气不错，适合"
reasoning_content = "..."
status = "streaming"
```

两者分别解决：

```text
Runtime Store    -> replay 缺失事件
Runtime Snapshot -> 确认当前已生成到哪里
```

可以把它们理解成：

```text
Runtime Store    -> 录像
Runtime Snapshot -> 当前画面缓存
```

### 3.4 数据模型设计

为了支撑可恢复并行会话，本阶段主要补充 `conversations` 与 `messages` 的运行态字段。

#### 3.4.1 `conversations`

本阶段新增：

| 字段 | 说明 |
| --- | --- |
| `has_active_run` | 会话级聚合信号，用于快速判断当前是否仍存在运行中的 branch |
| `next_message_sequence_no` | 会话内消息顺序号分配器，避免并发写入时使用 `max(sequence_no) + 1` |

同时继续保留：

| 字段 | 说明 |
| --- | --- |
| `active_leaf_message_id` | 当前可见路径末端缓存 |
| `active_stream_id` | 兼容字段，表示最近一条可恢复运行 |
| `active_stream_message_id` | 兼容字段，表示最近一条可恢复 assistant message |

设计上需要明确：

```text
has_active_run 是会话级缓存
真相来自 messages 中是否仍存在 streaming assistant
```

#### 3.4.2 `messages`

本阶段新增：

| 字段 | 说明 |
| --- | --- |
| `idempotency_key` | 请求幂等键，用于复用同一次发送、edit 或 regenerate |

同时继续使用：

| 字段 | 说明 |
| --- | --- |
| `parent_message_id` | 形成消息树 |
| `active_child_message_id` | 指向当前主路径 child |
| `source_message_id` | 标记 edit / regenerate 的来源消息 |
| `revision_type` | 标记 `normal` / `edit` / `regenerate` / `fork_copy` |
| `status` | 标记 `streaming` / `completed` / `failed` / `cancelled` |
| `sequence_no` | 会话内稳定顺序号 |

#### 3.4.3 幂等索引

本阶段幂等作用域定为：

```text
(conversation_id, parent_message_id, idempotency_key)
```

它表示：

```text
同一 conversation 中
同一 base message 下
同一个 idempotency_key
表示同一次用户请求
```

对于 root 场景，还需要补充：

```text
(conversation_id, idempotency_key) where parent_message_id is null
```

这样可以覆盖：

```text
首条消息发送
普通消息发送
edit
regenerate
```

#### 3.4.4 `sequence_no` 分配

`max(sequence_no) + 1` 在并发写入时容易产生冲突，也会让数据库重复扫描消息表。

因此本阶段改为：

```text
从 conversations.next_message_sequence_no 原子分配
```

这样可以保证：

```text
同一 conversation 内 sequence_no 唯一
并发下仍保持单调递增
不依赖 max() 扫描
```

### 3.5 事件协议设计

#### 3.5.1 统一事件结构

Orbit 统一输出应用级事件结构：

```json
{
  "stream_id": "str",
  "seq": 12,
  "conversation_id": "uuid",
  "message_id": "uuid",
  "event": "message.delta",
  "data": {}
}
```

SSE 编码时输出为：

```text
id: <stream_id>:<seq>
event: message.delta
data: {"message_id":"...","delta":"..."}
```

#### 3.5.2 事件集合

本阶段保留以下事件：

```text
conversation.created
conversation.updated
conversation.run_state_changed
message.created
message.delta
message.reasoning_delta
message.completed
message.failed
message.cancelled
```

其中：

```text
conversation.created / updated   -> 会话元信息变化
conversation.run_state_changed   -> 会话级后台运行状态变化
message.created                  -> user / assistant 占位消息出现
message.delta                    -> assistant 正文增量
message.reasoning_delta          -> reasoning 增量
message.completed / failed / cancelled -> 运行结束
```

#### 3.5.3 `Last-Event-ID` 与 `last_seq`

SSE 标准支持浏览器通过 `Last-Event-ID` 自动恢复，但 Orbit 当前前端采用的是：

```text
fetch + ReadableStream + 应用层手动解析 SSE
```

因此本阶段选择：

```text
前端记录最后收到的 seq
恢复请求时主动传递 last_seq
后端 replay (last_seq, latest_seq] 后切换到 live 订阅
```

同时，事件仍然输出标准 SSE 的 `id`：

```text
id: <stream_id>:<seq>
```

这样既保留了 SSE 标准事件编号语义，也不会把恢复流程绑定到浏览器自动重连行为上。

### 3.6 接口设计

#### 3.6.1 普通发送接口

普通发送从“隐式 active_leaf”升级为“显式 base message”：

```text
POST /conversations/{conversation_id}/messages
POST /conversations/{conversation_id}/messages/stream
```

请求体需要显式包含：

```text
content: str
parent_message_id: UUID | null
idempotency_key: str | null
```

本阶段约束为：

```text
1. parent_message_id 必须属于当前 conversation
2. parent_message_id **不要求必须等于当前 active_leaf**（本次迭代的重点，conversation从绑定单一流到多分支的基础）
3. idempotency_key 用于复用重复请求
4. 未传 parent_message_id 时，暂时回退到旧 active_leaf 语义以兼容旧调用方
```

#### 3.6.2 Edit / Regenerate

`edit` 与 `regenerate` 仍然保留原有产品语义：

```text
edit user
  -> base message = 被编辑 user 的 parent

regenerate assistant
  -> base message = 被重发 assistant 的 parent user
```

本阶段主要补齐的是：

```text
幂等
base message 级锁
branch 级恢复
```

没有改掉它们原本的产品语义。

#### 3.6.3 恢复接口

单流时代的会话级恢复入口：

```text
GET /conversations/{conversation_id}/stream
```

不再适合作为并行阶段的唯一恢复入口，因为它默认：

```text
一个 conversation 只有一条 active stream
```

因此本阶段拆成两类能力：

```text
1. branch / message scoped lookup
2. stream scoped subscribe
```

接口为：

```text
GET /conversations/{conversation_id}/messages/{message_id}/active-stream
GET /conversations/{conversation_id}/streams/{stream_id}
```

其中：

```text
messages/{message_id}/active-stream -> 查当前 branch 是否存在活跃流
streams/{stream_id}                 -> 对具体运行实例做 replay + live 订阅
```

前端恢复流程变为：

```text
1. 用户切到某个 branch
2. 前端先查询当前 message_id 对应的 active stream
3. 如果存在，则按 stream_id 恢复 replay + live
4. 如果不存在，则只展示当前消息树快照
```

#### 3.6.4 取消接口

取消仍然以运行实例为目标，而不是简单关闭当前 SSE 连接：

```text
POST /conversations/{conversation_id}/messages/{message_id}/cancel
```

它真正取消的是 producer，而不是订阅者。

### 3.7 并发控制与幂等设计

并行阶段同时引入“锁”和“幂等”，但两者职责不同。

#### 3.7.1 幂等解决什么问题

幂等负责处理：

```text
同一请求被重复提交
前端重试
网络闪断后重复点击
同一 payload 被再次送达
```

对于命中的请求：

```text
直接复用已有 user / assistant / stream
不重复创建消息
不重复创建运行实例
```

#### 3.7.2 锁解决什么问题

锁负责处理：

```text
同一 base_message_id 下两个不同请求同时进入
```

它保护的是：

```text
读取 parent
分配 sequence_no
创建 user sibling
更新 active_child
创建 assistant placeholder
```

锁不负责识别“是不是重复请求”，这部分仍由幂等完成。

#### 3.7.3 base message 级锁

本阶段按 base message 粒度加锁：

```text
root 场景 -> conversation:{conversation_id}:root
有 parent 的场景 -> message:{parent_message_id}
```

这样既能避免同一父节点下 sibling 写入打架，又不会把整个 conversation 都串行化。

### 3.8 前端交互设计

并行运行落地后，前端需要同步调整三类行为。

#### 3.8.1 发送行为

前端在发送、edit、regenerate 时都需要显式生成并传递 `idempotency_key`。

普通发送还需要显式传递当前可见 branch 的 `parent_message_id`。

#### 3.8.2 恢复行为

页面刷新、重新进入会话或切换 branch 后，前端不再按 conversation 唯一流恢复，而是：

```text
按当前 visible branch 查 active stream
再按 stream_id 做 replay + live 订阅
```

#### 3.8.3 会话列表与消息列表状态

会话列表主要使用：

```text
conversation.has_active_run
conversation.run_state_changed
```

来维护“该会话当前仍有后台运行”的动画提示。

消息列表则需要额外标识：

```text
当前 visible branch 的 leaf
当前 leaf 是否仍在运行
```

这样用户才能区分：

```text
整个会话里仍有人在跑
当前正在看的这条 branch 是否在跑
```

## 4. 实现思路

当前版本规划的基础业务链路为：

```text
用户进入会话
  -> 选择当前 visible branch
  -> 发送消息或 edit / regenerate
  -> 显式传递 parent_message_id 与 idempotency_key
  -> 后端创建 user / assistant placeholder
  -> 注册 runtime stream
  -> 后台 producer 调用模型并持续写入事件日志
  -> 前端按 stream_id + last_seq 进行 replay / live 恢复
  -> 会话列表与消息列表根据运行状态同步更新
```

### 4.1 请求阶段如何启动一条新运行

本阶段最大的变化，是把“请求负责创建运行”和“后台任务负责真正生成”彻底拆开。

普通发送在请求阶段只做以下工作：

```text
1. 校验 conversation 归属
2. 解析 parent_message_id，确定本次运行挂在哪个 base message 下
3. 计算 base message 级锁
4. 在锁内检查幂等是否命中
5. 命中则直接复用既有 stream
6. 未命中则创建新的 user message 与 assistant placeholder
7. 生成 stream_id，并更新 conversation 的运行态缓存
8. 提交事务
9. 创建 runtime stream，先写入 message.created
10. 启动后台 producer task
```

这里的关键是把这两者分成两个阶段：

```text
数据库事务阶段
  -> 负责把消息树和会话级缓存稳定落库

运行时注册阶段
  -> 负责创建 event log、写入初始事件、启动 producer
```

这样做的好处是：

```text
请求阶段的错误仍然可以作为普通 HTTP 错误返回
assistant placeholder 可以先稳定落库
第一个订阅者一定能先看到 message.created
后续 delta / completed / failed / cancelled 都围绕同一条 assistant message 展开
```

### 4.2 base message 锁、幂等与顺序号分配如何协作

并行 branch 的实现并不是只靠某一个机制，而是让“锁、幂等、顺序号分配”在同一个受控流程里配合起来。

#### 4.2.1 锁解决什么

锁回答的问题是：

```text
同一个 base_message_id 下，两个不同请求同时进来时，谁先写 sibling？
```

当前实现不会锁整个 conversation，而是按 base message 粒度加锁：

```text
root 场景               -> conversation:{conversation_id}:root
有 parent 的场景       -> message:{parent_message_id}
```

这样意味着：

```text
同一 parent 下的 sibling 写入会串行化
不同 parent 下的并行 branch 仍可同时运行
```

#### 4.2.2 幂等解决什么

幂等回答的问题是：

```text
这次请求是不是前一次请求的重放？
```

如果不在锁内做幂等判断，就会出现这样的竞争窗口：

```text
请求 A 和请求 B 同时进入
  -> 都查到当前没有同一 idempotency_key
  -> 都各自创建 user / assistant
  -> 同一语义被重复落树
```

因此当前顺序是：

```text
先锁住 base message
  -> 再查 idempotency_key
  -> 命中就复用
  -> 未命中才真正写新消息
```

锁负责避免并发写树，幂等负责避免重复请求复写，这是两套不同但必须串联的机制。

#### 4.2.3 顺序号为什么必须原子分配

`max(sequence_no) + 1` 的问题不只是性能，而是并发下根本不安全：

```text
不同事务可能读到相同的 max(sequence_no)
消息表需要重复扫描
```

所以本阶段改成：

```text
从 conversations.next_message_sequence_no 原子领取 sequence_no
```

这样 user message 和 assistant placeholder 无论在多少分支同时写入，都会拿到稳定且单调递增的会话内顺序号。

### 4.3 producer 如何把模型调用收口成稳定状态

真正的模型调用发生在后台 producer，而不再发生在 SSE 请求协程本身。

producer 的生命周期可以拆成以下阶段：

```text
读取 stream 运行态
  -> 读取 assistant placeholder 与 conversation
  -> 根据 assistant.parent_message_id 重建历史上下文
  -> 把当前 asyncio task 记录成 producer_task
  -> 进入 llm_client.stream 循环
  -> 持续写入 reasoning_delta / content_delta
  -> 在 completed / cancelled / failed 三种出口之一完成收口
  -> finally 中 complete_stream，进入短暂 retention 窗口
```

这个设计有两个很重要的作用。

第一，后台运行上下文来自 assistant placeholder，而不是来自前端当前页面状态：

```text
assistant_message.parent_message_id
  -> 决定这次生成任务的历史上下文
```

也就是说，一旦 assistant placeholder 已经创建，这次运行的上下文就冻结了。即使用户之后切换到别的 branch，也不会影响这条正在跑的 stream。

第二，producer 不只是“把 chunk 往外吐”，它负责把这次模型调用收口成一个最终可解释的状态：

```text
正常完成     -> message.completed
主动取消     -> message.cancelled
模型错误     -> message.failed 或 partial
未知异常     -> 兜底 failed
```

所有这些出口，在发最终事件之前都会先统一更新会话级状态，这样前端不会出现“消息已经结束，但会话仍显示 running”的漂移。

### 4.4 恢复为什么必须 branch-aware

并行之后，恢复已经不能再从 conversation 唯一入口出发，因为：

```text
一个 conversation 下可能同时有多条 active stream
conversation.active_stream_id 只能指向其中一条
```

所以当前恢复必须分成两步：

```text
1. 先从当前 branch 上的 message_id 解析出 active assistant
2. 再从这条 assistant 反查对应的 stream_id
```

这一步的真实语义不是“查 conversation 有没有活跃流”，而是：

```text
查当前这条 branch 的当前 leaf 是否是一条 streaming assistant（
两种状态：
- streaming（那么可以继续流式输出）;
- 其他的终止状态：根据不同情况（success/partial——全量从数据库恢复；fail——显示错误原因）进行处理。
）
如果是，再接到这条 assistant 对应的 stream 上
```

这样才能保证：

```text
刷新不会串流
branch 切换不会误接到别的并行运行
同一 conversation 内多个 branch 可以各自恢复
```

### 4.5 取消为什么也必须按 message 命中

取消逻辑和恢复逻辑一样，本质上都不能只停留在 conversation 级。

当前取消接口虽然路径里还带着 `conversation_id`，但真正命中的运行实例是：

```text
assistant message_id
```

runtime store 内部通过：

```text
message_id -> stream_id
```

找到对应的 producer task。

因此当前取消语义从：

```text
取消整个 conversation 的所有运行
```

迭代到：

```text
取消这条 assistant message 对应的单次运行
```

这样，同一个 conversation 下的其他并行 branch 不会被误伤。

### 4.6 会话级缓存如何和 branch 级真相保持一致

并行之后，真正的运行真相分散在：

```text
messages.status == streaming
runtime store 中仍 active 的 stream
```

但前端侧边栏仍然需要一个很轻的会话级信号。所以当前实现采用的是：

```text
branch 级真相  -> assistant message.status
会话级缓存    -> conversations.has_active_run
```

两者的同步策略是：

```text
stream 启动时，直接把 has_active_run 置为 true
stream 结束时，统一重新扫描 conversation 下是否还存在 streaming assistant
如果布尔值发生跳变，再发 conversation.run_state_changed
```

这样做的好处是：

```text
启动快，不必每次都扫消息表
结束准，最终以数据库消息状态为真相重算
前端能拿到轻量聚合信号，又不会长期漂移
```

### 5.1 `ConversationStreamStore`：运行时事实源，而不是 SSE 附属品

`ConversationStreamStore` 是本阶段最核心的运行时抽象。它负责保存内存中的活跃 stream、事件日志、订阅者队列，以及 `message_id -> stream_id` 的映射关系。

对应代码见 `backend/app/services/streaming.py:47`：

```python
class ConversationStreamStore:
    def __init__(self) -> None:
        self._streams: dict[str, ActiveConversationStream] = {}
        self._message_to_stream: dict[UUID, str] = {}
        self._lock = asyncio.Lock()
```

其中：

```text
_streams            -> 保存当前所有活跃或短暂保留中的 stream
_message_to_stream  -> 允许按 assistant message 精确找到对应运行实例
_lock               -> 保证 create / append / cancel / cleanup 的并发安全
```

这一层最重要的两个功能：
- 不只是广播 SSE，而是保存一条可回放的应用层事件序列
- 充当一层短生命周期的运行时事实源。

事件写入与回放的核心在 `backend/app/services/streaming.py:94` 和 `backend/app/services/streaming.py:115`：

```python
record = StreamEventRecord(
    stream_id=stream.stream_id,
    seq=stream.next_seq,
    event=event,
    data=data,
)
stream.next_seq += 1
stream.event_log.append(record)
```

```python
replay_records = [record for record in current.event_log if record.seq > last_seq]
```

这里实际完成了三件事：

```text
1. `append_event` 负责在单条 stream 内原子分配 seq，并把事件落进 event_log
2. `subscribe` 负责先根据 last_seq 回放缺失区间
3. 只有 stream 仍 active 时，订阅者才会被挂进 live queue
```

Replay / Resume 的核心：

```text
先把应用层事件顺序保存下来
恢复时按 last_seq 补发缺失区间
再无缝切到 live 队列
```

取消逻辑也在这一层完成，`backend/app/services/streaming.py:156`：

```python
stream_id = self._message_to_stream.get(message_id)
stream.cancel_event.set()
producer_task.cancel()
```

这意味着取消是：

```text
按 assistant message 定位到单条 stream
只中断这条 stream 的 producer task
不会取消整个 conversation 下所有并行流
```

另外，`complete_stream` 不会立刻删除 stream，而是只把它标记成 non-active，并记录 `retire_at`。这就是 replay retention 的实现基础。

### 5.2 `start_stream_user_message`：请求阶段如何稳定启动一条新运行

普通发送的主入口在 `backend/app/services/conversation.py:64`。

这一段代码真正重要的地方在于它把并行发送最容易出错的几个点放进了同一个受控流程：

```python
parent_message = await self._resolve_parent_message(
    conversation_id=conversation_id,
    conversation=conversation,
    parent_message_id=parent_message_id,
)
lock_key = self._base_message_lock_key(
    conversation_id=conversation_id,
    parent_message_id=parent_message.id if parent_message else None,
)
```

```python
existing_exchange = await self._find_idempotent_exchange(
    conversation_id=conversation_id,
    parent_message_id=parent_message.id if parent_message else None,
    idempotency_key=idempotency_key,
)
```

```python
user_message = await self.messages.create_user_message(...)
assistant_message = await self.messages.create_assistant_placeholder(...)
stream_id = self._build_stream_id(assistant_message.id)
```

这里有个值得注意的四层机制：

```text
1. parent 解析：支持显式 parent_message_id，同时兼容旧 active_leaf
2. base message 级锁：保证同一 parent 下写 sibling 时串行化
3. 幂等复用：同一 parent + 同一 idempotency_key 命中旧 exchange 时直接复用
4. 会话级缓存更新：active_stream_* 与 has_active_run 在启动阶段立即就绪
```

这段实现最终解决了三个关键问题：

```text
1. 普通发送不再隐式绑定 conversation.active_leaf
2. 同一 base_message 下重复请求优先复用而不是重复落树
3. 新 assistant placeholder 创建后立即拥有自己的 stream_id
```

消息写入真正依赖的底层仓储逻辑在 `backend/app/repositories/conversation.py:252` 和 `backend/app/repositories/conversation.py:285`。`create_user_message` 与 `create_assistant_placeholder` 都会从 `ConversationRepository.allocate_message_sequence_no` 原子领取顺序号，并在有 parent 的情况下同步刷新 `parent.active_child_message_id`。

同时，请求阶段还会更新会话级聚合状态：

```python
conversation.active_stream_id = stream_id
conversation.active_stream_message_id = assistant_message.id
conversation.has_active_run = True
```

需要注意的是，这里的 `active_stream_*` 更像会话级兼容指针，真正的恢复入口已经下沉到了 message / stream 级。

### 5.3 `_resolve_parent_message`、`_find_idempotent_exchange` 与仓储层查询：写入前的判定逻辑

并行发送真正难的部分不是“写消息”，而是“在写之前如何稳定判定当前语义”。

这部分核心逻辑集中在 `backend/app/services/conversation.py:1080` 之后。

`_resolve_parent_message` 负责把“显式分支发送”与“旧 active_leaf 兼容”统一起来：

```python
if parent_message_id is None:
    return await self._get_active_leaf(conversation)
```

这意味着系统在过渡期里同时支持：

```text
新前端：显式 parent_message_id
旧调用方：回退到 active_leaf
```

`_find_idempotent_exchange` 则把普通发送的幂等复用逻辑拆成了两步：

```python
user_message = await self.messages.find_user_message_by_idempotency(...)
assistant_message = await self.messages.get_first_assistant_child(...)
```

它的实际判断语义是：

```text
先按 conversation + parent + idempotency_key 找 user message
再从这个 user message 下恢复它当前对应的 assistant child
```

而 regenerate 的幂等更严格，`backend/app/services/conversation.py:1137`：

```text
不仅要看 parent_message_id
还要看 source_message_id 和 revision_type
```

这是为了避免：

```text
同一个 parent 下不同 regenerate 请求被错误串成同一条幂等记录
```

### 5.4 `_produce_stream`：后台任务如何把一次模型调用收口成稳定状态

真正的运行主循环在 `backend/app/services/conversation.py:1368`。

它的第一步不是立刻读模型，而是先恢复运行上下文：

```python
assistant_message = await self.messages.get_by_id(...)
conversation = await self._get_conversation_by_id(conversation_id)
history_messages = await self.messages.list_path_to_message(
    conversation_id=conversation_id,
    message_id=assistant_message.parent_message_id,
)
```

也就是说，producer 的上下文来源不是“前端当前页面状态”，而是：

```text
assistant placeholder
  -> parent_message_id
  -> 这条 parent 所在路径的历史消息
```

然后 producer 才会：

```python
await conversation_stream_store.attach_producer_task(stream_id)
async for chunk in self.llm_client.stream(...):
```

在流式循环里，正文和 reasoning 被拆成两类事件分别发送：

```python
event="message.reasoning_delta"
event="message.delta"
```

这样前端就可以把 thinking 和正文分开渲染，而不是等到最后一次性拿完整消息。

这段代码最值得注意的是它的三种主要收口路径：

```text
1. 正常完成 -> complete_assistant_message + message.completed
2. 主动取消 -> cancel_assistant_message + message.cancelled
3. 模型错误 -> fail_or_partial_streaming_message + message.failed
```

并且三种路径在发最终事件前，都会先走 `_finalize_stream_conversation_state`，所以会话级状态总是在消息级状态之后被收敛回正确值。

### 5.5 `_resolve_message_active_stream` 与恢复接口：为什么恢复是 branch-aware 的

恢复的关键在于 `backend/app/services/conversation.py:1157` 这一段 branch 解析逻辑。

```python
candidate = message
if message.role != "assistant":
    candidate = await self.messages.resolve_active_leaf_from(
        conversation_id=conversation_id,
        message=message,
    )
```

它表达的语义是：

```text
如果传进来的是 user message 或 branch 上的中间节点
就沿 active_child 链继续往下找这个子树当前可见的 leaf
```

然后再判断这个 leaf 是否真的是一条正在 `streaming` 的 assistant：

```python
if candidate.role != "assistant" or candidate.status != "streaming":
    raise HTTPException(...)
```

最后才通过：

```python
stream = await conversation_stream_store.get_stream_by_message_id(candidate.id)
```

把 branch 上的 active assistant 映射到真正的 runtime stream。

这就是为什么现在的恢复能力是：

```text
message / branch scoped lookup
  -> stream scoped subscribe
```

而不是旧时代的：

```text
conversation scoped single active stream
```

### 5.6 `_finalize_stream_conversation_state`：会话级缓存如何避免漂移

并行场景下最容易漂移的状态，其实不是消息本身，而是 conversation 上的缓存字段。

这部分核心逻辑在 `backend/app/services/conversation.py:1632`：

```python
if conversation.active_stream_id == stream_id and conversation.active_stream_message_id == message_id:
    conversation.active_stream_id = None
    conversation.active_stream_message_id = None
has_active_run = await self.conversations.recompute_has_active_run(conversation.id)
```

这里有两个很关键的保护。

第一，只在 conversation 当前指向的还是“本条流”时，才清空 `active_stream_*`：

```text
避免旧流结束时把更晚启动的新流指针误删掉
```

第二，`has_active_run` 从不靠本地推测减一，而是每次都重新扫描数据库里的 streaming assistant 真相：

```text
只要还有任意 branch 在 streaming，就保持 true
所有 branch 都结束后，才切回 false
```

如果布尔值发生跳变，还会追加：

```python
event="conversation.run_state_changed"
```

这样前端侧边栏既能实时更新，又不会因为并行分支交错完成而出现状态抖动或长期漂移。

### 5.7 取消接口：按 message 定向取消 producer

取消接口的真实实现现在是 `backend/app/api/v1/conversations.py:424`：

```python
@router.post(
    "/{conversation_id}/messages/{message_id}/cancel",
    response_model=MessageRead,
    status_code=status.HTTP_202_ACCEPTED,
)
```

服务层实现见 `backend/app/services/conversation.py:1017`：

```python
did_signal = await conversation_stream_store.cancel(message_id=message_id)
if did_signal:
    return await self._message_read(message)
```

如果 runtime store 中还能找到这条流，就先给 producer 发取消信号；producer 在运行过程中检测到取消后，会把 assistant message 落成 `cancelled`，再发出 `message.cancelled` 事件。

如果运行时流已经不存在，则服务层会兜底直接更新数据库状态：

```python
message = await self.messages.cancel_assistant_message(...)
```

这样做的好处是：

```text
订阅者断开不等于取消
取消一条 branch 不等于取消整个 conversation
即使 runtime store 丢失，也能避免数据库遗留 streaming 脏状态
```
