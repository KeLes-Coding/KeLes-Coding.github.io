# 4. MVP 流式消息实现：SSE / 断点重连 / 前端增量渲染

这一篇直接记录 Orbit 在 v0.1 阶段对流式消息的设计方案。

目标很简单：用户发送消息后，不要等整段回答生成完才一次性返回，而是让 assistant 的内容一边生成、一边显示；同时在页面刷新、前端重连时，尽可能继续跟随当前仍在进行的那条生成流。

本文主要讨论三件事：

```text
1. 后端如何把一次 assistant 回复变成 SSE 事件流
2. 如何给流式消息预留断点重连能力
3. 前端如何对同一条 assistant 消息做增量渲染
```

这里先不讨论更复杂的能力：

```text
多 Agent 运行时
工具调用事件流
RAG
多模态消息
跨进程分布式流存储
```

## 1. 设计目标

在最早的一问一答实现里，会话链路是：

```text
用户发送消息
  -> 写入 user message
  -> 创建 assistant 占位消息
  -> 调用模型
  -> 等待模型完整返回
  -> 更新 assistant.content
  -> 返回完整 JSON
```

这个方案实现简单，但用户体验一般。只要模型稍慢，前端就只能一直等待，直到拿到完整内容。

因此，这一阶段要改成：

```text
用户发送消息
  -> 写入 user message
  -> 创建 assistant 占位消息
  -> 立刻返回 SSE
  -> 后端持续产出 message.delta
  -> 前端把 delta 追加到同一条 assistant 消息
  -> 生成完成后更新最终状态
```

从产品角度看，这一版要解决的问题是：

```text
1. 用户尽快看到首个 token
2. assistant 回复过程可见
3. 页面刷新后尽量还能继续接当前生成
4. messages 表仍然是最终事实源
```

## 2. 总体方案

这一版的流式消息系统分成三层：

```text
模型生成层
  -> 持续产出文本增量

应用事件层
  -> 把增量包装成 Orbit 自己的事件协议

前端渲染层
  -> 按 message_id 把 delta 合并回 assistant 消息
```

其中，SSE 只负责传输；真正给前端消费的是 Orbit 应用事件。

### 2.1 基础链路

一次普通 assistant 回复的主链路如下：

```text
POST /conversations/{conversation_id}/messages/stream
  -> 校验用户与会话归属
  -> 写入 user message
  -> 创建 assistant placeholder
  -> 生成 stream_id
  -> 记录 active stream
  -> 返回 SSE
  -> 持续发送 message.created / message.delta / message.completed
```

这里仍然保留 assistant 占位消息，原因有两个：

```text
1. 前端可以立刻拿到真实的 assistant message_id
2. 即使流中断，数据库里也有一条明确的消息记录
```

### 2.2 应用事件协议

这一版不让前端直接消费底层模型的原始 chunk，而是先统一成 Orbit 自己的事件协议。

最小事件集合如下：

```text
message.created
message.delta
message.completed
message.failed
ping
```

一次正常回复的事件顺序大致是：

```text
message.created
  -> assistant 消息已经创建

message.delta
  -> assistant 正文增量

message.completed
  -> 回复完成
```

示例：

```text
event: message.created
data: {"message_id":"msg_2","role":"assistant","status":"streaming"}

event: message.delta
data: {"message_id":"msg_2","delta":"流式输出的核心是"}

event: message.delta
data: {"message_id":"msg_2","delta":"让前端尽早看到模型生成内容。"}

event: message.completed
data: {"message_id":"msg_2","status":"completed"}
```

这套协议有一个重要原则：

```text
前端只关心 Orbit 事件
前端不关心底层模型供应商的 chunk 结构
```

### 2.3 stream_id / event_id

为了支持断点重连，这一版从一开始就预留事件定位字段。

统一 envelope 可以写成：

```text
OrbitStreamEvent
  stream_id
  event_seq
  event_id
  event
  data
```

字段含义：

```text
stream_id
  -> 本次 assistant 生成流的唯一标识

event_seq
  -> 当前 stream 内递增序号

event_id
  -> SSE id，格式可为 {stream_id}:{event_seq}

event
  -> 应用事件类型

data
  -> 事件负载
```

在 MVP 阶段，最自然的做法是让：

```text
assistant_message.id
  -> 派生出 stream_id
```

这样事件和消息天然绑定，前端也容易理解。

## 3. 断点重连

这一版讨论的“断点重连”，不是指模型从某个 token 位置继续生成，而是指：

```text
前端连接断开后
  -> 重新订阅当前仍在进行的那条 stream
  -> 继续接后续事件
```

所以它解决的是：

```text
浏览器刷新
页面重新挂载
网络抖动后重新连接
同一会话的再次订阅
```

而不是：

```text
模型推理中断后继续从中断 token 恢复
```

这里需要先区分两种不同的中断场景。

```text
1. 刷新页面 / 网络断开 / 连接中断
2. 当前正在接流的聊天页面被切到后台
```

这两种情况表面上都叫“前端没在继续收流”，但 Orbit 对它们的处理语义并不完全一样。

### 3.1 两种中断场景

#### 3.1.1 刷新页面、断网、连接被动中断

这是最典型的 resume 场景：

```text
浏览器刷新
页面重新挂载
网络抖动
SSE 连接意外断开
```

在这种情况下，设计目标是：

```text
后端继续跑原来的生成任务
前端重新连上后
  -> 找回当前 active stream
  -> 继续订阅后续事件
```

也就是说，这一类场景对应的是标准的：

```text
replay / resume
```

#### 3.1.2 页面切到后台，前端主动不再持续接流

第二种情况不是“连接意外丢了”，而是前端本身有意做了限制。

当前 Orbit 的聊天流恢复和 live 订阅，主要围绕：

```text
当前正在查看的 active conversation
```

来工作。换句话说，真正持续维护接流的是前台聊天页；而侧边栏、抽屉、非当前会话视图，并不会一直替它保持同等级别的流消费。

代码上也能看到这个倾向：

```text
useConversations(..., { enableStreamResume: false })
```

已经明确用于 `SideNav` 和 `MobileDrawer`。而主聊天视图中的恢复逻辑，也只会在：

```text
hasUser
activeConversationId 存在
activeConversation.has_active_run = true
```

时触发。

因此，这一版需要明确一个产品语义：

```text
页面切到后台
  -> 不保证前端继续无条件保持 live 接流
  -> 但后端不应该因此停止生成
  -> 用户回到前台后，应通过 active stream 恢复当前可见会话的流
```

也就是说，这里的“后台”更接近：

```text
前端主动暂停持续消费
回前台后再 resume
```

而不是：

```text
后台页面也一直像前台一样完整消费所有流事件
```

这个限制需要写出来，因为它会直接影响用户预期：

```text
Orbit 当前保证的是：
  当前活跃聊天页可以恢复正在进行的流

Orbit 当前不保证的是：
  页面退到后台后，前端仍然无限期持续接收并渲染这条流
```

### 3.2 active stream 设计

要实现重连，关键是把“当前会话有没有活跃流”这件事单独记录下来。

可以抽象成：

```text
conversation
  -> active_stream_id

stream
  -> event log
  -> subscribers
  -> current status
```

完整生命周期如下：

```text
1. 用户发起 POST /messages/stream
   -> 创建 assistant 消息
   -> 创建 stream_id
   -> 保存 conversation.active_stream_id
   -> 开始发送 SSE

2. 前端正常接收 message.delta

3. 页面刷新 / 网络断开
   -> 当前连接断开

4. 前端重新连接
   -> GET /conversations/{conversation_id}/stream
   -> 后端读取 active_stream_id
   -> 若流仍活跃，则恢复订阅
   -> 若流已结束，则返回空结果或最终消息

5. 流完成
   -> assistant message 落最终状态
   -> 清空 conversation.active_stream_id
```

### 3.3 事件日志窗口

如果要让重连更稳一点，只靠“继续接后续事件”还不够，最好还要保留一个短暂事件窗口：

```text
新订阅者进入
  -> 先 replay 最近一段 event log
  -> 再切到 live 事件
```

这样即使前端在极短时间内断线，也不容易丢掉刚刚发出的几条 `message.delta`。

在 MVP 阶段，这个 event log 不一定要上 Redis，也不一定要长期持久化；先用进程内内存窗口就够了。重点是先把接口和协议形状留出来。

## 4. 前端增量渲染

前端的关键点不是“不断新增消息”，而是“不断更新同一条 assistant 消息”。

因此前端收到事件后的处理逻辑应该是：

```text
message.created
  -> 插入或确认 assistant 消息

message.delta
  -> 按 message_id 追加 content

message.completed
  -> 把 status 改为 completed

message.failed
  -> 把 status 改为 failed 或 partial
```

这样页面里始终只有一条 assistant 消息在增长，而不是多条碎片消息。

前端消息状态可以沿用：

```text
streaming
  -> 正在生成

completed
  -> 生成完成

failed
  -> 生成失败

partial
  -> 有部分内容，但未完整结束
```

### 4.1 为什么要保留占位消息

assistant placeholder 有几个实际作用：

```text
1. 前端能立刻知道本轮 assistant 的真实 message_id
2. delta 都能落到同一条消息上
3. 流途中断时，数据库里不会完全没有痕迹
4. 页面刷新时，也能先看到一条 streaming 中的 assistant 消息
```

## 5. 消息状态更新

这一版仍然坚持：

```text
messages 表是最终事实源
SSE 只是过程中的实时视图
```

因此 assistant 消息的状态流转可以定义为：

```text
创建占位
  -> streaming

正常完成
  -> completed

没有任何输出就失败
  -> failed

已经有部分输出后失败
  -> partial

用户主动取消
  -> cancelled
```

一个简单而稳妥的做法是：

```text
生成过程中
  -> 内存累计 full_content
  -> 实时通过 SSE 推给前端

生成结束
  -> 一次性写回 messages.content
  -> 更新最终 status
```

这样数据库写入压力小，逻辑也更容易收口。

## 6. 关键代码

这一版的关键代码不在于某个具体模型 SDK，而在于流式链路本身的四个锚点。

### 6.1 创建 assistant 占位并启动 stream

发送消息时，请求阶段要先完成：

```text
写入 user message
写入 assistant placeholder
生成 stream_id
记录 active_stream_id
返回 SSE
```

核心伪代码：

```python
async def start_stream_message(conversation_id: str, content: str):
    user_message = create_user_message(conversation_id, content)
    assistant_message = create_assistant_placeholder(conversation_id, parent=user_message)
    stream_id = f"stream_{assistant_message.id}"
    set_conversation_active_stream(conversation_id, stream_id)
    return subscribe_sse(stream_id)
```

这里最重要的是顺序：

```text
先落库 user / assistant
再启动流
```

这样前端拿到的每条 delta 才有明确的消息归属。

### 6.2 后端事件生产

模型层只要能持续吐出文本增量，后端就负责把它包装成 Orbit 事件：

```python
yield event("message.created", {
    "message_id": assistant_message.id,
    "status": "streaming",
})

async for delta in model_stream():
    full_content += delta
    yield event("message.delta", {
        "message_id": assistant_message.id,
        "delta": delta,
    })

yield event("message.completed", {
    "message_id": assistant_message.id,
    "status": "completed",
})
```

如果失败：

```python
yield event("message.failed", {
    "message_id": assistant_message.id,
    "status": "failed",
})
```

### 6.3 重连恢复

恢复接口的职责是：

```text
根据 conversation.active_stream_id
  -> 找到仍活跃的 stream
  -> 恢复订阅
```

伪代码：

```python
async def resume_stream(conversation_id: str):
    stream_id = get_conversation_active_stream(conversation_id)
    if not stream_id:
        return no_content()
    return subscribe_sse(stream_id)
```

如果还保留了短窗口 event log，则可以：

```text
先 replay 最近事件
再继续 live subscribe
```

### 6.4 前端按 message_id 合并 delta

前端核心逻辑是：

```python
if event == "message.created":
    upsert_assistant_message(message_id, status="streaming")

if event == "message.delta":
    append_message_content(message_id, delta)

if event == "message.completed":
    mark_message_completed(message_id)
```

这里不能把每个 delta 当成独立消息，而必须始终围绕同一条 `assistant message` 做增量更新。

## 7. 结论

这一版流式消息设计的核心，不是引入某个特定模型框架，而是先把 Orbit 自己的流式协议和会话运行时边界立住：

```text
assistant 占位消息
SSE 事件流
stream_id / event_id
active stream 恢复
前端按 message_id 增量渲染
```

这样后面不管底层模型调用怎么变，甚至将来是否接入更复杂的运行时，流式消息这层的产品体验和协议边界都已经提前稳定下来了。
