# 4. MVP 流式消息实现：LangChain Streaming / SSE / 前端增量渲染

本文记录 Orbit MVP 阶段的流式消息实现计划。梳理如何把最初一问一答式的非流式链路改造成可实时增量的聊天链路。

相关的通用方案、应用级事件协议和断点续传讨论，已经放在《流式传输的实现》[^streaming-note]笔记中。本文只聚焦 v0.1 的落地边界：使用 LangChain Streaming 获取模型增量输出，使用 SSE 把 Orbit 内部事件推给前端，并在生成结束后继续以 `messages` 表作为会话消息事实源。

当前版本的改造目标是：

```text
用户发送消息
  -> 写入 user 消息
  -> 创建 assistant streaming 占位
  -> LangChain astream 持续产出 chunk
  -> 后端转换为 OrbitStreamEvent
  -> SSE 推送 message.delta
  -> 前端增量渲染 assistant 消息
  -> 结束后更新 messages.content 和 messages.status
```

以下能力不属于 MVP 阶段：

```text
断点续传
Redis stream
持久化 event log
同一条 assistant 消息的多次 stream attempt
工具调用事件流
Agent Run 事件流
```

本阶段计划实现一部分 resume streams 结构基础：

```text
stream_id
event_seq
event_id
```

这些字段在 MVP 阶段不用于恢复连接，也不要求持久化到 Redis 或 event log。

```text
MVP 阶段
  -> 实时流式推送

后续阶段：
  -> 断线后的事件补发
  -> active stream 恢复
  -> last_event_id 查询

协议预留
  -> 事件先带 stream_id / event_seq / event_id
  -> 后续如果接入 resume streams，可以复用同一套事件协议
```

## 2. 实现思路

当前后端已经具备基础的一问一答的非流式链路：

```text
POST /conversations/{conversation_id}/messages
  -> ConversationService.create_user_message
  -> 写入 user message
  -> 写入 assistant streaming 占位
  -> LLMClient.generate
  -> LangChain model.ainvoke
  -> 更新 assistant message 为 completed / failed
  -> 返回 user_message + assistant_message
```

当前代码里的 `streaming` 主要还是消息状态上的占位，并没有把模型生成过程中的 token 增量实时推给前端。

流式传输的实现计划在现有链路上做一层改造：保留 `messages` 作为事实源，保留 `assistant` 占位消息，但把原来一次性的 `ainvoke` 改成 LangChain 的流式调用，并通过 SSE 返回 Orbit 自己定义的应用事件。

目标链路如下：

```text
用户发送消息
  -> 写入 user 消息
  -> 创建 assistant streaming 占位
  -> 返回 SSE 事件流
  -> LangChain 持续产出 chunk
  -> 后端转换为 OrbitStreamEvent
  -> 前端增量更新 assistant 消息
  -> 模型结束后更新数据库中的 assistant 消息
```

### 2.1 基础链路

当前发送消息入口位于 `backend/app/api/v1/conversations.py`：

```text
POST /conversations/{conversation_id}/messages
```

这个接口目前返回 `MessageExchangeRead`，也就是一次请求结束后返回本轮的 `user_message` 和 `assistant_message`。

要支持真正的流式传输，新增一个流式发送接口：

```text
POST /conversations/{conversation_id}/messages/stream
```

基础流程如下：

```text
请求进入
  -> 校验当前用户是否拥有 conversation
  -> 读取 conversation.llm_config_id 或用户默认模型配置
  -> 校验模型配置可用
  -> 写入 user 消息，status = completed
  -> 写入 assistant 占位消息，status = streaming，content = ''
  -> commit，让前端刷新或断线后也能看到占位消息
  -> 启动模型流式调用
  -> 持续 yield SSE 事件
```

这里仍然沿用现有 `ConversationService.create_user_message` 的核心顺序，只是返回值从普通 JSON 改成 `StreamingResponse`。

后端不应该等模型完整生成后才返回响应，而是应该在 assistant 占位消息创建成功后立即开始输出事件。这样前端可以先拿到真实的 `message_id`，后续所有 delta 都基于这个 `message_id` 更新同一条 assistant 消息。

### 2.2 LangChain 流式调用

当前 `backend/app/services/llm_client.py` 中的模型调用逻辑是：

```text
LLMClient.generate
  -> _build_langchain_messages
  -> provider.build_chat_model
  -> model.ainvoke(messages)
  -> LLMCompletion
```

流式实现在 `LLMClient` 中新增一个方法：

```text
LLMClient.stream
  -> _build_langchain_messages
  -> provider.build_chat_model
  -> async for chunk in model.astream(messages)
  -> yield LLMStreamChunk
```

其中，`_build_langchain_messages` 可以继续复用现有逻辑：

```text
summary
  -> 作为 SystemMessage 注入

messages
  -> 只使用 completed / partial
  -> 跳过 streaming / failed
  -> 转换成 HumanMessage / AIMessage / ToolMessage
```

这样流式调用不会改变现有的上下文组装规则，也不会破坏 `messages` 作为事实源的设计。

先定义一个内部 chunk 结构，避免服务层直接依赖 LangChain 的原始返回形状：

```text
LLMStreamChunk
  content_delta
  response_metadata
  token_usage
  finish_reason
```

MVP 阶段只需要处理文本增量：

```text
AIMessageChunk.content
  -> normalize
  -> content_delta
```

如果 `content` 是字符串，就直接作为 delta；如果是列表，则沿用当前 `_normalize_content` 的思路，只提取其中的文本部分。

模型流结束后，后端把累计的完整文本写回 assistant 消息：

```text
assistant.content = full_content
assistant.status = completed
assistant.token_usage = token_usage
assistant.response_metadata = response_metadata
```

如果流式调用过程中抛出异常，则根据是否已经产生内容决定状态：

```text
没有任何 delta
  -> status = failed

已有部分 delta
  -> status = partial
```

### 2.3 SSE 事件返回

FastAPI 可以通过 `StreamingResponse` 返回 SSE：

```text
StreamingResponse(
  event_generator(),
  media_type="text/event-stream"
)
```

SSE 只是传输层，真正给前端消费的是 Orbit 自己的事件协议。MVP 阶段可以定义如下事件：

```text
message.created
message.delta
message.completed
message.failed
ping
```

一次普通回复的事件顺序如下：

```text
message.created
  -> 告诉前端 assistant message 已创建

message.delta
  -> 每次发送一小段新增文本

message.completed
  -> 告诉前端生成完成，并携带最终状态
```

示例：

```text
event: message.created
data: {"message_id":"...","role":"assistant","status":"streaming"}

event: message.delta
data: {"message_id":"...","delta":"流式传输的核心是"}

event: message.completed
data: {"message_id":"...","status":"completed"}
```

后端提供一个小的 SSE 编码函数，把内部事件统一格式化：

```text
OrbitStreamEvent
  stream_id      本次生成流 ID
  event_seq      stream 内递增事件序号
  event_id       SSE id， stream_id + event_seq
  event          应用事件类型
  data           事件负载
```

其中：

```text
stream_id
  本次生成流的 ID。MVP 阶段由 assistant message id 派生。
  例如创建 assistant 占位消息后，使用 `stream_${assistant_message.id}`。
  这样唯一性由 messages.id 保证，不需要额外设计时间戳或随机后缀。

event_seq
  本次 stream 内递增序号，从 1 开始。

event_id
  SSE 的 id 字段，可以先使用 `${stream_id}:${event_seq}`。
  MVP 阶段不做断点续传，作为事件定位标识。

event
  Orbit 应用事件类型，例如 message.created / message.delta / message.completed。

data
  事件负载，不同事件类型对应不同 data 结构。
```
> [!note]
> 我的想法：把 `stream_id` 设计成 `conversation_id + sequence_no + 时间戳` 这类组合 ID？
> 
> 答：
> > 不建议把 `stream_id` 设计成 `conversation_id + sequence_no + 时间戳` 这类组合 ID。
> > 
> > 原因是 `conversation_id` 和 `messages.sequence_no` 更适合表达消息在会话内的位置，而不是表达一次运行中的流；时间戳也会引入精度、排序和冲突边界问题。

当前模型是一条 assistant 消息对应一次生成流，因此更自然的锚点是：

```text
assistant_message.id
  -> stream_id
  -> event_seq
  -> event_id
```

如果后续支持“同一条 assistant 消息多次生成尝试”，考虑引入：

```text
stream_attempt
```

或者单独的：

```text
message_streams
```

MVP 阶段先不把这个复杂度放进 `stream_id`。

SSE 输出时对应关系为：

```text
id: {event_id}
event: {event}
data: {json(data)}
```

示例：

```text
id: stream_xxx:1
event: message.created
data: {"stream_id":"stream_xxx","message_id":"...","conversation_id":"...","role":"assistant","status":"streaming"}

id: stream_xxx:2
event: message.delta
data: {"stream_id":"stream_xxx","message_id":"...","delta":"流式传输的核心是"}

id: stream_xxx:3
event: message.completed
data: {"stream_id":"stream_xxx","message_id":"...","status":"completed","finish_reason":"stop"}
```

这样 MVP 阶段只做实时 SSE；当后续做断点续传，不需要重写前端事件协议，只需要把 `stream_id`、`event_seq` 和 `event_id` 持久化到 Redis 或 event log。

服务端事件生成流程如下：

```text
yield message.created
async for chunk in LLMClient.stream(...)
  -> append delta 到 full_content
  -> yield message.delta
模型结束
  -> 更新 assistant message 为 completed
  -> yield message.completed
异常
  -> 更新 assistant message 为 failed / partial
  -> yield message.failed
```

如果模型生成时间较长，可以定期发送 `ping`，避免代理或浏览器认为连接空闲：

```text
event: ping
data: {}
```

### 2.4 前端增量渲染

当前前端发送消息位于 `frontend/src/composables/useOrbit.js` 的 `sendMessage`：

```text
创建 optimisticUser
创建 optimisticAssistant，status = streaming，content = 'Thinking...'
调用 conversationApi.sendMessage
接口返回后替换成本轮真实消息
```

改成 SSE 后，前端需要从“等待完整 JSON 返回”变成“边读边更新同一条 assistant 消息”。

基础流程如下：

```text
用户发送消息
  -> 本地先插入 user 乐观消息
  -> 调用 /messages/stream
  -> 读取 response.body
  -> 解析 SSE event
  -> message.created 时替换 assistant 临时 id
  -> message.delta 时把 delta 追加到 assistant.content
  -> message.completed 时把 status 改为 completed
  -> message.failed 时把 status 改为 failed / partial
```

由于当前项目已经有 `normalizeMessage` 和基于 `message.status === 'streaming'` 的 UI 样式，前端状态模型可以继续沿用：

```text
streaming
  -> 展示正在生成的 assistant 消息

completed
  -> 正常展示最终内容

failed
  -> 展示失败提示

partial
  -> 展示已生成内容，并提示生成未完整结束
```

前端 API 层也需要新增一个不会自动 `response.json()` 的请求方法。当前 `frontend/src/api.js` 中的 `apiRequest` 默认解析 JSON，不适合直接处理 SSE。可以新增：

```text
conversationApi.streamMessage(conversationId, content)
  -> fetch(...)
  -> return response.body reader
```

第一阶段可以手写一个简单的 SSE parser：

```text
按行读取
  -> 收集 event
  -> 收集 data
  -> 遇到空行后派发事件
```

后续如果事件类型变多，再把解析逻辑抽成独立 composable，例如 `useSseStream`。

### 2.5 消息状态更新

消息状态仍然以数据库中的 `messages.status` 为准。

一次流式回复建议采用如下状态变化：

```text
创建 assistant 占位
  -> status = streaming

模型正常结束
  -> status = completed

模型调用失败，且没有生成任何内容
  -> status = failed

模型调用失败，但已经生成部分内容
  -> status = partial

用户主动取消
  -> status = cancelled
```

现有 `MessageRepository` 已经有：

```text
create_assistant_placeholder
complete_assistant_message
fail_assistant_message
```

流式实现时可以补充两个仓储方法：

```text
append_assistant_delta
  -> 可选：边生成边把 content 追加进数据库

partial_assistant_message
  -> 写入已有 full_content，并把状态改为 partial
```

第一阶段不一定要每个 token 都写数据库。更稳妥的做法是：

```text
内存中累计 full_content
  -> SSE 实时推给前端
  -> 结束时一次性写入 messages.content
```

这样数据库写入压力较小，代码也更简单。

但如果要支持刷新页面后继续看到已经生成的部分内容，就需要更频繁地落库，或者引入独立的 event log / Redis stream：

```text
SSE delta
  -> 写入 stream event log
  -> 推送给前端
  -> 完成后合并为 messages.content
```

因此当前阶段可以先采用：

```text
MVP streaming
  -> SSE 实时返回 delta
  -> 完成后更新 messages
  -> 失败时保存 failed / partial

后续 resume streams
  -> 复用 stream_id / event_seq / event_id
  -> 持久化 event log
  -> 前端按 last_event_id 恢复
```

这样既能尽快实现可感知的流式体验，又不会过早引入 Redis、事件日志和断点续传复杂度。
