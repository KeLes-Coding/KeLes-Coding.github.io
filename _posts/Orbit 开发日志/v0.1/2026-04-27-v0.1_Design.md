# v0.1_Design

Orbit 是一个面向大模型应用开发的 Agent 项目。本文作为开发日志的第一篇，主要梳理 MVP （最小可行性产品）阶段的核心架构与数据模型设计。第一步的核心目标是构建一个比较坚实的底层基座。跑通“用户 ↔ 模型配置 ↔ 多轮会话 ↔ LangGraph 引擎”的基本业务逻辑闭环。

## 1. 功能介绍
MVP 阶段主要提供以下三个基础能力：

### 1.1 用户注册与登陆
系统需要具备多用户能力。当前版本聚焦核心业务逻辑，仅采用本地邮箱注册和登录机制。通过分配全局唯一的用户 ID 来隔离不同租户间的私有数据资源。

### 1.2 模型配置与接入
避免将系统绑定在单一的大模型供应商上。用户可以自定义 LLM 接入配置，包括选择 Provider（OpenAI、Anthropic、Ollama 等）、指定具体模型名称、配置自定义 Base URL（以支持代理网关或本地模型），并注入个人的 API Key。支持多套配置并存与快速切换。

### 1.3 会话管理
提供类似 ChatGPT 的多轮对话管理能力。支持会话的创建与查询、消息状态（生成中、已完成、失败等）的追踪；并针对长文本会话引入基于 token 或消息数阈值的摘要压缩（Summarization）机制，防止上下文超出模型窗口限制。

## 2. 数据模型设计
为了支撑上述功能，我们在 PostgreSQL 中设计了四张核心业务表：`users`、`llm_configs`、`conversations` 和 `messages`。

核心 ER 关系为：**用户**拥有**多套模型配置**和**多个会话**，**会话包含多条消息记录**。

### 2.1 `users`

`users` 保存系统用户的基础身份信息。
它是 `llm_configs` 和 `conversations` 的归属主体，用来完成 MVP 数据模型闭环。

```sql
create table users (
  id uuid primary key default gen_random_uuid(),

  email varchar(255) not null,
  display_name varchar(100),

  avatar_url text,

  metadata jsonb not null default '{}'::jsonb,

  is_enabled boolean not null default true,
  archived_at timestamptz,

  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now(),

  constraint ck_users_email_nonempty
    check (length(trim(email)) > 0)
);

create unique index uq_users_email_active
on users(lower(email))
where archived_at is null;

create unique index uq_users_auth_identity_active
on users(auth_provider, auth_subject)
where auth_provider is not null
  and auth_subject is not null
  and archived_at is null;

create index idx_users_enabled
on users(is_enabled)
where archived_at is null;
```

字段说明：

| 字段              | 说明                                                  |
| --------------- | --------------------------------------------------- |
| `id`            | 用户 ID                                               |
| `email`         | 用户邮箱，用作主要登录标识或展示标识                                  |
| `display_name`  | 用户展示名                                               |
| `avatar_url`    | 用户头像地址                                              |
| `metadata`      | 用户扩展信息                                              |
| `is_enabled`    | 是否启用，便于临时禁用账号                                       |
| `archived_at`   | 软删除时间                                               |
| `created_at`    | 创建时间                                                |
| `updated_at`    | 更新时间                                                |

## 设计说明

`users` 是当前 MVP 的顶层业务实体。主要承担三个职责：

```text
1. 标识用户
2. 作为 llm_configs 的归属主体
3. 作为 conversations 的归属主体
```

### 2.2 `llm_configs`

`llm_configs` 保存用户的 LLM 接入配置。
它记录某个用户如何连接某个模型服务。

后端根据这张表构造 LangChain chat model，例如 OpenAI、Anthropic、Google GenAI、Ollama 或 OpenAI-compatible provider。

```sql
create table llm_configs (
  id uuid primary key default gen_random_uuid(),

  user_id uuid not null references users(id) on delete cascade,

  name varchar(100) not null,
  provider varchar(50) not null,
  model varchar(120) not null,

  base_url text,

  api_key_ciphertext text,
  credential_ref text,

  provider_options jsonb not null default '{}'::jsonb,

  is_default boolean not null default false,
  is_enabled boolean not null default true,
  archived_at timestamptz,

  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now(),

  constraint ck_llm_configs_provider_nonempty
    check (length(trim(provider)) > 0),

  constraint ck_llm_configs_model_nonempty
    check (length(trim(model)) > 0)
);

create unique index uq_llm_configs_user_name_active
on llm_configs(user_id, name)
where archived_at is null;

create unique index uq_llm_configs_user_default_active
on llm_configs(user_id)
where is_default = true and archived_at is null;

create index idx_llm_configs_user_enabled
on llm_configs(user_id, is_enabled)
where archived_at is null;
```

字段说明：

| 字段                   | 说明                                                                        |
| -------------------- | ------------------------------------------------------------------------- |
| `id`                 | 模型配置 ID                                                                   |
| `user_id`            | 配置所属用户                                                                    |
| `name`               | 用户自定义展示名，例如“OpenAI 主账号”“本地 Ollama”                                        |
| `provider`           | 模型供应商，例如 `openai`、`openai_compatible`、`anthropic`、`google_genai`、`ollama` |
| `model`              | 实际传给 LangChain / provider SDK 的模型名                                        |
| `base_url`           | 自定义 API 地址，支持代理网关、OpenAI-compatible 服务和本地模型                               |
| `api_key_ciphertext` | 加密后的 API Key，不保存明文                                                        |
| `credential_ref`     | 外部密钥管理引用，后续可替代本地密文                                                        |
| `provider_options`   | Provider 特有参数，例如 `temperature`、`timeout`、`api_version`、`deployment_name`  |
| `is_default`         | 当前用户默认模型配置                                                                |
| `is_enabled`         | 是否启用，便于临时禁用失效配置                                                           |
| `archived_at`        | 软删除时间                                                                     |
| `created_at`         | 创建时间                                                                      |
| `updated_at`         | 更新时间                                                                      |

## 设计说明

`llm_configs` 的定位是**用户模型接入配置**。

会话中途如果用户切换模型，只应该影响后续调用，不应该改变历史消息的模型归属。因此，历史消息仍然需要在 `messages` 表里保存当次调用的 `provider` 和 `model` 快照。

`provider_options` 第一阶段可以保持一个 JSONB 字段，避免过早拆分。后续如果配置复杂，可以拆成：

```text
connection_options
generation_defaults
```

例如：

```json
{
  "connection": {
    "timeout": 60,
    "api_version": "2024-xx-xx",
    "deployment_name": "xxx"
  },
  "generation": {
    "temperature": 0.7,
    "max_tokens": 2048
  }
}
```

### 2.3 `conversations`

`conversations` 保存会话元信息。
它记录会话标题、所属用户、当前默认模型配置、聊天模式、摘要以及 LangGraph 使用的 `thread_id`。

> LangGraph checkpointer 使用 `thread_id` 作为保存和恢复 checkpoint 的关键标识。checkpointer 会基于 `thread_id` 存取 checkpoint。

```sql
create table conversations (
  id uuid primary key default gen_random_uuid(),

  user_id uuid not null references users(id) on delete cascade,

  thread_id text not null default gen_random_uuid()::text,

  title varchar(200),

  llm_config_id uuid references llm_configs(id) on delete set null,

  chat_mode varchar(32) not null default 'chat',

  summary text,
  summary_updated_at timestamptz,
  summary_message_count integer not null default 0,

  metadata jsonb not null default '{}'::jsonb,

  archived_at timestamptz,

  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now(),

  constraint ck_conversations_chat_mode
    check (chat_mode in ('chat', 'rag', 'agent', 'tool'))
);

create unique index uq_conversations_thread_id
on conversations(thread_id);

create index idx_conversations_user_updated
on conversations(user_id, updated_at desc)
where archived_at is null;
```

字段说明：

| 字段                      | 说明                                            |
| ----------------------- | --------------------------------------------- |
| `id`                    | 业务会话 ID                                       |
| `user_id`               | 会话所属用户                                        |
| `thread_id`             | LangGraph checkpointer 使用的 thread ID          |
| `title`                 | 会话标题                                          |
| `llm_config_id`         | 当前会话默认使用的模型配置                                 |
| `chat_mode`             | 聊天模式，MVP 使用 `chat`，后续可扩展 `rag`、`agent`、`tool` |
| `summary`               | 长会话压缩摘要                                       |
| `summary_updated_at`    | 最近一次摘要更新时间                                    |
| `summary_message_count` | 当前摘要覆盖到的消息数量                                  |
| `metadata`              | 会话扩展信息                                        |
| `archived_at`           | 软删除时间                                         |
| `created_at`            | 创建时间                                          |
| `updated_at`            | 更新时间                                          |

## 设计说明

`conversations.id` 是会话的业务会话 ID。
`conversations.thread_id` 是 LangGraph runtime 使用的 thread 标识。


```text
conversation.id = 产品侧会话 ID
thread_id       = LangGraph checkpointer 使用的运行时 ID
```

这样后续如果一个会话需要多个 graph thread，也不会被当前设计锁死。例如：

```text
chat thread
summary thread
background task thread
research thread
```

`summary` ：业务侧长会话压缩结果，保存的是产品侧可控、可查询、可迁移的上下文压缩结果。

LangGraph checkpoint 保存运行时 state。

### 2.4 `messages`

`messages` 保存会话内的消息。
它是 UI 展示、历史查询、导出、审计和成本统计的事实源。

```sql
create table messages (
  id uuid primary key default gen_random_uuid(),

  conversation_id uuid not null
    references conversations(id) on delete cascade,

  sequence_no integer not null,

  langgraph_message_id text,

  role varchar(32) not null,

  content text not null,

  content_parts jsonb not null default '[]'::jsonb,

  status varchar(32) not null default 'completed',

  llm_config_id uuid references llm_configs(id) on delete set null,

  provider varchar(50),
  model varchar(120),

  token_usage jsonb not null default '{}'::jsonb,
  response_metadata jsonb not null default '{}'::jsonb,

  created_at timestamptz not null default now(),

  constraint ck_messages_role
    check (role in ('system', 'user', 'assistant', 'tool')),

  constraint ck_messages_status
    check (status in ('streaming', 'completed', 'cancelled', 'failed', 'partial'))
);

create unique index uq_messages_conversation_sequence
on messages(conversation_id, sequence_no);

create index idx_messages_conversation_created
on messages(conversation_id, created_at);

create index idx_messages_conversation_sequence
on messages(conversation_id, sequence_no);

create index idx_messages_langgraph_message_id
on messages(langgraph_message_id)
where langgraph_message_id is not null;
```

字段说明：

| 字段                     | 说明                                                             |
| ---------------------- | -------------------------------------------------------------- |
| `id`                   | 消息 ID                                                          |
| `conversation_id`      | 所属会话                                                           |
| `sequence_no`          | 会话内消息顺序号                                                       |
| `langgraph_message_id` | LangChain / LangGraph message ID，可选保存，用于后续和 graph state 对齐     |
| `role`                 | 消息角色，使用 `system`、`user`、`assistant`、`tool`                   |
| `content`              | 消息正文，MVP 阶段主要使用纯文本                                             |
| `content_parts`        | 预留的结构化内容块，后续支持多模态、附件、工具结果、结构化输出                                |
| `status`               | 消息状态，例如 `streaming`、`completed`、`cancelled`、`failed`、`partial` |
| `llm_config_id`        | 生成该消息时使用的模型配置                                                  |
| `provider`             | 生成该消息时使用的 Provider 快照                                          |
| `model`                | 生成该消息时使用的模型名快照                                                 |
| `token_usage`          | Token 用量统计                                                     |
| `response_metadata`    | LangChain 或底层 SDK 返回的结构化元信息                                    |
| `created_at`           | 创建时间                                                           |

## 设计说明

`sequence_no` 用来保证会话内消息顺序稳定。
避免只依赖 `created_at` 排序，因为流式生成、重试、并发写入时，时间戳排序可能不够明确。

推荐读取消息时使用：

```sql
select *
from messages
where conversation_id = :conversation_id
order by sequence_no asc;
```

`langgraph_message_id` 是可选字段。
MVP 阶段可以不强依赖它，但先保留字段有价值。后续如果你需要删除、压缩、映射 LangGraph state 内部消息，就可以用它做关联。

`content_parts` 是扩展字段。
MVP 阶段所有消息都写入 `content`，`content_parts` 保持空数组。后续如果支持图片、文件、tool call、artifact、structured output，再逐步使用。

## 3. 实现思路

### 3.1 运行时状态与业务状态解耦

在本系统中，状态被明确分为两类：

业务事实状态：即我们设计的 4 张核心表，它是向前端提供历史回放、计费展示、导出的视图。

1. AI 运行时状态：存在 LangGraph 的 Checkpoint 中，它管理复杂 Agent 的内部流转、Tool Call 中间件以及大模型原生上下文。
2. 两者依靠 `conversations.thread_id` 作为唯一桥梁互相关联，系统不会通过解析业务表去强行还原 Agent 的运行现场。

### 3.2 流式生成（Streaming）状态机

大模型的响应是耗时且逐字返回的。我们在 `messages` 表中通过 `status` 字段实现了基础的状态机：

1. 用户的提问立即落库，状态设为 `completed。`
2. 服务端创建一条 Assistant 占位消息，状态设为 `streaming`，并立刻向前端返回该消息的 ID。
3. LangGraph 引擎开始运行，通过 SSE/WebSocket 将模型生成的文本 Chunk 实时推给前端展示。
4. 生成结束后，服务端在内存中完成内容拼接，更新该记录的 `content`，并将状态流转为 `completed`（若发生网络阻断或限流报错，则流转为 `partial` 或 `failed`）。

### 3.3 长会话上下文修剪（Summarization）

当单次会话长度持续增加，将面临 API Token 成本急剧上升和模型注意力涣散导致的幻觉的问题。
我们利用 `conversations.summary_message_count` 作为一个水位游标。当会话内的消息数量超过预设阈值（如 40 条），触发后台摘要逻辑：
将前 40 条历史记录压缩为一段摘要存入 `summary` 字段，并将游标更新为 40。下一次触发对话时，送入 LangGraph 的 Context 组装逻辑将变为：`System Prompt + Summary 文本 + 游标之后的新消息`。
