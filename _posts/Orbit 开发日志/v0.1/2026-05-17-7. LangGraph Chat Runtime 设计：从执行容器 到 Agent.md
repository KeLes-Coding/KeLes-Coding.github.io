# 7. LangGraph Chat Runtime 设计：从执行容器，到 Agent

Orbit 目前已经完成了简单的用户管理、模型配置、会话、消息树、流式消息、可恢复运行和 branch-aware 会话管理。到这一阶段，系统已经具备了比较稳定的产品侧会话模型。

但与此同时，底层执行内核仍然保留着一个明显的历史特征：

```text
会话产品模型已经逐渐复杂
执行内核却仍然偏向“单轮模型调用 + 扁平流式循环”
```

这会带来一个结构性问题：Orbit 的 Conversation / Message / Stream 基础设施已经准备好承接更复杂的 graph runtime，但当前的模型执行方式还没有完成这一步升级。

因此，本轮迭代没有直接把 DeepAgent、Search、Filesystem、Thought UI 一口气全部接进来，而是先做一件更底层但更关键的事[^deepagents-plan]：

```text
先把普通 Chat 执行链路迁入 LangGraph Runtime
先验证 Orbit 能否把 LangGraph 作为新的执行容器
```

本文只讨论这一阶段，也就是 Phase 1：LangGraph Chat Runtime。

## 1. 功能介绍

本阶段的目标是先解决一个基础问题：

```text
Orbit 当前的 Chat 执行链路，能否稳定跑在 LangGraph 这个执行容器里？
```

它主要解决三类问题。

### 1.1 统一 Chat 执行容器

在引入 LangGraph 之前，Orbit 的普通聊天执行主链路本质上还是：

```text
读取消息上下文
  -> 直接调用 LLMClient.stream()
  -> 在 service 内部手写 async for chunk 循环
  -> 逐步写 message.delta / message.reasoning_delta
  -> 收口 completed / failed / cancelled
```

这条链路没有错，但它更像是一个“会工作的工程实现”，而不是一个“能够继续长出复杂执行分支”的运行时底座。

LangGraph 阶段要做的第一件事，就是把这条链路迁入一个 graph runtime：

```text
User
  -> ChatGraph
      -> prepare_context
      -> call_model
      -> finalize_message
  -> Answer
```

### 1.2 保持现有产品模型稳定

Orbit 当前已经有比较明确的产品事实层：

- `Conversation`
- `Message`
- `conversation_stream_store`
- `thread_id`
- `langgraph_message_id`
- `message tree / branch / fork / replay / resume`

这一阶段不应该动这些模型的产品边界，而应该只是替换底层执行容器。

也就是说，本轮改造追求的是：

```text
换执行引擎
不换会话产品模型
```

### 1.3 为后续 Agent 能力预留 graph 落点

在没有统一 graph runtime 之前，很多后续能力都没有自然的插入点：

- memory compression 往哪里插
- route_execution 往哪里插
- planning / tool loop 往哪里插
- thought block 聚合往哪里插
- checkpoint / resume 要围绕什么 state 做

LangGraph 的引入是为了在后续将上述内容引入之前，将基础设施打牢。

## 2. 现状与问题

Orbit 在这一阶段之前，已经具备适合承接 LangGraph 的基础。

### 2.1 当前已经具备的基础

可以直接承接 graph runtime 的点主要有：

- `Conversation.thread_id`
- `Message.langgraph_message_id`
- `messages` 作为会话事实源
- `conversation_stream_store` 作为流事件 runtime store
- `stream_run.py` 作为后台 producer 编排入口
- `LLMClient` 作为 provider 归一化调用层

即，当前 Orbit 的情况不是“先上 LangGraph 再去设计会话系统”，而是：

```text
会话系统先成熟
LangGraph 再作为执行容器接入
```

### 2.2 当前执行内核的问题

如果继续让 `_produce_stream()` 保持“全能方法”的形态，问题会逐步放大：

```text
模型调用逻辑
流式 chunk 解析
SSE 事件映射
完成 / 失败 / 取消收口
全部堆在一个 service 方法里
```

这样做在 MVP 阶段是合理的，但到这一阶段会带来四个明显问题。

#### 2.2.1 无法自然分层

如果后续要加：

- `prepare_context`
- `route_execution`
- `agentic_chat`
- `finalize_message`

那么继续把所有逻辑写在一个扁平循环里，只会让 service 越来越长。

#### 2.2.2 无法明确区分执行边界

当前的逻辑更像：

```text
“一次模型流式调用”
```

而不是：

```text
“一条有状态、有节点边界、可 checkpoint 的执行流程”
```

这会导致后续很难回答：

- 现在执行到了哪一步
- 当前 state 长什么样
- 如果某一步失败，恢复语义该怎么定义

#### 2.2.3 无法承接后续 Agent 分支

如果直接把 DeepAgent 接到旧循环里，就会出现典型的耦合问题：

- 如果流式正文异常，不确定是 graph stream 适配问题，还是 agent loop 问题
- 如果消息收口异常，不确定是普通 chat 问题，还是 tool loop 问题
- 如果 thought block 显示异常，不确定是事件协议问题，还是 agent trace 问题

这也是为什么本轮必须先拆成两个阶段。

#### 2.2.4 无法把 checkpoint 作为一等能力引入

Orbit 当前已经有：

```text
SSE replay
resume subscription
branch-aware active stream 恢复
```

但这些能力都是事件传播层的恢复，不是 graph state 层的恢复。

如果要让系统真正开始围绕：

```text
state
node
checkpoint
resume
```

来组织执行过程，那么就必须先把执行内核迁入 LangGraph。

## 3. LangGraph Chat Runtime 设计

### 3.1 总体思路

这一阶段的目标架构如下：

```text
ConversationRunService / ConversationService
        ↓
LangGraphChatRuntime
        ↓
ChatGraph
  prepare_context
        ↓
    call_model
        ↓
 finalize_message
        ↓
conversation_stream_store + Message persistence
```

其中最重要的设计原则是：

```text
外层 service 继续负责产品级编排
LangGraph 负责执行流程本身
```

也就是说：

- API 层不重写
- Conversation / Message 模型不重写
- SSE store 不重写
- assistant message 的完成 / 失败收口语义不重写

变的是内部执行方式。

### 3.2 ChatGraph 接入

Phase 1 没有直接引入 route_execution，也没有引入 Agent 增强分支，而是只先实现一个最小的 graph 接入：

```text
prepare_context
  ↓
call_model
  ↓
finalize_message
```

三个节点的职责分别是：

#### 3.2.1 `prepare_context`

负责：

- 接收外层 service 已准备好的上下文
- 保留节点边界
- 为后续 memory compression / summary 注入预留位置

当前阶段它并不承担复杂逻辑。

#### 3.2.2 `call_model`

负责：

- 调用 `LLMClient.stream()`
- 消费已经归一化的 `LLMStreamChunk`
- 通过 `get_stream_writer()` 发射 graph 自定义事件
- 在 state 中累积 `response_text`、`reasoning_text`、`token_usage`、`response_metadata`

#### 3.2.3 `finalize_message`

当前阶段主要保留节点边界，不承担数据库写入。

数据库写入和 completed / failed / cancelled 收口，仍然由外层 `ConversationStreamRunService` 完成。

这样做的好处是：Phase 1 可以只替换执行容器，不一次性动太多产品层代码。

### 3.3 ChatState 设计

这一阶段的 graph state 只保留最小字段：

```text
conversation_id
assistant_message_id
stream_id
thread_id
llm_config_id
provider
model
input_messages
response_text
reasoning_text
token_usage
response_metadata
error
```

这里最重要的设计点有两个。

#### 3.3.1 `thread_id` 正式进入 runtime

在之前的产品模型中，`Conversation.thread_id` 已经存在，但主要还是“预留字段”。

LangGraph 阶段之后，它开始真正承担 graph config 的 thread 语义：

```text
conversation.id = 产品侧会话 ID
thread_id       = LangGraph checkpointer 使用的执行线程 ID
```

这一步非常关键，因为它意味着 Orbit 后续可以把 checkpoint / resume 建立在现有 Conversation 模型之上，而不是另起一套运行时标识体系。

#### 3.3.2 敏感配置不写入 state

这一阶段没有把 API Key、base_url 等敏感配置直接写进 `ChatState`。

真实模型配置通过 runtime 闭包注入：

```text
state 里保存：
  conversation / assistant / thread / model identity

闭包里注入：
  llm_config
  provider config
  API Key
  实际 stream_factory
```

这样可以避免未来 checkpoint 持久化时，把明文敏感信息直接带入 graph state。

### 3.4 StreamAdapter 设计

LangGraph 本身不是 Orbit 的对外协议。Orbit 需要的仍然是自己的 SSE 事件流。

因此，这一阶段新增了一个很关键的适配层：

```text
LangGraph custom event
  -> StreamAdapter
  -> conversation_stream_store
  -> message.delta / message.reasoning_delta / message.completed ...
```

`StreamAdapter` 的职责有两层：

#### 3.4.1 事件映射

把 graph 内部自定义事件，例如：

```text
content_delta
reasoning_delta
token_usage
response_metadata
finish_reason
```

映射为 Orbit 对外 SSE 事件。当前这层映射关系可以理解为：

```text
content_delta
  -> message.delta
  -> 用于前端逐字增量渲染 assistant 正文

reasoning_delta
  -> message.reasoning_delta
  -> 用于前端增量显示 thinking / reasoning 内容

token_usage
  -> 不单独发 SSE
  -> 只在 StreamAdapter 内部累积，供 completed / failed / cancelled 收口时落库

response_metadata
  -> 不单独发 SSE
  -> 只在 StreamAdapter 内部累积 provider / model / finish_reason 等元信息

finish_reason
  -> 不单独发 SSE
  -> 写入 response_metadata，最终随消息 completed 结果一起返回
```

也就是说，LangGraph 内部事件并不是全部一比一暴露给前端。当前 Phase 1 真正继续保持为“流式对外事件”的，主要还是：

```text
message.delta
message.reasoning_delta
```

而像 `token_usage`、`response_metadata`、`finish_reason` 这类运行时元信息，仍然更适合作为：

```text
最终消息收口的一部分
```

来处理，而不是作为高频流式事件不断广播。

#### 3.4.2 状态累积

除了写事件，它还负责累积：

```text
response_text
reasoning_text
token_usage
response_metadata
```

这样外层 service 在最终 completed / failed / cancelled 收口时，就不需要重新解析一次 chunk 流。

### 3.5 为什么先不接 Agent

这一步很重要，因为很多系统在引入 LangGraph 时，容易直接把 graph runtime 和 agent harness 混在一起上线。

Orbit 没有这么做，原因是要先验证下面几个更底层的问题：

```text
LangGraph 是否适合作为 Orbit Chat 的执行容器
现有 provider 调用是否能稳定进入 graph node
graph 自定义事件是否能稳定映射到 Orbit SSE
assistant message 的 completed / failed / cancelled 收口是否稳定
thread_id 是否已经开始形成 checkpoint 语义
```

如果这些基础问题还没有验证，就直接叠加 Agent，会让问题边界变得非常模糊。

所以 Phase 1 的核心取舍是：

```text
先验证 graph runtime
不在这一阶段引入 tool loop 和复杂中间态
```

## 4. 实现思路

### 4.1 外层 producer 由执行器过渡至编排器

`ConversationStreamRunService._produce_stream()` 在这一阶段依然负责：

- 读取 `stream`
- 读取 `conversation`
- 读取 `assistant_message`
- 读取 `llm_config`
- 读取 `history_messages`
- 创建 `StreamAdapter`
- 调用 `LangGraphChatRuntime.run_stream(...)`
- 最后做 completed / failed / cancelled 收口

也就是说，它从：

```text
执行器
```

转变成了：

```text
编排器 + 收口器
```

这一步是一个非常典型的架构演进：先把“做所有事的方法”拆成“控制流程的方法”和“真正执行的 runtime”[^phase1-report]。

### 4.2 普通 Chat 仍然复用现有 `LLMClient`

这一阶段没有重写 provider 适配层，而是继续让 LangGraph 节点消费：

```text
LLMClient.stream()
```

这样有两个直接收益：

#### 4.2.1 行为一致性

普通 chat 在引入 LangGraph 之后，仍然沿用之前已经稳定的：

- content 归一化
- reasoning 归一化
- token usage 提取
- finish reason 提取

这可以显著降低“换执行容器”本身带来的行为漂移。

#### 4.2.2 provider 适配层不需要跟着重做

Orbit 之前已经把 provider 差异封装在 `LLMClient` 中。

LangGraph 阶段没有必要再在 graph node 里直接接一遍 provider SDK。否则很容易出现：

```text
legacy chat 一套 provider 归一化
LangGraph chat 另一套 provider 归一化
```

这会让后续维护成本明显上升。

### 4.3 取消语义保持不变

Phase 1 没有重做取消协议，仍然沿用：

```text
conversation_stream_store.cancel()
```

只是把 cancel 检测点从外层平铺循环移进了 graph 节点内部。

这意味着：

- 前端不需要改取消接口
- stream runtime store 不需要重做
- producer task cancel 逻辑不需要重做

而用户可见层面，仍然是：

```text
message.cancelled
assistant status = cancelled
```

### 4.4 checkpoint 语义开始落地，但不急着做 durable execution

这一阶段已经开始使用：

```text
MemorySaver
thread_id
ChatState
```

但仍然只停留在内存 checkpointer。

这背后的取舍是：

```text
先验证 Orbit 的执行过程是否适合 graph/state/node 模型
再决定 durable checkpoint 要不要进入 SQLite / Postgres
```

换句话说，Phase 1 引入的是 checkpoint 语义，不是完整的持久化恢复能力。这里也需要和 Orbit 之前已经完成的 replay / resume 运行时能力区分开看[^resume-runtime]。

更直白一点说，这两层“恢复”恢复的根本不是同一种东西：

```text
replay / resume 运行时恢复
  -> 恢复的是“事件传播”
  -> 事件已经生成了，只是客户端断线后要补发、追平、继续订阅

LangGraph checkpoint 恢复
  -> 恢复的是“执行状态”
  -> 某个节点执行到一半或某一步执行完后，要从 state 快照继续往下跑
```

可以把它们理解成两条完全不同的链路：

```text
Orbit replay / resume
  = 后台任务还在跑
  = stream event log 还在
  = 新客户端连上来后，把缺失事件补给它

LangGraph checkpoint
  = 后台执行流程本身有 state 快照
  = 即使未来某一步失败、任务被中断，理论上可以从某个 graph 节点继续执行
```

所以当前 Orbit 已经完成的是：

```text
“客户端断了，怎么把同一条运行中的消息流重新接上”
```

而 LangGraph 这一阶段开始引入的是：

```text
“执行流程本身以后能不能按节点保存状态，并从中间继续”
```

前者更偏：

```text
流事件传播层
```

后者更偏：

```text
graph 执行状态层
```

也正因为如此，Phase 1 虽然已经把 `thread_id`、`ChatState`、`MemorySaver` 接进来了，但还不能简单理解成“Orbit 已经拥有 durable execution”。它现在只是先把：

```text
checkpoint 这套语义和接口
```

准备好了，真正的持久化恢复能力还要等后续把 checkpointer 从内存升级到 SQLite / Postgres。

## 5. 关键代码

### 5.1 `stream_run.py`：请求级编排如何切到 LangGraph runtime

这一阶段最关键的改造入口仍然是：

```text
backend/app/services/conversations/stream_run.py
```

这里的 `_produce_stream()` 不再自己扁平地 `async for chunk in ...`，而是负责：

```text
查库拿到 conversation / assistant_message / llm_config / history_messages
  -> 构建 ChatState
  -> 创建 StreamAdapter
  -> 创建 LangGraphChatRuntime
  -> 执行 runtime.run_stream(...)
  -> 按 final_state.error 做 completed / failed / cancelled 收口
```

对应代码如下：

```python
# backend/app/services/conversations/stream_run.py
effective_chat_mode = (
    getattr(assistant_message, "chat_mode", None) or conversation.chat_mode
)
if effective_chat_mode == "tool":
    await self._produce_stream_legacy(...)
    return

initial_state = self._build_langgraph_state(
    stream_id=stream_id,
    conversation_id=conversation_id,
    assistant_message=assistant_message,
    conversation=conversation,
    llm_config=llm_config,
    history_messages=history_messages,
    chat_mode=effective_chat_mode,
)

stream_adapter = StreamAdapter(
    stream_id=stream_id,
    message_id=assistant_message.id,
)

runtime = LangGraphChatRuntime(
    stream_factory=lambda: self.llm_client.stream(
        config=llm_config,
        messages=history_messages,
        summary=conversation.summary,
        model=_model,
        enable_tools=False,
    ),
    llm_invoke=_agent_llm_invoke,
    tool_runtime=self.llm_client.tool_runtime,
)

final_state = await runtime.run_stream(
    state=initial_state,
    stream_adapter=stream_adapter,
)
```

这段代码有三个设计点值得注意。

第一，`stream_run.py` 仍然保留产品级编排职责。它继续负责：

- 查询真实消息路径
- 选择当前模型配置
- 创建占位消息
- 最终把 assistant message 落库

LangGraph 并没有接管这些产品层语义。

第二，真正进入 graph 的不是 ORM 对象，而是 `ChatState`。这一层把 service 内部原本散落的参数，收敛成了一个可 checkpoint 的执行快照。

第三，真实模型配置没有直接塞进 state，而是通过闭包注入 `stream_factory`。这样后续即使把 checkpoint 持久化到 SQLite 或 Postgres，也不会把 API Key 等敏感信息写进 graph state。

### 5.2 `chat_runtime.py`：最小 ChatGraph 如何定义

当前 LangGraph runtime 的核心文件位于：

```text
backend/app/services/langgraph_runtime/chat_runtime.py
```

在 Phase 1 中，它先只定义最小 graph：

```python
# backend/app/services/langgraph_runtime/chat_runtime.py
builder = StateGraph(ChatState)

builder.add_node("prepare_context", self._prepare_context)
builder.add_node("route_execution", self._route_execution)
builder.add_node("normal_chat", self._normal_chat)
builder.add_node("agentic_chat", self._agentic_chat)
builder.add_node("finalize_message", self._finalize_message)

builder.add_edge(START, "prepare_context")
builder.add_edge("prepare_context", "route_execution")

builder.add_conditional_edges(
    "route_execution",
    self._route_decision,
    {
        "normal_chat": "normal_chat",
        "agentic_chat": "agentic_chat",
    },
)

builder.add_edge("normal_chat", "finalize_message")
builder.add_edge("agentic_chat", "finalize_message")
builder.add_edge("finalize_message", END)
```

虽然这份代码现在已经长成了 Phase 2 的五节点图，但 LangGraph 阶段真正验证的是两件事：

#### 5.2.1 Orbit 是否能把聊天执行显式表达成节点

在引入 graph 之前，普通 chat 更像：

```text
一个会不断累积局部变量的扁平循环
```

而在 graph 中，它第一次正式变成了：

```text
prepare_context
  -> normal_chat
  -> finalize_message
```

后续再引入 `route_execution` 与 `agentic_chat`，其实正是建立在这个最小结构之上。

#### 5.2.2 `normal_chat` 如何复用 Orbit 现有 LLM 归一化

`normal_chat` 节点没有重新接一遍 provider SDK，而是继续消费 `LLMClient.stream()` 已经归一化后的 `LLMStreamChunk`：

```python
async def _normal_chat(self, state: ChatState) -> dict:
    writer = get_stream_writer()
    stream_id = state.get("stream_id", "")

    accumulated_content: list[str] = []
    accumulated_reasoning: list[str] = []
    token_usage: dict = {}
    response_metadata: dict = dict(state.get("response_metadata") or {})

    async for chunk in self._stream_factory():
        if stream_id and await conversation_stream_store.is_cancelled(stream_id):
            return {..., "error": "cancelled"}

        if chunk.content_delta:
            accumulated_content.append(chunk.content_delta)
            writer({"type": "content_delta", "delta": chunk.content_delta})

        if chunk.reasoning_delta:
            accumulated_reasoning.append(chunk.reasoning_delta)
            writer({"type": "reasoning_delta", "delta": chunk.reasoning_delta})
```

这里的关键不是“把 delta 发出来”，而是：

- 继续复用 Orbit 现有的 provider 归一化规则
- graph 节点只处理统一的内部 chunk 结构
- 节点通过 `get_stream_writer()` 发内部事件，而不是直接写 SSE

这一步让 LangGraph 接入的代价明显小于“重写整套模型适配层”。

### 5.3 `state.py`：ChatState 如何成为 checkpoint 快照

LangGraph 阶段新增的最基础结构之一是：

```text
backend/app/services/langgraph_runtime/state.py
```

`ChatState` 当前定义如下：

```python
# backend/app/services/langgraph_runtime/state.py
class ChatState(TypedDict):
    conversation_id: str
    assistant_message_id: str
    stream_id: str
    thread_id: str

    llm_config_id: str
    provider: str
    model: str

    input_messages: list

    chat_mode: str
    execution_mode: str

    response_text: str
    reasoning_text: str
    token_usage: dict[str, Any]
    response_metadata: dict[str, Any]

    thought_events: list[dict[str, Any]]
    workspace_files: list[dict[str, Any]]

    error: str | None
```

从 Phase 1 的视角看，真正重要的是前半部分字段：

- `conversation_id`
- `assistant_message_id`
- `stream_id`
- `thread_id`
- `input_messages`
- `response_text`
- `reasoning_text`
- `token_usage`
- `response_metadata`
- `error`

这些字段把原本 service 内部的临时局部变量，变成了：

```text
一个完整的运行时状态快照
```

这意味着：

- 节点之间开始通过 state 传递数据，而不是靠闭包和局部变量串联
- `thread_id` 第一次真正开始承担 LangGraph 运行时 thread 语义
- checkpoint 未来可以围绕 `ChatState` 落地，而不是围绕零散的 service 变量落地

也就是说，`ChatState` 的价值不在于“字段多了”，而在于 Orbit 的聊天执行第一次有了一个明确的 graph state 形状。

### 5.4 `stream_adapter.py`：为什么 graph 节点不直接写 SSE

`StreamAdapter` 位于：

```text
backend/app/services/langgraph_runtime/stream_adapter.py
```

它的职责是把 graph 内部自定义事件，转换成 Orbit 当前对外的 SSE 协议：

```python
# backend/app/services/langgraph_runtime/stream_adapter.py
async def emit_custom_event(self, event: dict[str, Any]) -> None:
    event_type = event.get("type", "")

    if event_type == "content_delta":
        delta = event.get("delta", "")
        if delta:
            self._content_parts.append(delta)
            await conversation_stream_store.append_event(
                self._stream_id,
                event="message.delta",
                data={
                    "message_id": str(self._message_id),
                    "delta": delta,
                },
            )

    elif event_type == "reasoning_delta":
        ...

    elif event_type == "response_metadata":
        ...

    elif event_type == "finish_reason":
        ...
```

同时，它还负责累积内部状态：

```python
def get_accumulated_state(self) -> dict[str, Any]:
    full_content = "".join(self._content_parts)
    full_reasoning = "".join(self._reasoning_parts)

    return {
        "response_text": full_content,
        "reasoning_text": full_reasoning,
        "token_usage": self._token_usage,
        "response_metadata": self._response_metadata,
        "thought_events": self._thought_events,
    }
```

这里的设计意义很大。

如果 graph 节点直接写 SSE，那么执行层和传输层就会绑死在一起。后续只要协议有变化，就必须回过头改 graph node。

而在现在的结构中：

```text
graph 节点
  -> 发内部事件
  -> StreamAdapter
  -> conversation_stream_store
  -> SSE
```

这样后续无论是：

- 增加 thought 事件
- 增加 tool 事件
- 增加 run-level 事件

都可以优先在适配层里扩展，而不必让 graph 节点直接承担最终协议格式。

## 6. 总结

LangGraph Chat Runtime 阶段看起来只是“给普通聊天换了个执行壳”，但它实际上完成的是一次非常关键的基础设施升级：

```text
Orbit 从“会话系统已经复杂，但执行内核仍偏扁平”
升级为“会话系统稳定，执行内核也开始围绕 graph/state/checkpoint 组织”
```

这个阶段最大的价值，主要是结构层面的可扩展性。

它让 Orbit 具备了以下能力基础：

- 普通 chat 与 agent 增强 chat 可以共享同一个 graph runtime
- `thread_id` 开始真正承担运行时 thread 语义
- Chat 执行过程开始有节点边界和 state 快照
- SSE 协议开始可以通过 graph 自定义事件稳定扩展

也正因为这一步先做扎实了，DeepAgent 阶段才可以不必重新设计一套独立产品链路，而是直接作为：

```text
LangGraph Chat Runtime 内部的可选执行分支
```

进入现有会话体系。
