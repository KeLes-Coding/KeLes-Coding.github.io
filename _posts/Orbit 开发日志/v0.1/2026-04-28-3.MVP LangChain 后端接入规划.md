# 3. MVP LangChain 后端接入：多 Provider / 模型列表 / 聊天调用

本文记录 Orbit MVP 后端第二阶段的 LangChain 接入实现。

上一篇《MVP 后端实现：认证 / 模型配置 / 会话消息》[^mvp-backend-implementation]记录了认证、模型配置、会话与消息的一问一答闭环。

本文聚焦与 LangChain 引入的具体实现。

实现 LangChain 替换手写模型调用，并在后端抽象统一的 provider 层，支持多供应商聊天调用、模型列表获取、provider 能力描述和临时模型探测。

当前版本的模型调用链路是：

```text
用户发送消息
  -> ConversationService 读取会话与模型配置
  -> LLMClient 组装 LangChain messages
  -> provider registry 查找供应商实现
  -> provider 构造 LangChain chat model
  -> await model.ainvoke(messages)
  -> 统一提取 content / token_usage / response_metadata
  -> assistant 消息落库
```

当前版本的模型配置辅助链路是：

```text
前端读取 provider 列表
  -> 选择 provider
  -> 使用已保存配置获取模型列表
  或使用临时 api_key / base_url 探测模型列表
  -> 保存模型配置
  -> 会话聊天复用同一 provider 实现
```

以下能力不在本阶段范围内：

```text
LangGraph runtime
LangGraph checkpointer
RAG
Tool calling
Agent mode
流式响应
流式断点恢复
多模态消息完整链路
```

## 1. 后端分层变化

这一版主要是在 service 层新增 LLM runtime 分层。

新增和调整后的代码分布如下：

| 层级 | 目录 / 文件 | 职责 |
| --- | --- | --- |
| API | `backend/app/api/v1/llm_configs.py` | 暴露 provider 列表、已保存配置模型列表、临时模型探测接口 |
| Schemas | `backend/app/schemas/llm_config.py` | 定义 provider / model list / probe 的请求响应结构 |
| Services | `backend/app/services/llm_config.py` | 校验用户归属，调用 provider 获取模型列表 |
| LLM Client | `backend/app/services/llm_client.py` | 组装 LangChain 消息，统一聊天调用返回结构 |
| Provider Base | `backend/app/services/llm/providers/base.py` | 定义 provider 抽象、运行时配置、模型信息和错误类型 |
| Provider Registry | `backend/app/services/llm/providers/registry.py` | 管理 provider 实例和别名规范化 |
| Provider Impl | `backend/app/services/llm/providers/*_provider.py` | 适配不同供应商的 LangChain chat model 和模型列表 API |

后端依赖增加：

```text
langchain-core
langchain-openai
langchain-ollama
langchain-anthropic
langchain-google-genai
openai
anthropic
google-genai
httpx
```

这些依赖定义在：

```text
backend/pyproject.toml
```

## 2. Provider 抽象实现

Provider 抽象定义在：

```text
backend/app/services/llm/providers/base.py
```

核心目标是把不同供应商的 SDK 差异限制在 provider 层，让会话服务只面对统一入口。

### 2.1 运行时配置

数据库中的 `LLMConfig` 是 ORM 对象，不直接传给具体 SDK。Provider 会先把它转换成运行时配置：

```text
LLMConfig
  -> decrypt_secret(api_key_ciphertext)
  -> LLMRuntimeConfig
```

运行时配置字段：

```text
provider
model
base_url
api_key
provider_options
```

这样做有两个好处：

```text
1. 聊天调用可以使用已保存配置
2. 模型列表探测可以使用未保存的临时配置
```

两条链路最终都复用同一套 provider 代码。

### 2.2 provider_options 

这一版保留 `llm_configs.provider_options` 的 JSON 结构，不新增数据库字段。

当前推荐写法：

```json
{
  "connection": {
    "timeout": 60
  },
  "generation": {
    "temperature": 0.7,
    "max_tokens": 2048
  },
  "langchain": {},
  "extra_body": {}
}
```

同时继续兼容早期扁平写法：

```json
{
  "timeout": 60,
  "temperature": 0.7,
  "extra_body": {}
}
```

解析规则在 `BaseLLMProvider.parse_provider_options` 中完成：

```text
connection.timeout -> 请求超时
generation -> 传给 LangChain chat model 的生成参数
langchain -> 传给 LangChain integration 的额外参数
extra_body -> 传给 OpenAI-compatible 请求体的额外字段
```

如果 `connection`、`generation`、`langchain` 或 `extra_body` 不是对象，provider 层会抛出业务可读的 `LLMProviderError`。

### 2.3 Provider Registry

Provider registry 定义在：

```text
backend/app/services/llm/providers/registry.py
```

当前注册的 provider：

```text
openai
openai_compatible
ollama
anthropic
gemini
deepseek
qwen
```

同时支持常见别名：

```text
claude -> anthropic
google -> gemini
google_genai -> gemini
```

创建或更新模型配置时，`LLMConfigService` 会通过 registry 校验 provider，并把别名规范化后入库。

这样历史消息中的 provider 快照更稳定：

```text
用户输入 claude
  -> normalize_provider
  -> 实际保存 anthropic
```

## 3. 已支持的 Provider

当前 provider 实现分布如下：

```text
backend/app/services/llm/providers/openai_provider.py
backend/app/services/llm/providers/ollama_provider.py
backend/app/services/llm/providers/anthropic_provider.py
backend/app/services/llm/providers/google_provider.py
```

### 3.1 OpenAI / OpenAI-compatible

`OpenAICompatibleProvider` 是 OpenAI-compatible 协议的基础实现。

聊天调用：

```text
LLMRuntimeConfig
  -> require_api_key
  -> require_model
  -> resolve_base_url
  -> parse_provider_options
  -> ChatOpenAI(...)
```

模型列表：

```text
AsyncOpenAI(api_key, base_url, timeout)
  -> client.models.list()
  -> LLMModelInfo[]
```

官方 OpenAI 复用同一实现，只补默认地址：

```text
https://api.openai.com/v1
```

DeepSeek 和 Qwen 也复用 OpenAI-compatible 实现，并提供默认地址：

```text
deepseek -> https://api.deepseek.com
qwen -> https://dashscope.aliyuncs.com/compatible-mode/v1
```

因此前端可以把 DeepSeek / Qwen 作为独立 provider 展示，用户不需要手动记住 compatible base_url。

### 3.2 Ollama

Ollama provider 的特点：

```text
requires_api_key = false
default_base_url = http://127.0.0.1:11434
默认 timeout = 120
```

聊天调用使用：

```text
ChatOllama(model, base_url, timeout, ...)
```

模型列表不走 OpenAI-compatible `/models`，而是调用 Ollama 本地接口：

```text
GET /api/tags
```

返回后统一转换成 `LLMModelInfo`。

### 3.3 Claude / Anthropic

Anthropic provider 对外规范 ID 是：

```text
anthropic
```

聊天调用使用：

```text
ChatAnthropic(model_name, api_key, timeout, ...)
```

模型列表使用官方 SDK：

```text
anthropic.AsyncAnthropic(...).models.list()
```

`claude` 只作为输入别名存在，入库后统一保存为 `anthropic`。

### 3.4 Gemini

Gemini provider 对外规范 ID 是：

```text
gemini
```

聊天调用使用：

```text
ChatGoogleGenerativeAI(model, api_key, request_timeout, ...)
```

模型列表使用 `google-genai`：

```text
genai.Client(api_key).models.list()
```

`google-genai` 的模型列表接口是同步迭代接口，因此实现中通过 `asyncio.to_thread` 包装，避免阻塞 FastAPI 的事件循环。

Google 返回的模型名可能带 `models/` 前缀，后端会在返回前规范化成短 ID。

## 4. 模型配置接口扩展

这一版在原有 `/api/v1/llm-configs` 下新增三个能力接口。

| 方法 | 路径 | 说明 |
| --- | --- | --- |
| `GET` | `/api/v1/llm-configs/providers` | 返回后端支持的 provider 能力列表 |
| `GET` | `/api/v1/llm-configs/{config_id}/models` | 使用已保存配置获取模型列表 |
| `POST` | `/api/v1/llm-configs/models` | 使用临时配置探测模型列表，不保存 API Key |

所有接口仍然依赖当前登录用户。

### 4.1 Provider 列表

Provider 列表接口返回：

```json
[
  {
    "id": "openai",
    "name": "OpenAI",
    "requires_api_key": true,
    "supports_custom_base_url": true,
    "supports_model_list": true,
    "default_base_url": "https://api.openai.com/v1"
  }
]
```

这些信息来自 provider registry，而非数据库。

前端可以据此决定：

```text
是否展示 API Key 输入框
是否展示 base_url 输入框
是否展示“自动获取模型列表”按钮
是否预填默认 base_url
```

### 4.2 已保存配置的模型列表

已保存配置模型列表接口：

```http
GET /api/v1/llm-configs/{config_id}/models
```

服务层流程：

```text
校验 config_id 属于当前用户
  -> 根据 config.provider 查找 provider
  -> 解密 api_key_ciphertext
  -> provider.list_models(runtime_config)
  -> 返回 LLMModelRead[]
```

如果配置不存在，返回：

```text
404 模型配置不存在
```

如果 provider 不支持或配置缺少必要参数，返回：

```text
400 暂不支持的模型供应商 / 当前模型配置缺少 API Key / base_url 缺失
```

如果远端模型列表请求失败，返回：

```text
502 获取模型列表失败：...
```

### 4.3 临时模型探测

临时模型探测接口：

```http
POST /api/v1/llm-configs/models
```

请求结构：

```json
{
  "provider": "deepseek",
  "base_url": null,
  "api_key": "sk-...",
  "provider_options": {
    "connection": {
      "timeout": 60
    }
  }
}
```

这个接口用于配置尚未保存时获取模型列表。

关键规则：

```text
api_key 只用于本次请求
不会写入 llm_configs
不会返回给前端
不会经过 API schema 暴露密文
```

返回结构与已保存配置模型列表一致：

```json
[
  {
    "id": "deepseek-chat",
    "name": "deepseek-chat",
    "description": null,
    "owned_by": null
  }
]
```

## 5. 聊天调用实现

聊天调用统一入口仍然是：

```text
backend/app/services/llm_client.py
```

对上层暴露的方法没有扩大：

```text
LLMClient.generate(config, messages, summary)
```

这意味着 `ConversationService` 不需要知道具体 provider，也不需要关心 LangChain integration 的参数差异。

### 5.1 LangChain 消息组装

`LLMClient` 会把数据库消息转换成 LangChain message 类型。

转换规则：

| 数据库 role | LangChain message |
| --- | --- |
| `system` | `SystemMessage` |
| `user` | `HumanMessage` |
| `assistant` | `AIMessage` |
| `tool` | `ToolMessage` |

上下文过滤规则：

```text
只使用 status in completed / partial 的消息
只使用 role in system / user / assistant / tool 的消息
跳过空 content
```

如果会话存在 `summary`，会先注入一条 system message：

```text
以下是此前对话摘要，请在后续回复中作为上下文参考：
{summary}
```

当前阶段只是消费已有 `summary` 字段，不负责自动生成或更新摘要。

### 5.2 Provider 调用

`LLMClient.generate` 的核心流程：

```text
get_provider(config.provider)
  -> provider.from_model_config(config)
  -> provider.build_chat_model(runtime_config)
  -> await model.ainvoke(langchain_messages)
  -> normalize response
```

如果 provider 不存在，会抛出：

```text
暂不支持的模型供应商：{provider}
```

如果配置初始化失败，会抛出：

```text
模型配置初始化失败：...
```

如果远端请求失败，会抛出：

```text
模型服务请求失败：...
```

这些异常最终会被 `ConversationService` 捕获，并落到 assistant 消息的失败状态中。

### 5.3 返回结构统一

LangChain 不同 integration 的响应结构并不完全一致。

`LLMClient` 会统一收敛成：

```python
@dataclass
class LLMCompletion:
    content: str
    token_usage: dict[str, Any]
    response_metadata: dict[str, Any]
```

文本内容规范化规则：

```text
str -> 直接使用
list[str | dict] -> 提取字符串和 dict.text 后拼接
其他非空类型 -> str(content)
None -> 空字符串
```

如果规范化后为空，视为失败：

```text
模型服务没有返回 assistant 内容
```

token usage 提取优先级：

```text
response.usage_metadata
response_metadata.token_usage
response_metadata.usage
{}
```

`response_metadata` 中会补充：

```text
provider
model
raw_id
```

其中 `raw_id` 只有在 LangChain 响应对象存在 id 时才写入。

## 6. 会话消息落库行为

这一版保持上一篇文档中的消息落库策略：

```text
先写 user message
  -> 读取历史消息
  -> 写 assistant placeholder，status = streaming
  -> commit
  -> 调用 LangChain
  -> 成功：assistant status = completed
  -> 失败：assistant status = failed
  -> commit
```

成功时，assistant 消息保存：

```text
content
token_usage
response_metadata
provider
model
llm_config_id
```

失败时，assistant 消息不删除，而是更新为：

```json
{
  "status": "failed",
  "response_metadata": {
    "error": "模型服务请求失败：..."
  }
}
```

这个策略让前端始终能拿到本轮交互的两条消息：

```json
{
  "user_message": {},
  "assistant_message": {}
}
```

即使模型调用失败，用户消息仍然保留，assistant 消息也能展示失败原因。后续实现重试时，可以基于这条 failed assistant 消息继续扩展。

## 7. 当前实现边界

这一版已经完成：

```text
LangChain chat model 接入
provider 抽象和 registry
OpenAI / OpenAI-compatible / Ollama / Anthropic / Gemini / DeepSeek / Qwen 支持
provider 列表接口
已保存配置模型列表接口
临时配置模型探测接口
provider 别名规范化
provider_options 分组写法与扁平写法兼容
模型调用成功后的 content / token_usage / response_metadata 落库
模型调用失败后的 assistant failed 状态落库
```

仍然需要前端继续补齐的体验：

```text
配置页展示 provider 列表
根据 provider 能力动态展示 api_key / base_url 字段
自动获取模型列表
保留手动输入模型名能力
模型列表获取失败时展示可理解的错误
```

仍然需要后端后续单独规划的能力：

```text
流式响应
请求取消
同一会话并发发送消息的顺序控制
长会话自动摘要
LangGraph runtime / checkpointer
Tool calling
RAG
多模态 content_parts 的完整读写和模型调用
```

## 8. 小结

这一版更新的核心不是增加一个单独的模型供应商，而是把 Orbit 的模型调用入口从“具体 provider 的一次性代码”升级为“可扩展的 LangChain provider runtime”。

后端现在有三条稳定边界：

```text
ConversationService
  只负责会话、消息和模型配置选择

LLMClient
  只负责 LangChain 消息组装和统一返回结构

Provider
  只负责供应商差异、chat model 构造和模型列表获取
```

这个结构让 MVP 普通聊天链路保持简单，同时为后续流式响应、LangGraph、工具调用和更多模型供应商留下了清晰的接入点。
