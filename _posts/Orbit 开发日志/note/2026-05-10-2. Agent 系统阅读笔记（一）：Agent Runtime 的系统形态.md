# 2. Agent 系统阅读笔记（一）：Agent Runtime 的系统形态

最近集中阅读了有关 LangChain Agent、LangChain DeepAgents、Claude Code / Claw Code，以及 ChatGPT Memory 逆向分析的几篇相关博客。

一个比较直观的感受是，对于复杂任务型 Agent，尤其是 Coding Agent、Deep Research Agent 和长程自动化 Agent，系统形态正在从“LLM + function calling”过渡为围绕模型、工具、状态、权限、沙箱和观测能力构建的 Runtime System。模型负责推理，而工具、状态、上下文、文件系统、记忆、权限、沙箱、观测和前端交互，构成了 Agent 执行复杂任务的基础。

LangChain 文档对 Agent 的基础定义是：`An LLM Agent runs tools in a loop to achieve a goal.`，Agent 将语言模型和工具组合起来，使系统能够围绕任务进行推理、决定使用哪些工具，并通过循环逐步完成目标。Agent 会持续运行，直到模型输出最终答案，或者达到迭代上限。[^langchain-agents]

即当前的 Agent 与此前的偏向聊天场景的差异，体现在是否能够驱动一个可持续执行、观察结果、调整策略、继续行动的执行闭环。

```Mermaid
graph TD
    Input([input])
    Model{model}
    Tools(tools)
    Output([output])

    Input --> Model
    Model -->|tool call / action| Tools
    Tools -->|observation| Model
    Model -->|final answer| Output
```

## 1. 从 Chatbot 到 Agent Runtime

传统 Chatbot 的核心流程为：
```text
用户输入
  -> 拼接上下文
  -> 调用模型
  -> 返回文本
```

Agent 流程更接近：
```text
用户输入
  -> 构造任务上下文
  -> 模型规划下一步
  -> 调用工具
  -> 观察工具结果
  -> 更新状态
  -> 必要时继续调用工具
  -> 输出最终结果
```
关键的变化为，模型需要在一个 Runtime System 做持续的决策。

Claw Code 的架构拆分也能说明这一点。它把系统拆成多个 crate：[^claw-overview]
| 模块                     | 职责                                          |
| ---------------------- | ------------------------------------------- |
| CLI binary             | 处理 REPL、一次性 prompt、流式展示、命令行参数主二进制入口。REPL 循环、单次提示词分发、流式显示、CLI 参数解析。
| runtime                | 核心引擎。对话运行时、配置加载、会话持久化、权限策略、MCP 客户端生命周期、系统提示词组装。 |
| tools                  | 内置工具规范与执行。Bash、文件 I/O、搜索、Web 工具、Agent 接口、技能解析——共计 40 个暴露的工具规范。 |
| api                    | 模型 provider client、鉴权、请求预检、SSE streaming    |
| commands               | slash command 注册、解析和帮助输出                    |
| telemetry              | session trace 和 usage telemetry             |
| mock service / harness | 测试、mock、兼容性验证                               |

一个 Agent Runtime 至少可以拆成六层：

1. Model Loop：模型生成下一步行动意图
2. Tool Surface：Runtime 暴露给模型的行动接口
3. Tool Dispatch：工具注册、权限校验、执行和结果反馈
4. Context Assembly：把 memory、summary、task state、工具结果装配进模型上下文
5. Safety Boundary：权限、沙箱、HITL
6. Observability & Recovery：event log、checkpoint、trace、resume

在 Claude Code、Claw Code、ChatGPT Memory 等系统或逆向材料中，已经可以看到它们的具体形态：Coding Agent 需要文件系统、Shell、权限和任务状态；对话型 Agent 需要多层记忆和上下文压缩；复杂 Agent 则需要工具注册、执行追踪、恢复和人类确认。[^claw-overview][^llmrefs-memory][^deepagents-context]

## 2. Claude Code/Claw Code : Agent Runtime 工程样本

Claude Code / Claw Code 这类 Coding Agent，是观察现代 Agent Runtime 的一个优质入口。不再只是围绕自然语言回答问题，而是直接进入用户的开发环境：读取代码库、编辑文件、执行命令、根据运行结果继续推理。[^claw-overview]

用户可以用自然语言让它使用这些框架自带的文件读写、shell 工具，在终端中读取、编写、编辑和执行代码，完成从仓库总结到模块重构等任务。更关键的是，背后不再只是一次性模型调用，而是一条持续运行的 pipeline：[^claw-overview]
```text
用户提出任务
  -> Runtime 构造上下文与系统提示
  -> 模型判断下一步
  -> 模型发起结构化工具调用
  -> Runtime 校验权限
  -> Runtime 执行工具
  -> 工具结果返回模型上下文
  -> 模型继续判断
  -> 最终输出结果或修改项目
```
这里的一个核心是 **runtime 对工具调用的调度权**。

模型可以“请求”读文件、写文件、执行命令，但真正执行这些动作的是 **runtime**。**runtime 负责决定**：
```text
这个工具是否存在？
当前权限是否允许？
路径是否越界？
命令是否危险？
输出是否过长？
错误如何返回？
是否需要用户确认？
执行过程是否记录？
```
我们可以拆分成两层结构：
```text
模型负责提出行动意图；
Runtime 负责验证、执行、记录和反馈行动结果。
```

## 3. 工具系统： 使用 Tool Surface 作为 Agent 的能力边界

ccunpacked 对 Claude Code 的 tool surface 进行了拆解，分为几个类别：[^ccunpacked]
| 类别        | 工具示例                                                                      | 含义                             |
| --------- | ------------------------------------------------------------------------- | ------------------------------ |
| 文件操作      | `FileRead`、`FileEdit`、`FileWrite`、`Glob`、`Grep`、`NotebookEdit`            | 让 Agent 能理解和修改代码库              |
| 执行环境      | `Bash`、`PowerShell`、`REPL`                                                | 让 Agent 能运行命令、测试和脚本            |
| 搜索与获取     | `WebBrowser`、`WebFetch`、`WebSearch`、`ToolSearch`                          | 让 Agent 获取外部信息                 |
| Agent 与任务 | `Agent`、`SendMessage`、`TaskCreate`、`TaskList`、`TaskUpdate`、`TaskOutput` 等 | 让 Agent 管理子任务和多 Agent 协作       |
| 规划        | `EnterPlanMode`、`ExitPlanMode`、`VerifyPlanExecution` 等                    | 让 Agent 区分计划阶段与执行阶段            |
| MCP       | `ListMcpResources`、`ReadMcpResource`、`McpAuth` 等                          | 让 Agent 接入外部工具协议               |
| 系统工具      | `AskUserQuestion`、`TodoWrite`、`Skill`、`Config`、`CronCreate` 等             | 让 Agent 管理用户交互、任务状态、技能、配置和定时任务 |

Tool Surface 不是散落的函数列表，而是 Agent Runtime 暴露给模型的行动边界：
```text
模型本身只会生成 token；
工具系统决定这些 token 能否变成行动；
权限系统决定这些行动是否允许发生；
Runtime 决定行动如何执行、记录和反馈。
```

如果只停留在有哪些工具，其实远没有触及工具系统的核心，一个比较关键的点在于：这些工具如何注册、如何被暴露给模型、如何被权限系统约束、如何被 Runtime 调度执行，以及执行结果如何重新进入 Agent Loop。[^claw-tools]

在普通 function calling 场景中，我们可能只是手写几个函数，再把 schema 暴露给模型。但成熟 Agent Runtime 往往需要一个更系统的工具注册层。工具可能来自内置能力、运行时配置、插件，或者 MCP server。不同来源的工具会被合并到统一的 registry 中，由 registry 负责命名、描述、参数结构、权限等级和执行入口。[^claw-tools]

因此，一个工具至少包含几类信息：[^claw-tools]
```text
name：工具叫什么
description：什么时候应该使用它
input_schema：参数结构是什么
required_permission：调用它需要什么权限
executor：实际如何执行
```
这几个字段分别服务于两个对象：模型和 Runtime。模型需要通过 name、description、input_schema 理解“什么时候调用、如何调用”；Runtime 则需要通过 required_permission 和 executor 判断“能不能调用、如何执行”。

这里也能看出 Tool Surface 和 Context Engineering 的相似性：工具不是越多越好。一次性暴露过多工具，会占用上下文、增加模型选错工具的概率，也会扩大高风险能力的暴露面。因此，成熟系统通常会区分常驻工具、按需发现工具和仅 Runtime 内部可见的工具。[^langchain-agents][^claw-tools]

可以粗略分为：
| 暴露策略             | 含义                          |
| ---------------- | --------------------------- |
| Always exposed   | 每次都暴露给模型，例如基础文件读取、搜索、todo   |
| Deferred exposed | 默认不暴露，需要时通过工具搜索或 skill 机制发现 |
| Runtime-only     | 模型不可见，只由 Runtime 内部使用       |

工具调用本身也不应该直接执行，而应该经过一条统一的 dispatch pipeline：[^claw-tools]
```text
模型请求工具调用
  -> Registry 查找工具定义
  -> Runtime 解析参数
  -> Permission Enforcer 检查权限
  -> 必要时请求用户确认
  -> Tool Executor 执行工具
  -> Tool Result 返回 Agent Loop
  -> Event Log 记录调用过程
```

## 4. 权限、沙箱与 HITL
Agent 只能够输出文本时，风险主要集中在回答质量上。即使它答错了，通常也只是信息错误或表达不准确。

但当 Agent 可以读文件、写文件、运行命令、访问网络、调用外部 API 时，它就不再只是一个文本生成系统，而是开始拥有真实的行动能力。

这时，权限系统、沙箱机制和 HITL 就不应再是附加功能，而是 Agent Runtime 的基础设施。

### 4.1 权限：行动能力不能默认开放

在工具系统中，每一个工具都代表一种行动能力。

例如：
```text
FileRead    -> 读取文件
FileWrite   -> 写入文件
FileEdit    -> 修改文件
Bash        -> 执行命令
WebFetch    -> 访问外部网络
MCP Tool    -> 调用外部服务
Agent       -> 创建子任务或子 Agent
CronCreate  -> 创建定时任务
```
这些工具的风险级别并不一样。读取 README 和执行 rm -rf 显然不是同一种操作；搜索代码和修改生产配置文件，也不是同一种风险。

成熟 Agent Runtime 架构有必要在工具调用前加入权限检查。[^claw-permission]

### 4.2 沙箱：限制 Agent 的行动范围

权限系统回答的是：
```text
这个操作是否允许？
```
沙箱回答的是：
```text
即使允许，它最多能影响哪里？
```
这两个问题并不相同。

比如允许 Agent 执行测试命令，并不意味着它应该能访问用户的整个 home 目录；允许它写文件，也不意味着它可以修改 workspace 之外的任意路径。

因此，Coding Agent 通常需要一个隔离环境。这个隔离环境可以是：[^deepagents-sandbox]
| 类型                   | 作用                  |
| -------------------- | ------------------- |
| Workspace boundary   | 只能访问当前项目目录          |
| Temporary sandbox    | 在临时目录中运行代码          |
| Container sandbox    | 通过容器隔离文件系统和进程       |
| Network policy       | 限制是否允许访问外部网络        |
| Credential isolation | 防止 Agent 接触宿主机密钥和凭据 |

沙箱的价值是让 Agent 的错误被限制在可控范围内。

### 4.3 HITL：在不确定和高风险处交还控制权

Human-in-the-loop，指的是在人机协作系统中，让人类在关键节点参与决策、确认、补充信息或接管流程。[^deepagents-hitl]

在人类日常使用 Agent 的场景中，最常见的 HITL 形式就是人类确认。

例如：
```text
Agent 准备修改多个文件
Agent 准备删除文件
Agent 准备安装依赖
Agent 准备执行 shell 命令
Agent 准备访问外部网络
Agent 准备提交 git commit
Agent 准备调用付费 API
Agent 准备创建定时任务
```
这些操作不一定必须禁止，但也不应该默认自动执行。更合理的方式是：Runtime 暂停执行，把操作意图、影响范围和风险展示给用户，由用户决定是否继续。

### 4.4 权限、沙箱与 HITL 的关系
可以这样区分：
| 机制   | 核心问题            |
| ---- | --------------- |
| 权限   | 这个操作是否允许？       |
| 沙箱   | 这个操作最多能影响哪里？    |
| HITL | 这个操作是否需要人类参与决策？ |

## 5. Memory 与 Context
在 Agent Runtime 中，Memory 和 Context 很容易被混在一起讨论，但二者并不是同一个概念。

这里做一个分类：
```text
Memory 是外部保存的信息库存；
Context 是每次模型调用时，被 Runtime 装配进上下文窗口的工作集。
```

ChatGPT Memory 的逆向分析提供了一个很好的观察角度。llmrefs 的文章认为，ChatGPT 的记忆效果并不一定来自复杂的全量历史 RAG，而更像是由几层结构化上下文共同构成：Session Metadata、User Memory、Recent Conversations Summary、Current Session Messages。文章也强调，每次用户发送消息时，模型实际接收到的是一个经过组织的 context window。[^llmrefs-memory]

可以简单概括为：
| 层级                           | 含义                 | 作用        |
| ---------------------------- | ------------------ | --------- |
| Session Metadata             | 当前会话、设备、位置、时间等临时信息 | 让模型理解当前环境 |
| User Memory                  | 用户长期事实、偏好、项目等      | 提供跨会话个性化  |
| Recent Conversations Summary | 近期对话摘要             | 提供轻量连续性   |
| Current Session Messages     | 当前会话消息             | 保持本轮对话连贯  |

这个结构带来的启发是：连续性不一定依赖“保存并检索所有历史”。很多时候，轻量摘要、稳定用户事实、当前会话窗口的组合，就足以制造一种“系统理解上下文”的体验。

对于 Agent Runtime 来说，这一点更重要。因为 Agent 面对的不只是聊天连续性，还包括任务连续性。

一个长程 Agent 需要记住的内容可能包括：
```text
用户当前目标是什么
已经完成了哪些步骤
读过哪些文件
调用过哪些工具
哪些假设被验证或推翻
哪些文件被修改
下一步计划是什么
哪些操作需要用户确认
```
比起用户记忆、对话历史，这几个更像是 Task State，所以，在 Agent 系统里，Memory 至少可以拆成几类：
| 类型                   | 保存内容                | 主要用途      |
| -------------------- | ------------------- | --------- |
| User Memory          | 用户偏好、长期事实、稳定项目背景    | 个性化和跨会话理解 |
| Conversation Summary | 当前或近期对话摘要           | 压缩上下文     |
| Task State           | 当前任务计划、步骤状态、中间发现    | 支撑长程执行    |
| Project Memory       | 项目结构、技术决策、历史约定      | 支撑项目级连续性  |
| Event Log            | 工具调用、权限确认、文件变更、错误信息 | 恢复、审计、调试  |
| Artifact Store       | 草稿、报告、代码 patch、中间产物 | 保存工作成果    |

这部分最关键的是区分 **Memory** 和 **Context Assembly**。

Memory 可以保存很多东西，但每次模型调用时，不应该把所有 memory 都塞进上下文。Runtime 需要判断：当前任务真正需要哪些信息？哪些应该完整注入？哪些只需要摘要？哪些只需要保留一个文件路径或引用？哪些应该通过工具按需读取？

LangChain DeepAgents 的 context engineering 文档也强调，复杂 Agent 需要处理 input context、runtime context、context compression、context isolation 和 long-term memory 等问题；这说明上下文管理本身已经成为 Agent 系统的一部分，而不是简单的 prompt 拼接。[^deepagents-context]

如果没有这层上下文管理机制，就会出现两个极端：
```text
什么都塞进 prompt：
  -> token 成本高
  -> 上下文噪音大
  -> 模型难以抓住重点

什么都不保存：
  -> Agent 无法跨步骤执行
  -> 用户需要反复解释背景
  -> 长任务中断后难以恢复
```

对于现代 Agent 来说，连续性来自三层能力：
```text
保存：把重要信息放到合适的 memory 层
压缩：把长历史转化为可用摘要
调度：在每次模型调用前选择合适内容进入 context
```
这也解释了为什么 Memory 会和前面的工具系统、权限系统产生关系。工具调用结果需要进入 Event Log；文件变更需要进入 Project Memory；用户确认需要进入任务轨迹；阶段性结论需要进入 Summary；而真正进入模型上下文的，只是 Runtime 在当前步骤中筛选后的部分。

## 6. 任务状态与事件日志：Agent 的运行轨迹
如果说 Memory 解决的是“Agent 如何获得连续性”，那么任务状态和事件日志解决的就是另一个问题：
```text
Agent 执行过程中，到底发生了什么？
```
在普通 Chatbot 中，我们通常只关心最终回答。用户发出问题，模型返回文本，对话记录本身就是主要状态。

但在 Agent Runtime 中，执行过程会复杂得多。一次任务可能包括多轮模型调用、多次工具调用、权限判断、用户确认、文件读写、命令执行、错误重试和阶段性总结。此时，仅仅保存最终回答已经不够了。

一个 Agent 运行过程可能包含：
```text
收到用户任务
构造上下文
模型开始推理
模型请求调用工具
Runtime 检查权限
用户批准或拒绝
工具开始执行
工具返回结果
Agent 更新 todo
Agent 修改文件
Agent 执行测试
测试失败
Agent 读取错误日志
Agent 再次修改
最终完成任务
```
这些中间步骤共同构成了 Agent 的运行轨迹。

如果没有运行轨迹，Agent 就会变成一个黑盒：用户只看到它最后说了什么，却不知道它读了什么、改了什么、执行了什么、失败在哪里、为什么继续下一步。

### 6.1 Task State：记录任务进度
Task State 可以理解为 Agent 当前任务的结构化状态。

它和 Memory 有关系，但不完全相同。Memory 更偏向长期保存的信息，Task State 更偏向当前任务执行中的动态状态，即 Plan / TODO List。

例如，一个 Coding Agent 在修复 bug 时，Task State 可能包括：
```text
任务目标：修复登录接口偶发 500 的问题
当前阶段：定位错误来源
已完成步骤：
  - 阅读 auth controller
  - 阅读 user service
  - 运行相关测试
当前发现：
  - 失败发生在 token refresh 分支
  - 数据库查询返回空值时没有处理
下一步计划：
  - 修改空值判断
  - 补充测试
  - 重新运行测试
阻塞点：
  - 需要确认是否兼容旧 token 逻辑
```
这类状态不应该完全依赖模型上下文临时记忆。因为一旦上下文被压缩、连接中断、页面刷新，任务进度就可能丢失。

因此，成熟 Agent Runtime 往往需要把任务状态外部化，例如：
```
todo list
plan
current step
intermediate findings
open questions
modified files
pending approvals
```
这也是为什么很多 Agent 系统会有 `TodoWrite`、`TaskCreate`、`TaskUpdate`、`PlanMode` 这类工具。它们表面上是在展示进度，实际上是在把模型的隐式思考过程转化为可保存、可恢复、可展示的外部状态。[^ccunpacked]

### 6.2 Event Log：记录执行过程日志
例如：
```text
Task State:
  当前步骤 = 运行测试
  状态 = failed

Event Log:
  10:01 Agent 请求执行 npm test
  10:01 Runtime 检查权限，通过
  10:02 Tool Executor 开始执行
  10:03 测试失败
  10:03 stderr 返回 TypeError
  10:04 Agent 读取相关文件
```

### 6.3 Event Log 的作用
Event Log 的作用可以简单概括为四点。

第一是**调试**。当 Agent 做错事时，开发者需要知道问题出在哪里：是上下文构造错了，还是模型选错工具；是工具参数错误，还是工具结果被截断；是权限拒绝后没有正确处理，还是测试失败后模型误判为成功。没有运行轨迹，就只能看最终回答，很难定位问题。

第二是**恢复**。长任务很容易因为浏览器刷新、SSE 断开、模型超时、工具失败、后端重启而中断。如果关键事件已经持久化，Runtime 就有机会恢复到某个稳定边界，例如最后一次完成的模型调用、最后一次完成的工具调用、最后一次更新的 task state，或者最后一次用户确认点。

第三是**审计**。当 Agent 具备真实行动能力后，用户需要知道它读了哪些文件、改了哪些文件、执行了哪些命令、是否访问了外部网络、是否在用户批准前执行了高风险操作。权限系统负责判断能不能做，HITL 负责让用户参与关键决策，Event Log 则负责记录这些判断和决策是否真的发生过。

第四是**前端展示**。Agent 的执行轨迹可以成为 UI 的数据源。前端可以展示当前计划、正在执行的步骤、工具调用、命令输出、文件变更、测试结果、用户确认点、错误和重试。这样用户看到的不只是最终答案，而是一个可理解、可追踪的执行过程。

所以，Event Log 不只是后端日志，也可以成为 Agent Runtime 的观测层、恢复层和交互层。[^langchain-agents][^claw-overview]

### 6.4 Checkpoint：保存可恢复的运行时状态

Event Log 记录“发生过什么”，Task State 记录“任务当前进展”，Checkpoint 则记录“Runtime 如何从某个稳定边界继续执行”。

在 LangGraph 这类系统中，checkpoint 通常和 thread_id 绑定。thread_id 可以理解为运行时线程或执行上下文的命名空间，checkpointer 会基于它保存和恢复图状态。这样 Agent 在遇到中断、HITL 审批、长任务暂停、后端重启时，才有机会从稳定状态继续运行。

[^langchain-agents]: [LangChain Docs, “Agents”](https://docs.langchain.com/oss/javascript/langchain/agents)。该文档说明 Agent 将语言模型和工具组合起来，使系统能够围绕任务进行推理、选择工具并迭代完成目标；`createAgent()` 构建在 LangGraph 的 graph-based runtime 之上；文档也讨论了 tools、dynamic tools、tool error handling、ReAct loop、memory、streaming、middleware 和 LangSmith tracing。

[^claw-overview]: [zread, “Claw Code Overview”](https://zread.ai/instructkr/claude-code/1-overview)。该文将 Claw Code 描述为一个 Rust 编写的 autonomous CLI coding agent，并拆解了其 9-crate workspace，包括 CLI、runtime、tools、api、commands、plugins、telemetry、mock service 和 compat harness；其中 runtime 涉及会话运行时、配置、session 持久化、权限策略、MCP 生命周期和系统提示组装。

[^ccunpacked]: [ccunpacked, “Claude Code Unpacked”](https://ccunpacked.dev/)。该站点对 Claude Code 的 agent loop、tool surface、规划工具、任务工具、MCP 工具、系统工具和部分实验性能力做了结构化拆解。

[^claw-tools]: [zread, “Tool System and Registry”](https://zread.ai/instructkr/claude-code/11-tool-system-and-registry)。该文拆解了 Claw Code 的工具系统，包括 `GlobalToolRegistry`、builtin/runtime/plugin tools、`ToolSpec`、`required_permission`、deferred tools、MCP bridge、permission enforcer，以及 ConversationRuntime 通过 `ToolExecutor` 与工具系统解耦的 dispatch pipeline。

[^claw-permission]: [zread, “Claw Code Overview”](https://zread.ai/instructkr/claude-code/1-overview) 与 [zread, “Tool System and Registry”](https://zread.ai/instructkr/claude-code/11-tool-system-and-registry)。相关章节提到 Claw Code 的权限模式包括 `read-only`、`workspace-write`、`danger-full-access`，并且工具会携带权限等级，执行前经过 permission enforcer 检查。

[^deepagents-sandbox]: [LangChain Docs, “DeepAgents: Sandboxes”](https://docs.langchain.com/oss/python/deepagents/sandboxes)。文档说明 deep agent 会生成代码、与文件系统交互并运行 shell 命令，因此需要隔离环境，避免访问宿主机凭据、文件或网络资源；sandbox backend 可以提供文件系统工具、`execute` 工具和安全边界。

[^deepagents-hitl]: [LangChain Docs, “DeepAgents: Human-in-the-loop”](https://docs.langchain.com/oss/python/deepagents/human-in-the-loop)。文档将 HITL 落到敏感工具操作的人类审批上，可通过 `interrupt_on` 配置需要审批的工具，并支持 approve、edit、reject、respond 等决策。

[^llmrefs-memory]: [LLMrefs, “How ChatGPT Memory Works, Reverse Engineered”](https://llmrefs.com/blog/reverse-engineering-chatgpt-memory)。该文基于逆向观察，将 ChatGPT 的上下文结构拆解为 Session Metadata、User Memory、Recent Conversations Summary、Current Session Messages 等层级，并强调模型每轮实际接收的是经过组织的 context window。

[^deepagents-context]: [LangChain Docs, “DeepAgents: Context engineering”](https://docs.langchain.com/oss/python/deepagents/context-engineering)。文档将 context engineering 定义为以正确格式提供正确的信息和工具，使 deep agent 能可靠完成任务，并讨论 input context、runtime context、context compression、context isolation 和 long-term memory 等类型。
