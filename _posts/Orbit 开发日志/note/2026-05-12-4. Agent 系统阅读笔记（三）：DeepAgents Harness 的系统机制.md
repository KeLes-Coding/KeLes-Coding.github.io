# 4. Agent 系统阅读笔记（三）：DeepAgents Harness 的系统机制

上一篇主要从 LangChain Agent、LangGraph Runtime 和 DeepAgents 的关系出发，说明一个基础 Agent Loop 如何逐步长成复杂任务型 Agent 的运行结构。

这一篇继续拆 DeepAgents Harness 的具体机制。DeepAgents 并没有推翻基础 tool calling loop，而是在这条循环之上加入一组面向复杂任务的默认能力：Planning、Filesystem、Context Engineering、Subagents、Backends、Sandbox、Permissions、Memory、Skills 和 HITL。

可以先把它们放在一张图里理解：
```text
DeepAgents Harness
  -> Planning: 任务计划外部化
  -> Filesystem: 中间产物外部化
  -> Context Engineering: 上下文选择、压缩和隔离
  -> Subagents: 子任务隔离执行
  -> Backends / Sandbox / Permissions: 状态、文件和命令执行边界
  -> Memory / Skills / HITL: 长期连续性、能力加载和人工控制
```

## 1. Planning：任务计划外部化
复杂任务首先遇到的问题是：Agent 如何知道自己正在做什么、已经完成什么、下一步应该做什么。

### 1.1 计划外部化原因
在基础 Agent Loop 中，模型可以根据 messages 和 tool results 临时判断下一步。对于短任务，这通常已经足够。比如用户问一个问题，Agent 调用一次搜索工具、读取一段资料，然后生成回答。整个过程很短，任务状态可以隐含在当前上下文里。

但复杂任务不一样。

复杂任务往往会持续更久，跨越更多步骤，并且在执行过程中不断产生新的信息。例如调研一个技术主题、阅读一个代码库、修复一个 bug、生成一份长报告，或者把一个大任务拆成多个子任务执行。此时，Agent 需要持续维护一组动态信息：
```text
当前目标是什么？
任务被拆成了哪些步骤？
哪些步骤已经完成？
当前正在执行哪一步？
哪些发现改变了原计划？
哪些问题仍然阻塞？
下一步应该继续做什么？
```

如果这些信息只存在于模型上下文中，就会非常脆弱。

一方面，模型上下文会不断变化。随着工具结果、文件内容、用户反馈、阶段性摘要不断进入 messages，早期计划可能被挤出上下文，也可能在压缩过程中丢失细节。另一方面，长任务经常会遇到中断：前端连接断开、工具调用失败、模型请求超时、用户暂停审批、Runtime 恢复执行。只依赖模型临时记忆，很难保证任务恢复后还能准确知道“做到哪里”。

因此，复杂 Agent 需要把计划从模型内部移到 Runtime 外部。

### 1.2 write_todos 与 Task State
DeepAgents 的 planning 能力正是围绕这一点设计的。官方 overview 中提到，Deep Agents 内置 `write_todos` 工具，使 Agent 能把复杂任务拆成离散步骤、追踪进度，并随着新信息出现调整计划。[^deepagents-overview] Harness 文档进一步说明，`write_todos` 用于维护结构化任务列表，可以追踪多个任务及其状态，例如 `pending、in_progress、completed`，并且这些 todos 会持久化在 agent state 中。[^deepagents-harness]

简单总结：
```text
模型内部计划:
  临时、隐式、依赖上下文窗口

write_todos:
  外部、显式、结构化、可持久化

Task State:
  Runtime 可以保存、展示、恢复和审计的任务进度
```

### 1.3 结构化任务状态作用
更重要的是，这个任务列表不只是给模型自己看的。它可以同时服务于四个对象。

- 第一，服务于模型自身。
模型在后续调用中可以重新看到当前任务计划，减少目标漂移。长任务中很常见的问题是，模型在多轮工具调用后逐渐偏离最初目标，或者反复做已经完成的步骤。外部化的 Task State 可以让模型每次重新进入执行循环时，都有一个明确的进度锚点。

- 第二，服务于 Runtime。
Runtime 可以把 todos 作为 agent state 的一部分保存下来。这样，即使任务中断，系统也可以在恢复后知道当前任务处于哪个阶段，而不是只依赖 messages 中的自然语言记录。

- 第三，服务于前端展示。
如果任务状态是结构化的，前端就可以展示当前计划、正在执行的步骤和已完成步骤。DeepAgents 的 todo list 前端文档也描述了一个典型流程：用户提交请求后，Agent 在 state 中生成 todos；执行过程中 todo 状态从 pending 变为 `in_progress` 再变为 `completed`；`stream.values.todos` 实时更新，UI 可以据此重新渲染任务列表。[^deepagents-todo-ui]

- 第四，服务于审计和调试。
当 Agent 最终结果不符合预期时，开发者可以回看任务计划：Agent 是否拆错了步骤？是否跳过了关键任务？是否把某个未完成步骤标记为完成？是否在新信息出现后没有更新计划？相比只看最终回答，结构化 Task State 更容易定位问题。

Planning 在 DeepAgents 中承担的是一个连接点/中枢：
```text
它连接模型推理与 Runtime 状态；
连接任务执行与前端展示；
连接长任务恢复与执行审计。
```

这也能解释为什么 `write_todos` 这样一个看起来很轻量的工具，会被放在 DeepAgents Harness 的核心能力中。复杂任务的困难往往不在于“能不能想到一个计划”，而在于这个计划能不能在长时间执行过程中持续存在、持续更新，并被 Runtime 和用户共同观察。

### 1.4 Plan、Todo、Task State 与 Event Log
在普通 Agent 中，计划常常是一次性生成的文本：
```text
我将分三步完成：
1. 先搜索资料
2. 再整理要点
3. 最后生成总结
```
这种计划对用户来说有解释作用，但它很难直接成为系统状态。Runtime 不知道哪一步正在执行，也不知道模型后来是否真的完成了这些步骤。

DeepAgents 的 `write_todos` 则把计划变成结构化对象：
```text
[
  { "content": "搜索资料", "status": "completed" },
  { "content": "整理要点", "status": "in_progress" },
  { "content": "生成总结", "status": "pending" }
]
```
此时，计划就不再只是自然语言说明，而是 **Agent Runtime 可以管理的状态**。

这种设计与上一篇提到的 Task State 是连贯的。Task State 记录的是 Agent 当前任务的结构化进度，例如当前阶段、已完成步骤、下一步计划、阻塞点和待确认事项。Planning 工具的作用，就是让模型把这些信息写入外部状态，而不是把它们留在一段随时可能被压缩掉的上下文里。[^Agent-Runtime]

可以把它概括为：
```text
Plan:
  模型对任务的拆解

Todo:
  Plan 的结构化表现

Task State:
  Runtime 保存和管理的任务进度

Event Log:
  Runtime 记录任务执行中发生了什么
```
这几个概念需要区分。Plan 更偏向模型的任务理解；Todo 是可以被更新的步骤列表；Task State 是 Runtime 中的当前进度；Event Log 则记录每一步具体发生了什么。比如，todo 中可能只写“运行测试”，而 Event Log 会记录具体运行了什么命令、返回了什么错误、是否重试、用户是否批准。

### 1.5 Planning 是持续状态机制
所以，Planning 不应该被理解为“模型先想好再执行”的静态过程。更准确地说，它是一个持续更新的外部状态机制：
```text
任务开始时：
  Agent 创建初始计划

执行过程中：
  Agent 根据工具结果更新计划

遇到阻塞时：
  Agent 标记当前问题或等待用户输入

任务恢复时：
  Runtime 重新提供当前计划

任务完成时：
  todos 记录最终完成状态
```
这也是 DeepAgents 相比基础 Agent Loop 的一个关键增强。基础 Loop 让模型能够持续行动；Planning 让这些行动围绕一个外部可见的任务状态展开。

从下一节开始，就要看另一个问题：Agent 在执行计划时会产生大量中间产物，这些内容不适合全部塞进 messages。那么，复杂任务的草稿、笔记、代码、摘要和阶段性结果应该放在哪里？

这就进入 Filesystem 与 Context Engineering。

## 2. Filesystem 与 Context Engineering：上下文窗口管理
Planning 解决的是“Agent 如何知道自己做到哪里”。但一旦 Agent 开始执行计划，马上会遇到另一个问题：执行过程中产生的大量中间产物应该放在哪里。

这一章的核心可以先概括为一句话：filesystem 负责把信息保存到上下文窗口之外；context engineering 负责决定何时把哪些信息重新装配进上下文。

### 2.1 Messages 不适合承载所有产物
在基础 Agent Loop 中，messages 是最主要的状态容器。用户输入、模型输出、tool call、tool result 都会进入消息流。对于短任务来说，这种方式足够直接：工具返回一段结果，模型读取这段结果，然后继续回答。

但复杂任务中的中间产物往往远远超过 messages 适合承载的范围。

如果这些内容全部进入 messages，Agent 的上下文会迅速膨胀。模型每一轮调用都会被迫携带越来越多历史信息，token 成本上升，噪音增加，关键目标也更容易被淹没。更严重的是，很多中间产物并不需要在每一轮都出现。它们只需要被保存，并在相关步骤中按需读取。

这就是 DeepAgents 引入 filesystem 的原因。

DeepAgents overview 中提到，文件系统工具 `ls`、`read_file`、`write_file`、`edit_file` 可以让 Agent 将大量上下文卸载到内存或文件系统存储中，避免 context window overflow，并处理长度不固定的工具结果。[^deepagents-overview] Harness 文档也说明，virtual filesystem 会被 skills、memory、code execution、context management 等多种 harness capability 使用。[^deepagents-harness]

```text
Messages:
  记录对话过程、模型输出、工具调用和工具结果

Filesystem:
  保存草稿、笔记、报告、代码、日志和阶段性产物

Context Assembly:
  决定哪些文件或片段在当前步骤重新进入模型上下文
```

Messages 更适合记录“发生了什么”。Filesystem 更适合保存“产出了什么”。例如，Agent 调用搜索工具这件事可以记录在消息流或事件日志中；但搜索结果整理后的长笔记，更适合写入文件。Agent 执行测试这件事可以进入执行轨迹；但完整测试日志、失败堆栈、修复草稿，更适合保存到工作区。

这样做的好处是，模型不需要每一轮都重新读取全部内容。它可以通过文件路径、摘要、目录结构来知道有哪些产物存在，在需要时再读取相关片段。

### 2.2 保存、引用、读取、装配
可以简单理解为：
```text
保存：
  将长内容、中间产物、工具输出写入 filesystem

引用：
  在 messages 或 task state 中保留文件路径、摘要和当前状态

读取：
  在当前步骤需要时，通过 read_file / grep / glob 等工具按需取回

装配：
  Runtime 将相关 messages、summary、file snippets、tools 和 memory 组织成当前模型调用的 context
```
这样，Agent 的工作方式就从“记住所有内容”转向“知道信息在哪里，并在需要时读取”。

### 2.3 Filesystem 作为 Artifact Store
DeepAgents 的文件系统工具也服务于这一点。`read_file` 可以读取文件内容；`write_file` 可以写入草稿或结果；`edit_file` 可以修改已有文件；`ls、glob、grep` 可以帮助 Agent 发现和定位文件。Backends 文档进一步说明，Deep Agents 暴露给 Agent 的 filesystem surface 包括 `ls、read_file、write_file、edit_file、glob、grep` 等工具，这些工具会通过 pluggable backend 执行。[^deepagents-backends]

从系统角度看，这些工具共同组成了一个 Artifact Store 的操作闭环：
```text
write_file:
  保存中间产物

read_file:
  按需读取上下文

edit_file:
  增量修改已有产物

ls / glob:
  发现工作区结构

grep:
  在工作区中定位相关信息
```
对复杂 Agent 来说，外部工作区至少有四个作用。

- 第一，降低上下文压力。
长文档、工具输出、日志和草稿不需要每轮进入模型上下文，只需要在需要时读取。

- 第二，保存中间产物。
复杂任务往往不是一步到位完成，而是不断生成草稿、摘要、结论和修改版本。Filesystem 可以保存这些阶段性结果。

- 第三，支持任务恢复。
如果任务中断，只要工作区内容被保存，Agent 恢复后可以重新读取任务计划、草稿、日志和结果，而不是完全依赖旧 messages。

- 第四，支持子任务协作。
后面会讲到 subagents。主 Agent 可以让子 Agent 把结果写入文件，自己只读取最终摘要或关键片段。这样可以减少上下文污染，也方便多个任务之间共享产物。

Filesystem 提供外部化存储，Context Engineering 决定如何使用这些外部化内容。

### 2.4 Context Assembly 的取舍
这与第一篇中对 Memory 和 Context 的区分是连贯的：Memory 是外部保存的信息库存，Context 是每次模型调用时被 Runtime 装配进上下文窗口的工作集。[^Agent-Runtime] 在 DeepAgents 中，filesystem 可以被看作一种更面向任务产物的外部库存；它保存的是当前任务产生的草稿、笔记、日志、代码和报告。真正进入模型上下文的，只是当前步骤需要的一部分。

这也能解释为什么 Context Engineering 会成为 DeepAgents 的核心能力。因为复杂任务的执行质量，很大程度上取决于每一轮模型调用前的上下文装配质量。

如果装配太少，模型缺少必要信息，会重复提问、误判状态或走错方向。

如果装配太多，模型会被噪音干扰，token 成本上升，也更容易忽略当前步骤的重点。

一个合理的 Context Assembly 应该大致回答：
```text
当前目标是什么？
当前 todo 状态是什么？
需要保留哪些最近 messages？
哪些历史可以摘要？
哪些文件需要读取？
哪些文件只需要提供路径？
哪些工具应该暴露？
哪些 memory 与当前任务相关？
哪些 Runtime Context 不应该暴露给模型？
```

DeepAgents 的 context engineering 文档还提到，DeepAgents 会通过 middleware 将 harness capabilities 相关说明加入 system prompt，包括 planning prompt、filesystem prompt、subagent prompt、HITL prompt 和 local context prompt 等。[^deepagents-context]

## 3. Subagents：用上下文隔离处理复杂子任务

前面两节分别讨论了 Planning 和 Filesystem / Context Engineering。

Planning 让任务计划外部化，使 Agent 能够持续知道自己做到哪里；Filesystem 与 Context Engineering 则把中间产物移出上下文窗口，让模型在当前步骤只读取必要信息。

但这仍然没有解决复杂任务中的另一个问题：主 Agent 是否应该亲自处理所有细节。

### 3.1 主上下文为什么会被污染
在短任务中，主 Agent 自己完成所有步骤通常没有问题。比如调用一次搜索工具、读取一个文件、生成一段总结，这类任务的上下文规模有限，模型可以直接处理。

复杂任务则不同。一个长程 Agent 经常会遇到大量局部任务：
```text
搜索某个技术概念的背景资料
阅读某个目录下的所有代码文件
比较多个实现方案
整理某组文档的共同观点
检查某个测试失败的原因
为某一节内容生成初稿
验证某个命令输出是否符合预期
```
这些任务本身可能都不算大，但它们会产生大量过程性信息：
```text
搜索结果
网页片段
文件内容
命令输出
失败尝试
临时假设
局部推理
中间结论
```
如果全部由主 Agent 亲自处理，这些信息就会不断进入主上下文。时间一长，主 Agent 的上下文会被大量局部细节占满。它原本应该负责总体目标、任务计划和最终整合，但却被迫携带每一个子任务中的搜索过程、工具输出和失败分支。

这就是 subagents 的价值。

### 3.2 Subagent 的核心是上下文隔离
DeepAgents overview 中提到，内置 task tool 可以让 Agent 生成 specialized subagents，用于 context isolation，使 main agent 的 context 保持干净，同时仍然能够深入处理具体子任务。[^deepagents-overview]

也就是说，subagent 的意义不只体现在“把任务分给另一个 Agent”。更关键的是，它提供了一种上下文隔离机制。

它把局部任务的细节隔离在子上下文中，只把经过压缩和整理的结果返回主 Agent。

这种机制对复杂任务很重要。

### 3.3 什么时候适合委派
- 第一，它能降低主 Agent 的上下文负担。
主 Agent 不需要亲自携带每一个局部任务的全过程，只需要维护总体计划和关键结果。

- 第二，它能让不同子任务使用不同指令。
一个负责资料检索的 subagent，可以强调来源筛选和引用；一个负责代码阅读的 subagent，可以强调文件路径、函数调用关系和潜在风险；一个负责写作的 subagent，可以强调结构、风格和段落衔接。DeepAgents 的 subagents 文档也说明，自定义 subagent 可以定义自己的 system prompt、tools、model、middleware、interrupt_on 和 skills 等配置。[^deepagents-subagents]

- 第三，它能减少任务之间的相互干扰。
如果所有子任务都在主上下文中展开，模型很容易把某个局部任务中的临时结论误用到另一个任务中。Subagent 隔离后，局部探索可以在自己的上下文中完成，主 Agent 只接收最终可用结果。

- 第四，它能让主 Agent 更像 coordinator。
主 Agent 负责判断任务拆分、调用哪个 subagent、如何合并结果、是否需要继续追问或补充执行。Subagent 负责具体执行。这个结构更接近复杂任务中的调度关系。

需要注意的是，subagents 并不适合所有任务。

如果任务很简单，只需要一步工具调用或一小段推理，直接由主 Agent 执行更高效。因为创建 subagent 本身也有成本，包括额外模型调用、额外上下文装配、额外工具调用和结果整合。DeepAgents subagents 文档也明确提到，subagents 适合会污染主上下文的多步骤任务、需要专门指令或工具的领域任务、需要不同模型能力的任务，以及希望主 Agent 专注高层协调的场景；简单单步任务、需要保留完整中间上下文的任务，则不适合拆给 subagent。[^deepagents-subagents]

总结：
```text
适合：
  多步骤局部任务
  工具输出很长的任务
  需要专门指令的任务
  需要专门工具集的任务
  会污染主上下文的探索任务

不适合：
  简单单步任务
  必须完整保留中间过程的任务
  拆分成本高于收益的任务
```

从 Agent Runtime 的角度看，subagents 扩展了任务执行结构。

### 3.4 嵌套的 Agent Loop
基础 Agent Loop 只有一个模型循环：
```text
Main Agent Loop:
  model -> tools -> model -> final answer
```
加入 subagents 后，呈现嵌套结构：
```text
Main Agent Loop:
  model -> task tool -> Subagent Loop -> result -> model
```
Subagent 内部仍然是类似的工具调用循环，但它拥有自己的上下文、提示词、工具集和执行过程。主 Agent 看到的是 subagent 返回的结果，而不是子循环中的全部细节。

这也说明，subagents 并没有改变 Agent Loop 的基本形态。它改变的是复杂任务的组织方式：把某些局部循环从主上下文中移出去，让它们在隔离环境中完成。

这一层之后，DeepAgents 已经具备了复杂任务的基本工作方式：有计划、有工作区、有上下文装配，也能委派子任务。

但接下来还要回答两个更工程化的问题：
```text
这些文件和状态最终保存在哪里？
Shell 命令和代码执行发生在哪里？
Agent 的读写权限和执行边界如何约束？
```
这就进入 Backends、Sandbox 与 Permissions。

## 4. Backends、Sandbox 与 Permissions：持久化边界和安全边界
到目前为止，DeepAgents 已经具备了复杂任务执行的几层基础能力。

Planning 让任务进度外部化，Filesystem 与 Context Engineering 让中间产物可以移出上下文窗口，Subagents 则让局部任务在隔离上下文中执行。接下来还要回答一个更工程化的问题：
```text
这些文件、状态和命令到底运行在哪里？
```
在一个简单 Agent 中，这个问题并不突出。工具调用可能只是访问一个 API，或者读取少量外部信息。但在 DeepAgents 这类复杂任务型 Agent 中，Agent 会读写文件、保存草稿、修改代码、运行测试、执行 Shell 命令，甚至访问不同类型的持久化存储。此时，Runtime 必须明确几个边界：
```text
文件保存在哪里？
状态是否跨 thread 持久化？
哪些路径可以读？
哪些路径可以写？
Shell 命令在哪里执行？
执行环境是否隔离？
Agent 是否能访问宿主机文件、凭据和网络？
```

DeepAgents 用 Backends、Sandbox 和 Permissions 来组织这些问题。

一句话概括：
```text
Backend 决定文件和状态落在哪里；
Sandbox 决定命令和代码在哪里执行；
Permissions 决定 Agent 能够读写哪些路径和资源。
```

### 4.1 Backends：文件和状态落在哪里
DeepAgents 的 Backends 文档说明，Agent 看到的是一组 filesystem surface，例如 `ls、read_file、write_file、edit_file、glob、grep`，但这些工具背后的执行方式由 pluggable backend 决定。也就是说，模型看到的是统一的文件系统工具，Runtime 背后可以把这些操作路由到不同存储实现。[^deepagents-backends]

这就是 Backend 的核心作用：它定义 Agent 工作区的持久化边界。

比如，同样是 write_file("/draft/report.md")，背后可以有不同落点：
```text
State Backend:
  写入当前 thread 的临时状态

Filesystem Backend:
  写入本地磁盘或项目目录

Store Backend:
  写入跨 thread / 跨 session 的长期存储

Composite Backend:
  根据路径前缀路由到不同 backend

Sandbox Backend:
  写入隔离执行环境中的文件系统
```
从 Agent 的角度看，它只是在读写路径；从 Runtime 的角度看，每个路径都可能对应不同的生命周期和风险级别。

例如：
```text
/tmp/notes.md
  当前任务临时笔记，任务结束后可以丢弃

/project/src/app.py
  项目真实文件，需要权限控制和变更审计

/memory/user-preferences.md
  跨任务长期记忆，需要稳定持久化

/sandbox/output.log
  隔离环境中的命令输出，不应污染宿主机
```
由此，Backend 决定的是 Agent 的工作产物如何生存：
```
短期产物：
  保存到当前 thread state

项目产物：
  保存到 workspace 或 filesystem

长期信息：
  保存到 store

危险执行产物：
  保存到 sandbox

混合场景：
  通过 composite backend 按路径路由
```
这也解释了为什么 DeepAgents 要提供 pluggable filesystem backends。复杂任务的文件并不都属于同一种状态：有些只是 scratch notes，有些是最终交付物，有些是项目源码，有些是长期记忆，有些是命令执行结果。把它们全部放进同一个目录或同一个 state，边界会很模糊。

Backends 的价值就在于把这些边界显式化。

这也和上一节的 Subagents 产生关系。Subagent 可以在自己的上下文中执行局部任务，但它仍然需要一个地方保存结果。这个结果可以返回给主 Agent，也可以写入 filesystem。Backend 决定这个文件是否只对当前任务可见，是否能被主 Agent 读取，是否会保留到长期存储中。

不过，Backend 主要回答的是“保存在哪里”。当 Agent 开始执行代码或 Shell 命令时，还需要另一个边界：Sandbox。

### 4.2 Sandbox：命令和代码在哪里执行
DeepAgents 的 Sandboxes 文档说明，Agent 会生成代码、与文件系统交互并运行 shell 命令，因此需要隔离环境，避免访问宿主机凭据、文件或网络资源；sandbox backend 会提供标准 filesystem tools，并额外提供 `execute` 工具，让 Agent 在 sandbox 中运行 shell 命令。[^deepagents-sandboxes]

这意味着 Sandbox 和普通 Backend 的区别在于：普通 Backend 只提供文件操作，Sandbox Backend 还提供代码执行环境。

可以这样区分：

| 机制                 | 主要能力        | 核心问题         |
| ------------------ | ----------- | ------------ |
| Filesystem Backend | 读写文件        | 文件和状态落在哪里    |
| Sandbox Backend    | 读写文件 + 执行命令 | 命令和代码在哪里安全运行 |
| Permissions        | 路径和操作约束     | 哪些读写动作被允许    |

如果 Agent 只需要读写文件，选择合适的 filesystem backend 就可能足够。但如果 Agent 需要运行测试、安装依赖、执行脚本、调用命令行工具，Sandbox 就会变得重要。DeepAgents 的生产部署文档也指出，filesystem backends 足以支撑文件读写；一旦 Agent 需要运行代码、安装包或执行超过文件 I/O 的操作，就应该使用 sandbox。

Permissions 继续提供更细粒度的控制。

### 4.3 Permissions：哪些操作被允许
DeepAgents overview 中提到，permissions 可以声明文件系统访问规则，用来控制 Agent 可以读写哪些文件和目录；subagents 可以继承或覆盖父 Agent 的规则。[^deepagents-overview] Permissions 文档也说明，当需要对内置 filesystem tools 进行 path-based allow / deny 控制时，可以使用 permissions；如果需要更复杂的校验逻辑，例如 rate limiting、audit logging、content inspection，或者控制自定义工具，则应使用 backend policy hooks。[^deepagents-permissions]

如果把 Backends、Sandbox 和 Permissions 放进一个执行链路中，可以写成：
```text
模型请求文件或命令操作
  -> Runtime 解析工具调用
  -> Permission 检查路径和操作规则
  -> Backend 决定文件操作落点
  -> Sandbox 决定命令执行环境
  -> Tool Executor 执行
  -> 结果返回 Agent Loop
  -> Event Log 记录执行轨迹
```
在这条链路中，模型并不直接接触真实文件系统或宿主机 shell。它只是请求一个工具调用。Runtime 则负责判断这个调用是否允许、应该在哪个 backend 上执行、是否需要 sandbox、是否要进入 HITL 审批、执行结果如何返回和记录。

这就是复杂 Agent 与普通脚本自动化的差异。脚本通常默认拥有运行它的进程权限；Agent Runtime 则需要把每一种行动能力都显式纳入边界控制。

### 4.4 三类边界的关系
这一章可以用一个总结表收束：

| 机制           | 回答的问题          | 在 DeepAgents 中的意义                               |
| ------------ | -------------- | ----------------------------------------------- |
| Backends     | 文件和状态落在哪里      | 让工作区可以使用临时 state、本地磁盘、长期 store、sandbox 或混合路由    |
| Sandbox      | Shell 和代码在哪里执行 | 把命令执行限制在隔离环境中，保护宿主机文件、凭据和网络                     |
| Permissions  | 哪些路径和操作被允许     | 对文件系统读写做路径级 allow / deny 控制，也可为 subagent 设置不同规则 |
| Policy Hooks | 是否需要更复杂的校验     | 支持审计、内容检查、速率限制、自定义工具控制等策略                       |
| HITL         | 哪些敏感动作需要人类确认   | 在执行前暂停，等待用户 approve、edit、reject 或 respond       |

## 5. Memory、Skills 与 HITL：长期连续性、能力加载和人工控制
到第 4 章为止，DeepAgents 的复杂任务执行结构已经比较清楚了。

Planning 让 Agent 拥有外部化的任务状态；Filesystem 与 Context Engineering 让中间产物可以移出上下文窗口，并在需要时重新装配；Subagents 让局部任务在隔离上下文中执行；Backends、Sandbox 与 Permissions 则为文件、状态和命令执行提供持久化边界与安全边界。

接下来还需要补上几类配套机制。它们不一定构成主执行链路，但会改变 Agent Loop 的工作条件，可以理解为复杂任务执行的“条件层”：
```text
Memory:
  Agent 能记住什么

Skills:
  Agent 能按需加载什么能力

HITL:
  哪些动作必须交由人类确认
```
它们分别决定 Agent 是否具有跨任务连续性，是否能在执行中加载专门能力，以及在高风险操作前是否会暂停并等待用户决策。

### 5.1 Memory：跨任务连续性
Memory 解决的是跨任务连续性问题。

前一篇已经讨论过，Memory 和 Context 需要区分：Memory 是外部保存的信息库存，Context 是每次模型调用时被 Runtime 装配进上下文窗口的工作集。[^Agent-Runtime]

在 DeepAgents 中，这个区分仍然成立。

DeepAgents memory 文档明确区分 short-term memory 和 long-term memory。Short-term memory 包括单个 session 内的 conversation history 和 scratch files，由 agent state 自动管理；long-term memory 则指跨 conversations 持久化的信息。[^deepagents-memory] LangChain 的长期记忆文档也说明，long-term memory 可以跨不同 conversations 和 sessions 保存与召回信息，并基于 LangGraph stores，以 namespace 和 key 组织 JSON documents。[^langchain-long-term-memory]

可以简单整理为：
```text
Short-term memory:
  当前 thread / session 内的消息历史、scratch files、任务状态

Long-term memory:
  跨 conversations / threads 保存的用户偏好、项目背景、长期事实和经验
```
这一区分很重要。因为并非所有信息都应该进入长期记忆。

比如，当前任务中的临时草稿、某次测试输出、某个失败分支，通常只需要保存在当前 thread 或 filesystem 中。它们服务于当前任务，不一定值得跨任务复用。

而用户长期偏好、项目约定、稳定背景信息、常用技术栈、某个仓库的长期结构认知，则更适合进入 long-term memory。

Memory 的价值在于，让 Agent 不必每次从零开始理解用户和任务环境。

但这些记忆不应该被无脑塞进每一轮上下文。Runtime 仍然需要通过 Context Assembly 判断哪些 memory 与当前任务相关，哪些可以忽略，哪些只需要摘要，哪些需要完整注入。

这也避免了一个常见误区：Memory 越多，Agent 越强。实际上，未经筛选的 memory 也可能变成上下文噪音。真正重要的是记忆的写入策略、检索策略、冲突处理和上下文装配。

所以，在 DeepAgents 这一层，Memory 的意义不是让模型“永远记住一切”，而是为复杂任务提供可控的长期连续性。

### 5.2 Skills：按需加载能力

Skills 解决的是能力加载问题。

如果说 Memory 保存的是“Agent 过去知道什么”，那么 Skills 更接近“Agent 在当前任务中可以临时学会什么”。

DeepAgents skills 文档将 skills 定义为 reusable agent capabilities，可以为 deep agent 提供 specialized workflows 和 domain knowledge；skills 遵循 Agent Skills specification，通常通过 `SKILL.md` 这样的文件提供说明和元数据。[^deepagents-skills]

复杂 Agent 常常需要很多专门能力，例如：

```text
如何写一篇技术博客
如何审查 Python 代码
如何分析 LangGraph 项目
如何生成数据库迁移
如何做安全检查
如何处理论文阅读笔记
如何调用某个内部 API
如何遵守某个团队文档规范
```
如果把这些内容全部常驻在 system prompt 中，会带来几个问题：
```text
prompt 过长
当前任务无关信息过多
模型更容易混淆不同规范
维护成本高
工具和知识更新不灵活
```
Skills 的思路是把这些能力做成可按需加载的能力包。Agent 在判断当前任务需要某类专门知识时，再读取对应 skill 的说明、规范、示例或辅助文件。

可以这样理解：
```text
System Prompt:
  放通用身份、基础规则和默认工作方式

Skills:
  放特定任务、领域或工作流的详细说明

Context Assembly:
  当前任务需要时，再把相关 skill 装配进上下文
```
Skills 与工具也不同。

工具提供的是行动接口，例如搜索、读文件、写文件、执行命令。Skills 提供的是操作知识，例如什么时候使用某个流程，输出结构应该是什么，如何检查结果，哪些坑需要避免。
```text
Tools:
  Agent 能做什么动作

Skills:
  Agent 应该如何完成某类任务
```
在复杂 Agent 中，Skills 可以降低 system prompt 的常驻负担，也能让能力扩展更模块化。新增一种任务能力时，不一定需要重写整个 Agent，只需要添加一个新的 skill 包，并让 Agent 在合适时机加载它。

### 5.3 HITL：高风险动作的人类控制点

HITL 解决的是人类控制问题。

前一章讨论了 Sandbox 和 Permissions，它们提供的是系统层面的边界：哪些路径可以读写，命令在哪里执行，执行环境是否隔离。HITL 则进一步回答：
```text
哪些动作即使技术上允许，也需要人类确认？
```
DeepAgents HITL 文档说明，Deep Agents 通过 LangGraph 的 interrupt 能力支持 human-in-the-loop workflow，可以使用 `interrupt_on` 参数配置哪些工具需要审批。[^deepagents-hitl] 该文档也提到，面对敏感工具操作时，系统可以在执行前暂停，并让人类做出 approve、edit、reject、respond 等决策。[^deepagents-hitl]

把这一章和前面几章合起来看，DeepAgents Harness 的结构就比较完整了：
```text
Planning:
  把任务进度外部化

Filesystem:
  把中间产物外部化

Context Engineering:
  决定哪些信息进入当前模型上下文

Subagents:
  把局部复杂任务隔离出去

Backends / Sandbox / Permissions:
  决定文件、状态和命令执行的边界

Memory / Skills / HITL:
  分别提供长期连续性、按需能力加载和人工控制点
```

所以，DeepAgents 的核心不只是提供更多工具，而是把复杂任务执行中最容易失控的部分系统化：计划要能保存，产物要能落盘，上下文要能压缩，子任务要能隔离，执行环境要有边界，高风险动作要能暂停，长期经验和专门能力也要能被 Runtime 按需装配。
