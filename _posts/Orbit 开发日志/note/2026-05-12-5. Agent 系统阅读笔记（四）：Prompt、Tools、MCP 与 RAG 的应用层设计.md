# 5. Agent 系统阅读笔记（四）：Prompt、Tools、MCP 与 RAG 的应用层设计

前面三篇主要在看 Agent Runtime 与 Harness。

第一篇从 Claude Code / Claw Code、ChatGPT Memory 和 Tool Surface 等角度，整理现代 Agent Runtime 的系统形态。[^Agent-Runtime] 第二篇从 LangChain Agent、LangGraph Runtime 到 DeepAgents，讨论基础 tool calling loop 如何获得状态、持久化、中断恢复和复杂任务 Harness。[^Agent-Harness] 第三篇则进一步拆开 DeepAgents Harness：Planning、Filesystem、Context Engineering、Subagents、Backends、Sandbox、Permissions、Memory、Skills 和 HITL。[^DeepAgents-Harness]

这些内容更多回答的是框架层问题：
```text
Agent Loop 如何成立？
Runtime 如何保存、恢复和观察执行过程？
Harness 如何组织任务、文件、上下文、子任务和安全边界？
```

但真正做一个 Agent 应用时，还会遇到另一组问题：
```text
Prompt 应该怎么组织？
Context 应该怎么选择、压缩和注入？
Tool schema 应该怎么设计？
Tool call 失败后怎么处理？
Skills 和 Tools 如何分工？
MCP 如何把外部工具接入 Tool Surface？
CLI Agent 如何把运行过程暴露给用户？
RAG、Memory、Filesystem 和 Context Assembly 的边界在哪里？
```

这些问题不完全属于 Runtime，也不完全属于 Harness。它们更接近 Agent Application Engineering。

可以先粗略分层：
```text
Runtime Layer:
  loop、state、checkpoint、streaming、interrupt、store

Harness Layer:
  planning、filesystem、subagents、sandbox、permissions、memory、skills、HITL

Application Layer:
  prompt、context assembly、tool schema、tool lifecycle、MCP、CLI、RAG、eval
```

也就是说，Runtime 和 Harness 提供的是 Agent 应用的底座；Application Layer 负责决定这些能力如何组合成一个真正可用、可控、可解释的产品。

## 1. 从 Harness 到 Application Layer

如果只看框架层，一个 Agent 可以被描述成：
```text
Model + Tools + Messages + Runtime + Harness
```

但对应用开发者来说，问题通常更具体：
```text
用户输入进入系统后，应该被改写吗？
哪些系统提示每次都要注入？
哪些工具应该暴露给模型？
工具返回值应该完整给模型，还是写入文件后只给摘要？
检索结果应该作为 RAG context 注入，还是作为工具结果返回？
技能说明应该常驻，还是按需加载？
MCP server 中的工具是否需要二次包装？
CLI 中哪些动作需要展示、确认或阻断？
```

这些问题共同决定 Agent 的实际表现。很多时候，Agent 失败并不是因为 Runtime 不够强，而是因为应用层没有把正确的信息、工具和约束交给模型。

LangChain 的 context engineering 文档也强调，Agent 失败常常不是模型能力不足，而是模型调用时没有拿到正确上下文；context engineering 的核心就是以正确格式提供正确的信息和工具。[^langchain-context-engineering]

因此，应用层的核心任务可以概括为：
```text
让模型在每一次行动前，看到它当前真正需要看的东西；
让模型在每一次行动时，只能调用当前真正应该调用的能力；
让 Runtime 在每一次行动后，正确保存结果、展示过程并处理风险。
```

这也是为什么 prompt、context、tool、skills、MCP、CLI 和 RAG 不能分开看。它们共同构成模型与外部世界之间的应用层接口。

## 2. Prompt Engineering：定义默认行为和边界

Prompt Engineering 在 Agent 应用中不是单纯写一段好看的 system prompt，而是定义模型在 Runtime 中的默认行为。

### 2.1 Prompt 放什么

一个 Agent 的 prompt 至少需要说明几类内容：
```text
角色:
  这个 Agent 是什么类型的助手

目标:
  当前任务应该优化什么结果

工作方式:
  先读上下文、再计划、再行动、最后总结

工具使用规则:
  什么时候调用工具，什么时候不要调用工具

安全边界:
  哪些动作需要确认，哪些内容不能泄露或修改

输出格式:
  最终回答、阶段性更新、错误说明应该如何组织
```

在简单 Chatbot 中，prompt 更多影响回答风格；在 Agent 中，prompt 会直接影响模型如何行动。

例如，同样是一个代码助手，如果 prompt 没有明确要求“修改前先阅读相关文件、修改后运行相关测试、测试失败要继续定位原因”，模型可能会更倾向于直接给出解释，而不是进入工具执行闭环。

所以 Agent prompt 不只是语言风格约束，而是行为协议。

### 2.2 Prompt 不应该放什么

Prompt 也不应该承载所有东西。

如果把所有领域知识、团队规范、工具说明、项目背景、用户偏好和历史经验都塞进 system prompt，会带来几个问题：
```text
prompt 过长；
当前任务无关信息过多；
模型容易混淆不同规范；
维护成本高；
不同用户、项目、任务之间难以隔离。
```

更合理的做法是分层：
```text
System Prompt:
  放通用身份、基础工作方式和安全边界

Task Prompt:
  放当前任务目标和用户输入

Context:
  放当前任务真正需要的背景、文件片段、检索结果和状态

Skills:
  放可按需加载的专门工作流和领域规范

Tools:
  放行动能力及其 schema，而不是长篇操作说明
```

这也解释了为什么 DeepAgents 要有 skills 和 context engineering。Prompt 不是越长越强；Agent 的稳定性更多来自把不同类型的信息放在正确位置。

### 2.3 Prompt 与 Harness Prompt

在 DeepAgents 中，planning、filesystem、subagent、HITL 和 local context 等 Harness 能力也会通过 middleware 向 system prompt 注入相关说明。[^deepagents-context]

这意味着 Agent prompt 不是一个静态字符串，而是由多层内容组成：
```text
Base System Prompt:
  Agent 的通用身份和行为规则

Harness Prompt:
  planning、filesystem、subagent、HITL 等机制的使用说明

Task Prompt:
  用户当前任务

Dynamic Context:
  当前状态、记忆、文件片段、检索结果、工具选择等
```

从应用工程角度看，prompt engineering 与 harness engineering 会交织在一起。开发者不只是写提示词，还要设计哪些运行时能力会以什么形式进入模型上下文。

## 3. Context Engineering：决定模型每轮看到什么

Context Engineering 是 Agent 应用层最核心的问题之一。

在普通问答应用中，context 可能只是用户消息、系统提示和少量检索结果。但在 Agent 中，context 来源更多：
```text
用户输入
系统提示
当前 messages
任务计划 / todos
最近工具调用结果
文件系统中的草稿和日志
长期 memory
RAG 检索结果
可用 tools
skills 说明
runtime context 中的权限和配置
```

这些内容不应该无差别进入模型上下文。

### 3.1 Context 的几类来源

LangChain context engineering 文档把数据来源区分为 Runtime Context、State 和 Store：Runtime Context 更像会话级静态配置，例如用户 ID、API key、数据库连接、权限和环境设置；State 是当前会话中的短期状态，例如 messages、上传文件、认证状态和工具结果；Store 则是跨会话长期记忆，例如用户偏好、历史数据和抽取出的 insights。[^langchain-context-engineering]

结合前几篇，可以把 Agent context 来源整理为：

| 来源 | 保存内容 | 是否直接给模型看 |
| --- | --- | --- |
| Messages | 用户输入、模型输出、tool call、tool result | 经常进入，但需要压缩 |
| Task State | todos、当前步骤、阻塞点 | 应该进入，通常以结构化摘要进入 |
| Filesystem | 草稿、日志、代码、报告、中间产物 | 按需读取，不应全量进入 |
| Memory / Store | 用户偏好、长期事实、项目背景 | 检索后选择性进入 |
| RAG Index | 外部知识库、文档、代码说明 | 根据查询检索后进入 |
| Runtime Context | 用户 ID、权限、API client、数据库连接 | 通常不给模型直接看 |
| Skills | 专门工作流和领域规范 | 按需加载 |
| Tools | 可执行能力及 schema | 根据阶段和权限选择性暴露 |

这张表的关键是，context 不只是文本。工具列表本身也是 context，文件路径也是 context，任务状态也是 context，甚至当前用户权限也会影响模型可见的能力边界。

### 3.2 Select、Compress、Isolate

复杂 Agent 的上下文管理至少要处理三个动作：
```text
Select:
  从 messages、files、memory、RAG、skills 中选择当前任务相关内容

Compress:
  对长历史、长工具结果、长文件和搜索结果做摘要或裁剪

Isolate:
  把局部任务放入 subagent 或临时上下文，避免污染主上下文
```

这和第三篇讨论的 Filesystem、Context Engineering 和 Subagents 是连贯的。Filesystem 让信息可以移出上下文窗口；Context Engineering 决定何时取回；Subagents 则把局部探索隔离在独立上下文中。

### 3.3 Transient 与 Persistent

这里还需要区分 transient context 和 persistent context。

```text
Transient Context:
  只影响当前这一次模型调用
  例如临时注入的检索结果、当前文件片段、动态筛选后的工具列表

Persistent Context:
  会写入 state、store 或 filesystem
  例如 todo 状态、阶段性摘要、长期记忆、生成的报告草稿
```

LangChain 文档也强调，middleware 可以在 agent 生命周期中更新 context，其中 transient message updates 只改变当前模型调用，而 persistent updates 会修改 state，使后续轮次继续看到变化。[^langchain-context-engineering]

这个区分很重要。很多上下文只应该临时出现，例如一次 RAG 检索结果；但任务计划、阶段摘要和用户偏好则可能需要持久化。应用层需要决定每类信息的生命周期。

## 4. Tool Call Engineering：让模型可靠使用行动能力

Tool Call Engineering 关注的是：模型如何可靠、安全、可观察地调用外部能力。

LangChain 的 Tools 文档将 tools 描述为让 Agent 获取实时数据、执行代码、查询数据库和采取行动的 callable functions；模型会根据上下文决定何时调用工具，以及提供什么输入参数。[^langchain-tools]

但应用层不能只把函数暴露出去。一个成熟工具至少要设计几类内容：
```text
name:
  工具叫什么

description:
  什么时候应该使用它，什么时候不应该使用

input schema:
  参数结构、类型、必填项、枚举范围

output format:
  返回给模型的内容应该多长、是否结构化、是否包含 artifact

permission:
  调用前是否需要权限或用户确认

error handling:
  失败后返回什么信息给模型

observability:
  如何记录调用、参数、耗时、结果和错误
```

### 4.1 Tool Schema 是模型的操作界面

Tool schema 不只是类型约束，也是模型理解工具的主要界面。

例如，一个检索工具可以设计成：
```text
retrieve(query: string)
```

也可以设计成：
```text
retrieve(
  query: string,
  source: "docs" | "code" | "issues",
  recency: "latest" | "stable",
  max_results: number
)
```

前者简单，但模型没有办法表达搜索范围；后者更可控，但也增加了模型填错参数的可能。工具参数不是越多越好，而是要把当前任务中真正有价值的控制维度暴露出来。

LangChain 的 RAG agent 示例中，retrieval tool 既可以只接收一个 query，也可以增加额外搜索参数，例如类别；同时还可以使用 `content_and_artifact` 形式，让模型看到序列化内容，而应用侧保留原始文档 metadata。[^langchain-rag]

这说明工具返回值也需要设计：
```text
给模型的内容:
  简洁、可读、足够完成当前推理

给应用的 artifact:
  原始文档、metadata、trace、可审计对象
```

模型不一定需要看到全部原始数据；应用可能需要保留原始数据用于引用、审计和后续处理。

### 4.2 Tool Call 生命周期

一个工具调用从模型视角看可能只是：
```text
call tool -> get result
```

但从应用层看，它至少经历一条 pipeline：
```text
模型生成 tool call
  -> Runtime 解析工具名和参数
  -> Schema 校验
  -> Permission / HITL 检查
  -> Tool Executor 执行
  -> 捕获 stdout / stderr / exception
  -> 结果压缩或写入文件
  -> ToolMessage 返回模型
  -> Event Log / Trace 记录
  -> UI 展示 pending / completed / error
```

LangChain 的 tool calling UI 文档也把工具调用拆成 `pending`、`completed` 和 `error` 等状态，并建议 UI 处理 loading、error 和未知工具的 fallback 展示。[^langchain-tool-calling-ui]

这说明 tool call 不只是模型和函数之间的接口，也是用户观察 Agent 行动过程的接口。

### 4.3 Tool Error Handling

工具失败不是异常情况，而是 Agent 执行过程的常态。

常见失败包括：
```text
参数不合法
目标资源不存在
网络超时
权限不足
命令执行失败
返回内容过长
外部 API 限流
```

应用层需要决定：这些错误是直接抛出，还是转成可供模型继续推理的 ToolMessage。

LangChain Agents 文档中展示了通过 middleware 包装 tool call，在工具异常时返回自定义 ToolMessage，使模型能够看到错误并尝试修正输入或采取替代路径。[^langchain-agents]

对 Agent 来说，错误处理本身也是 context engineering。一个好的错误消息应该告诉模型：
```text
失败在哪里；
为什么失败；
是否可以重试；
应该调整什么参数；
是否需要用户介入；
是否已经产生了部分结果。
```

如果只返回一段堆栈，模型可能无法有效恢复；如果过度隐藏错误，模型又会误以为操作成功。

## 5. Skills、Tools 与 Prompt 的分工

前一篇已经讨论过 Skills：它更像按需加载的操作知识，而不是行动接口。[^DeepAgents-Harness]

在应用层，Skills、Tools 和 Prompt 的分工可以更明确：

| 机制 | 主要回答 | 例子 |
| --- | --- | --- |
| Prompt | Agent 默认应该如何工作 | 先计划、再执行、必要时确认 |
| Skill | 某类任务应该如何完成 | 如何做代码审查、如何写技术博客 |
| Tool | Agent 可以执行什么动作 | 读文件、搜索、运行测试、调用 API |

可以用一句话区分：
```text
Prompt 定义默认行为；
Skills 定义专门工作流；
Tools 提供外部行动能力。
```

### 5.1 为什么 Skills 不等于 Tools

工具是可执行接口。它们通常有明确输入、输出和副作用。

Skills 则是任务知识包。它们告诉模型如何完成某类工作，包括步骤、约束、质量标准、常见陷阱和检查方式。

例如：
```text
Tool:
  run_tests(command)
  read_file(path)
  search_docs(query)

Skill:
  代码审查应该先看接口契约，再看状态变化，再看错误处理和测试覆盖
  技术博客应该先建立问题，再给系统分层，最后回到工程启发
```

如果把 Skill 写成 Tool，模型会得到一个“执行接口”，但不一定知道如何组织任务。反过来，如果把 Tool 写成 Skill，模型知道原则，却没有行动能力。

### 5.2 Skills 与 Prompt 的边界

Skills 也不应该全部塞进 system prompt。

System prompt 更适合放稳定、通用、每次都需要的行为规则；Skills 更适合放可选、专门、任务相关的工作流。

例如：
```text
System Prompt:
  你是一个谨慎的 Coding Agent，修改前先阅读相关文件，修改后运行验证。

Code Review Skill:
  审查时优先找行为 bug、回归风险、缺失测试和安全边界问题。

Blog Writing Skill:
  写作时先建立主线，再用系统分层组织概念，避免只堆材料。
```

这样做的好处是：系统提示保持稳定，专门能力可以按需加载，任务之间也不容易互相污染。

## 6. MCP：把外部能力接入 Tool Surface

MCP 解决的是外部工具和上下文接入问题。

Model Context Protocol 官方规范将 MCP 描述为一个开放协议，用于让 LLM 应用和外部数据源、工具之间实现标准化集成。[^mcp-spec] 从 Agent 应用角度看，MCP 的价值在于：它把外部系统转换成 Agent 可以发现和调用的标准能力。

### 6.1 MCP 在 Agent 系统中的位置

可以把 MCP 放在 Tool Surface 和外部系统之间：
```text
Model
  -> Tool Call
  -> Agent Runtime / Tool Registry
  -> MCP Client
  -> MCP Server
  -> External System
```

MCP server 可以暴露工具、资源和 prompts：
```text
Tools:
  可被模型调用的动作，例如查询数据库、调用 API、执行计算

Resources:
  可读取的上下文，例如文件、文档、数据库记录、日志

Prompts:
  可复用的提示模板或工作流入口
```

MCP Tools 文档说明，MCP server 可以暴露可由语言模型调用的 tools，每个 tool 有唯一 name，并包含描述 schema 的 metadata。[^mcp-tools] MCP Prompts 文档则说明，server 可以暴露 prompt templates，client 可以发现、获取并传入参数进行定制。[^mcp-prompts]

这说明 MCP 不只是“远程函数调用协议”。它更像是 Agent 应用发现外部能力和上下文的一层协议。

### 6.2 MCP 不等于直接信任外部工具

MCP 让接入更标准，但不意味着外部工具可以直接裸露给模型。

应用层仍然需要处理：
```text
工具命名是否适合模型理解；
description 是否足够清楚；
参数 schema 是否需要二次包装；
哪些工具应该常驻暴露；
哪些工具需要按需发现；
哪些工具需要权限确认；
工具结果是否需要压缩、过滤或写入文件；
资源内容是否可能包含 prompt injection。
```

MCP 把外部能力接进来；Tool Call Engineering 决定这些能力如何安全、可靠地进入 Agent Loop。

## 7. CLI Agent：应用层的交互外壳

Coding Agent 很多时候以 CLI 形态出现。CLI 不是简单的输入输出壳，而是 Agent Runtime 暴露给用户的操作界面。

一个成熟 CLI Agent 至少要处理：
```text
一次性 prompt 和交互式 REPL
流式输出
工具调用展示
文件 diff 展示
命令执行确认
权限模式切换
工作区边界
会话保存和恢复
任务状态展示
错误和重试
MCP server 配置
```

第一篇中拆过 Claw Code 的模块划分，其中 CLI binary 处理 REPL、一次性 prompt、流式展示和命令行参数；runtime 则处理会话运行时、配置加载、持久化、权限策略、MCP 生命周期和系统提示组装。[^Agent-Runtime]

这说明 CLI Agent 不是单纯把 Chatbot 放进终端，而是把 Agent 的执行过程产品化：
```text
模型正在想什么？
准备调用什么工具？
即将修改哪些文件？
命令是否需要确认？
执行失败后如何展示？
任务中断后如何恢复？
```

CLI 的价值在于把 Runtime / Harness 的状态变成用户可以理解和控制的交互过程。

## 8. RAG：上下文供给机制，而不是完整记忆系统

RAG 在 Agent 应用中经常和 Memory、Filesystem、Context Engineering 混在一起，但它们不是同一个东西。

LangChain Retrieval 文档指出，LLM 有有限上下文和静态知识两个限制；retrieval 在查询时获取相关外部知识，是 RAG 的基础。[^langchain-retrieval] LangChain 的 RAG agent 文档也展示了两类实现：一种是 agent 通过 retrieval tool 自主检索，另一种是固定的两步 RAG chain，先检索再生成。[^langchain-rag]

可以这样区分：

| 机制 | 保存什么 | 什么时候用 |
| --- | --- | --- |
| RAG Index | 外部知识库、文档切片、代码文档 | 当前问题需要外部知识时检索 |
| Memory | 用户偏好、长期事实、项目背景 | 需要跨会话连续性时召回 |
| Filesystem | 当前任务草稿、日志、报告、代码改动 | 当前任务执行过程中读写 |
| Context Assembly | 当前模型调用的工作集 | 每轮模型调用前装配 |

RAG 的核心不是“让 Agent 记住更多东西”，而是让 Agent 能在需要时从外部知识库取回相关材料。

### 8.1 RAG 作为 Tool

在 Agentic RAG 中，检索可以表现为一个工具：
```text
retrieve_context(query)
```

模型根据当前任务判断是否需要检索、检索什么、是否需要多次检索或改写查询。LangGraph 的 agentic RAG 教程也将 retrieval agent 描述为让 LLM 决定是否从 vectorstore 检索上下文，还是直接回答。[^langgraph-agentic-rag]

这种方式适合复杂问题，因为模型可以迭代：
```text
先检索背景；
发现信息不足；
改写查询；
再次检索；
综合回答。
```

### 8.2 RAG 作为 Middleware

RAG 也可以不暴露为工具，而是在模型调用前由 middleware 自动注入。

例如：
```text
用户问题进入系统
  -> middleware 根据问题检索相关文档
  -> 把文档片段注入当前 model context
  -> 模型直接回答
```

这种方式更像传统 RAG chain，适合问题类型稳定、检索策略明确的场景。

两种方式的区别在于：
```text
RAG as Tool:
  模型决定是否检索、检索什么、检索几次

RAG as Middleware:
  应用决定每次都检索，并控制注入方式
```

应用层需要根据任务类型选择。复杂调研、代码问答和多步任务更适合 agentic RAG；稳定问答、客服知识库和固定业务查询更适合 middleware / chain 形式。

## 9. 小结：应用层设计的是模型如何使用能力

回到这一篇的主线。

Runtime 和 Harness 提供了 Agent 的执行底座：
```text
Runtime:
  loop、state、checkpoint、streaming、interrupt、store

Harness:
  planning、filesystem、subagents、sandbox、permissions、memory、skills、HITL
```

Application Layer 则决定这些能力如何被模型实际使用：
```text
Prompt:
  定义默认行为和边界

Context Engineering:
  决定模型每轮看到什么

Tool Call Engineering:
  让模型可靠、安全、可观察地调用工具

Skills:
  按需加载专门工作流和领域规范

MCP:
  把外部工具、资源和 prompts 接入 Tool Surface

CLI:
  把执行过程变成用户可观察、可确认、可恢复的交互

RAG:
  在需要时为模型提供外部知识
```

所以，Agent 应用工程的关键并不是“多接几个工具”，而是设计一套模型使用能力的方式：
```text
什么信息应该进入上下文；
什么能力应该暴露为工具；
什么知识应该放进 skill；
什么外部系统应该通过 MCP 接入；
什么内容应该通过 RAG 检索；
什么操作应该由 CLI 展示和确认；
什么状态应该被 Runtime 持久化。
```

如果说 Runtime 解决 Agent 能不能持续运行，Harness 解决复杂任务能不能被组织起来，那么 Application Layer 解决的是：Agent 能不能在具体产品场景中可靠地使用这些能力。
