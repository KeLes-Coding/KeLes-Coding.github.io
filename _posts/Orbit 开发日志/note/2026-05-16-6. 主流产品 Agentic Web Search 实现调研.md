# 6. 主流产品 Agentic Web Search 实现调研

Orbit 计划引入 DeepAgent，计划最小实现链路使用 Agentic Web Search 来实现，本篇关注于：

> 如果用一个多轮 `web search -> fetch -> summarize -> maybe search again -> final answer` 的 loop 作为第一阶段验证，当前主流产品大致是如何实现的？

---

## 1. OpenAI / ChatGPT
### 1.1 实现形态

OpenAI 当前公开路线更接近：

```text
Responses API
  -> model
  -> hosted tools
      -> web_search
  -> tool events
  -> final response
```

也就是说，搜索不是业务侧手写的普通接口，而是作为模型可调用的 hosted tool，被纳入一次 response / run 中。[^openai-web-search]

### 1.2 关键特征

- 模型可以在回答过程中调用 `web_search`。同时，发现即使不做手动调用，模型遇到复杂任务/有需要的场景也会自动开启 web search。（web 端）
- 搜索结果和最终回答属于同一次生成过程。
- 对于长耗时任务，OpenAI 提供 background mode，用于支持更长时间的 reasoning / agent 任务。[^openai-background]
- 结合上一篇对 ChatGPT Web 产品的观察，页面刷新后也存在恢复当前回答流的机制。

---

## 2. Gemini（AI Studio）

### 2.1 实现形态

Gemini 的公开能力可以拆成两层：

```text
Google Search Grounding
  -> 负责搜索与引用

URL Context
  -> 负责读取指定网页内容
```

其中 Search Grounding 用于让模型连接实时 Web 信息并提供来源，URL Context 则用于读取和理解指定 URL。[^gemini-search][^gemini-url-context]

### 2.2 关键特征

- Search Grounding 负责获取实时 Web 信息。
- URL Context 负责深入读取具体网页。
- 两者可以组合使用：先搜索，再读取 URL，再生成答案。
- 最终回答通常可以带来源信息。

### 2.3 简单判断

Gemini 的形态非常接近 Orbit 想验证的：

```text
search
  -> get candidate urls
  -> fetch/read url
  -> synthesize answer
```

说明 search 和 fetch/read 最好是两个不同层级的能力，而不是一个混在一起的大工具。

## 3. Qwen

### 3.1 实现形态

Qwen 公开 API 中提供了 `enable_search` 等搜索相关参数。其搜索策略中包含 `turbo`、`max`、`agent`、`agent_max` 等模式。[^qwen-search]

大致可以理解为：

```text
turbo      快速搜索
max        更全面搜索
agent      多轮搜索与整合
agent_max  多轮搜索 + 网页抓取
```

其中 `agent` / `agent_max` 已经接近 Agentic Web Search。

### 3.2 关键特征

- 可以让模型自行决定是否联网搜索。
- 也可以强制搜索。
- agent 模式下允许多轮检索与内容整合。
- agent_max 进一步加入网页抓取能力。
- Qwen 还提供网页抓取 / Web Extractor 能力，用于访问指定 URL 并提取网页内容。[^qwen-web-extractor]

### 3.3 简单判断

Qwen 的设计比较有工程参考价值，因为它明确区分了：

```text
是否搜索
搜索强度
是否进入 agent 多轮搜索
是否抓取网页正文
```

这说明 Agentic Web Search，可以按成本、速度和质量拆成多档策略。

## 4. DeepSeek

### 4.1 实现形态

DeepSeek 网页端已经支持联网搜索，但 DeepSeek 官方文档说明，目前 API 暂不支持搜索功能。[^deepseek-search]

也就是说：

```text
DeepSeek 网页端：
  DeepSeek 自己实现搜索系统

Orbit 接 DeepSeek API：
  Orbit 需要自己实现 search / fetch 工具
```

### 4.2 关键特征

- 网页产品中搜索体验是产品内置能力。而且需要手动开启。
- API 侧不能直接依赖 DeepSeek 提供 search。
- 如果 Orbit 使用 DeepSeek 作为模型，需要在 Orbit 侧自行实现工具层。
- DeepSeek 官方描述其联网搜索会深入阅读网页，并在复杂问题下自动提取多个关键词并行搜索。[^deepseek-search]

## 5. 腾讯元宝

### 5.1 实现形态
腾讯云公开文章中提到，腾讯 AI 搜索基于混元模型，结合内部多生态检索增强与 Agent 架构，已应用于腾讯元宝、QQ 浏览器等产品场景。复杂需求下可以启动多轮反思机制。[^tencent-yuanbao]

其 DeepSearch 形态大致是：

```text
Agentic RAG
  -> 规划
  -> 搜索
  -> 阅读
  -> 反思
  -> 再搜索
  -> 回答
```

### 5.2 关键特征

- 搜索不是单一 Web Search，而是多生态检索。
- 复杂问题会进入多轮搜索与反思。
- DeepSearch 会通过“规划-搜索-阅读-反思”循环调用多个工具。[^tencent-yuanbao]
- 结合上一篇观察，元宝页面刷新后也可以恢复当前回答流。

### 5.3 简单判断

```text
多源检索
  -> Agentic RAG
  -> 多轮反思
  -> 结构化回答
```

## 6. 主流实现共性

### 6.1 搜索属于 Agent Run，而不是普通接口

主流产品不会简单做：

```text
search -> 拼 prompt -> answer
```

而更接近：

```text
model
  -> decide search (planning)
  -> search
  -> read
  -> reflect
  -> maybe search again
  -> final answer
```

也就是说，搜索过程本身就是一次 Agent Run 的一部分。

---

### 6.2 search 和 fetch/read 通常是两层能力

从 Gemini 的 Search Grounding + URL Context，以及 Qwen 的 search + web extractor 都可以看出，搜索和网页读取最好分开处理。

推荐抽象为：

```text
web_search:
  负责发现网页

fetch_url:
  负责读取网页正文
```

这样后续更容易做：

```text
工具轨迹展示
网页缓存
来源去重
失败重试
引用管理
```

---

### 6.3 多轮 loop 必须有边界

Agentic Web Search 一旦允许多轮循环，就必须限制成本和时长。

第一版至少需要限制：

```text
max_iterations
max_search_calls
max_fetch_calls
max_chars_per_page
timeout_seconds
```

否则很容易出现：

```text
反复搜索
抓取过多网页
上下文爆炸
长时间无输出
用户不知道系统在做什么
```

---

### 6.4 前端展示不应直接展示 thinking

更稳妥的方式是展示产品化事件：

```text
plan
tool_start
tool_end
source
content_delta
final
error
```

而不是直接展示模型私有 thinking。

产品上可以做成：

```text
计划：我会先搜索资料，再阅读关键网页。
工具：web_search(...)
工具：fetch_url(...)
结果：找到 4 个相关来源。
回答：……
```

---

## 7. 结论

当前主流产品的趋势很明确：

```text
普通搜索问答
  -> Agentic Web Search
  -> DeepResearch / DeepSearch
```

Orbit 第一阶段选择 `web_search + fetch_url` 作为 DeepAgent 验证场景是合适的。

它刚好覆盖：

```text
多轮 loop
工具调用
外部 I/O
事件流
中间状态
来源展示
运行恢复
失败处理
```

第一阶段不要追求搜索质量极致，而是优先打通：

```text
一次 Agent Run 是否可观察、可恢复、可调试、可复现。
```

这条链路打通后，后续再扩展 DeepResearch、文件系统、GitHub、本地项目工具，都会自然很多。
