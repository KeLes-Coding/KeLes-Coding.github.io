# 3. Agent 系统阅读笔记（二）：从 LangChain Agent 到 DeepAgents Harness

上一篇[^Agent-Runtime]主要从对 Claude Code 的拆解，以及 ChatGPT Memory 的逆向分析中，观察现代 Agent Runtime 的系统形态。

一个比较直观的感受是，对于复杂任务型 Agent 来说，问题已经从“模型能不能调用工具”，过渡到模型调用工具之后，系统如何继续管理任务、上下文、中间产物、权限、状态和执行环境。

这一篇聚焦于从 LangChain 到 DeepAgents 的演变。

LangChain Agent 可以看作一个基础入口。它把模型、工具和消息组织成一个可持续运行的工具调用循环。模型负责判断下一步是否需要调用工具；Runtime 负责执行工具，并把观察结果重新放回消息流。LangChain 文档中也明确提到，Agent 会组合语言模型和工具，使系统能够围绕任务进行推理、选择工具，并迭代推进到最终结果；create_agent 构建的是一个基于 LangGraph 的 graph-based agent runtime。[^langchain-agents]

但如果只停留在这一层，Agent 仍然更像一个基础执行循环：
```text
用户输入
  -> Runtime 构造 messages 和 tools
  -> 模型判断下一步
  -> 模型发起 tool call
  -> Runtime 执行工具
  -> tool result 返回 messages
  -> 模型继续判断
  -> 输出最终结果
```

这一循环相较于传统的 Chatbot 的一次性问答模式，更加适合复杂、长时序、需要多步推理的工作，但是局限在于没能够回答复杂任务中的几个问题：
```text
任务计划如何外部化？
中间产物如何保存？
长上下文如何压缩和隔离？
子任务如何拆分？
文件系统如何管理？
Shell 命令在哪里执行？
工具权限如何约束？
人类确认如何插入执行过程？
运行状态如何持久化和恢复？
```

这些问题其实已经超出了“tool calling loop”本身。

LangGraph 提供了更底层的运行时能力。LangChain 的 create_agent 底层运行在 LangGraph runtime 上，Runtime 对象可以提供 context、store、stream writer、execution info、server info 等运行期信息。其中 context 可以注入用户 ID、数据库连接等依赖；store 可以用于长期记忆；execution info 可以提供 thread ID、run ID 和重试次数等执行身份。

也就是说，LangChain Agent 负责把模型带入工具调用循环，LangGraph Runtime 负责让这个循环具备状态、依赖注入、流式输出、记忆和执行身份。

DeepAgents 的定位也正好落在这里。官方文档把 deepagents 描述为一个 agent harness：它使用的仍然是和其他 Agent 框架相同的核心 tool calling loop，但额外内置了 task planning、filesystem、subagent spawning、long-term memory 和 pluggable filesystem backends 等能力。DeepAgents 构建在 LangGraph 运行时之上，并复用 LangChain / LangGraph 的 Agent 基础能力，因此能够获得 streaming、persistence、checkpointing 等运行时特性。[^deepagents-overview]

可以简单概括为：
```text
LangChain Agent:
  解决模型如何进入工具调用循环

LangGraph Runtime:
  解决这个循环如何拥有运行时状态

DeepAgents:
  解决复杂任务如何被规划、拆分、隔离、持久化和安全执行
```

这里的关键是，DeepAgents 并不是把 Agent Loop 推翻重做，而是在基础循环之上增加了一套复杂任务所需的 Harness。

可以理解为：
```text
普通 Agent:
Model + Tools + Loop

DeepAgents:
Model + Tools + Loop
+ Planning
+ Filesystem
+ Context Engineering
+ Subagents
+ Memory
+ Sandbox
+ Backends
+ HITL
```

## 1. LangChain Agent：基础工具调用循环
LangChain Agent 可以作为观察基础 Agent Loop 的入口。

LangChain Agent 的核心是它把模型、工具和消息组织到一个循环中：模型根据当前 messages 和可用 tools 判断下一步，如果需要行动，就生成 tool call；Runtime 执行对应工具，并把 tool result 重新放回 messages；模型再基于新的观察结果继续判断，直到输出最终答案。[^langchain-agents][^langchain-context-engineering]

可以表示为：
```text
User Message
  -> Model Node
  -> Tool Call
  -> Tools Node
  -> Tool Result
  -> Model Node
  -> Final Answer
```

作为对比，普通 function calling 更像是一次模型调用中的结构化输出能力：模型判断是否调用某个函数，应用侧执行函数，再把结果返回给模型。LangChain Agent 则把这条路径框架化成一个可重复执行的 agent loop。模型不是只在单轮回答中补充一次外部信息，而是在 Runtime 的驱动下反复观察、调用工具、接收结果，并继续推进任务。

LangChain Agent 的最小结构可以拆成四个部分：
| 组成       | 职责                                  |
| -------- | ----------------------------------- |
| Model    | 根据当前上下文判断下一步，是直接回答，还是请求工具调用         |
| Tools    | Runtime 暴露给模型的可执行动作                 |
| Messages | 承载用户输入、模型输出、tool call 和 tool result |
| Runtime  | 调度模型调用、工具执行和循环终止                    |

> [!NOTE]
> 这里的 Messages 需要特别注意。
> 
> 在传统 Chatbot 中，messages 通常只是对话历史；在 Agent Loop 中，messages 还记录工具调用和工具结果。也就是说，Agent 的消息流不再只是“用户说了什么、模型答了什么”，还包括模型曾经请求了什么动作，以及 Runtime 执行后观察到了什么结果。
>
> 在这个过程中，模型看到的不只是用户最初的问题，还包括每一次工具调用产生的观察结果。工具结果成为下一轮推理的输入，Agent Loop 由此形成闭环。
> 
> 可以概括为：
> ```
> 模型提出行动意图；
> Runtime 执行动作；
> 工具结果变成新的上下文；
> 模型继续判断下一步。
> ```

LangChain 的 create_agent 把这套循环封装成一个基于 LangGraph 的 graph-based agent runtime。官方文档中提到，Agent 会在图中的不同节点之间移动，例如调用模型的 model node、执行工具的 tools node，以及介入执行过程的 middleware。[^langchain-agents]

我们可以来简单组织其运行过程：
```text
model node:
  调用 LLM
  读取 messages 和 tools
  返回 final answer 或 tool calls

tools node:
  接收 tool calls
  调用对应工具
  生成 tool results
  将结果写回 messages

middleware:
  在模型调用、工具调用或状态流转前后插入额外逻辑
```

这里的关键是，工具调用不直接等于行动。模型只是生成了一个结构化的 tool call，真正执行动作的是 Runtime。

这和上一篇提到的 Agent Runtime 分层是连贯的：模型负责提出行动意图，Runtime 负责验证、执行、记录和反馈行动结果。只是在 LangChain Agent 这一层，权限、沙箱、文件系统、长期记忆等能力还没有完全展开，最核心的机制仍然是 model node 与 tools node 之间的循环。[^Agent-Runtime]

LangChain Agent 可以被看作基础 Agent Runtime 的最小实现形态：
```text
已经实现了多次循环模型调用；
已经具备工具调用循环；
已经有 Runtime 来调度模型和工具；
但还没有完整解决复杂任务的长期记忆、计划、上下文、中间产物和安全边界问题。
```
因此，LangChain Agent 更适合作为这篇文章的前半部分。它告诉我们一个基础 Agent Loop 如何成立；接下来需要看的 LangGraph Runtime，则进一步回答：这个循环如何拥有运行时状态、依赖注入、流式输出、持久化和恢复能力。

## 2. LangGraph Runtime：让 Agent Loop 拥有运行时状态
上一节讨论的是 LangChain Agent 的基础工具调用循环。

这个循环可以让模型在 messages 和 tools 之间持续行动，但是它无法解决：
```text
这个循环如何保存状态？
工具如何拿到运行时依赖？
流式输出如何写出？
中断后如何恢复？
HITL 如何暂停和继续？
长期记忆如何接入？
```
此时需要引入 LangGraph Runtime。

官方文档明确提到，LangChain agents 构建在 LangGraph 之上，以获得 durable execution、streaming、human-in-the-loop、persistence 等能力。[^langchain-reference]

即，LangChain Agent 提供的是基础的 Agent 抽象，而 LangGraph 提供的是更底层的 orchestration runtime。

做一个简单总结：
```text
LangChain Agent:
  提供模型、工具、消息和中间件组成的基础 Agent Loop

LangGraph Runtime:
  提供状态、流式输出、持久化、中断恢复和人类介入等运行时能力
```

LangGraph 的官方定位也比较清楚：LangChain 是 agent framework，负责模型、工具和 agent loop 等抽象；LangGraph 是 orchestration runtime，负责 durable execution、streaming、human-in-the-loop 和 persistence；DeepAgents 则是在 LangGraph 之上提供 planning、subagents、filesystem tools 和 context management 的 agent harness。[^langgraph-overview]

> [!NOTE]这个分层对理解现代 Agent 系统很关键。
> ```text
> LangChain:
>   面向开发者的 Agent 框架
> LangGraph:
>   面向执行过程的运行时底座
> DeepAgents:
>   面向复杂任务的 Harness
> ```

### 2.1 Agent Loop 的运行时底座
如果只看上一章的工具调用循环，Agent 似乎只是一个不断重复的过程：
```text
调用模型
  -> 解析 tool call
  -> 执行工具
  -> 写回 tool result
  -> 再次调用模型
```
但真实 Agent 系统并不能只靠一个内存中的循环来支撑。

原因在于，Agent 的执行过程往往具有几个特点：
```text
执行时间更长
中间步骤更多
工具调用更多
用户可能中途介入
前端连接可能断开
模型或工具可能失败
任务状态需要恢复
执行过程需要被观察
```
一旦出现这些问题，单纯的 Agent Loop 就不够了。它需要知道当前运行到哪里，如何保存中间状态，如何把流式事件推给前端，如何在用户审批后继续执行，以及如何在失败或中断后恢复。

所以，LangGraph Runtime 的价值并不只是“用图来组织流程”。更关键的是，它把 Agent Loop 放进一个可持续运行、可中断、可恢复、可观察的执行环境中。

可以理解为：
```text
Agent Loop:
  描述模型和工具如何循环交互

LangGraph Runtime:
  描述这个循环如何被执行、保存、观察和恢复
```

### 2.2 Runtime 提供了什么
如果进一步拆开，LangGraph / LangChain Runtime 至少提供了几类基础能力。

其中，`context`、`store`、`stream_writer` 和 `execution_info` 来自 LangChain Runtime 对象。官方文档中提到，LangChain 的 `create_agent` 底层运行在 LangGraph runtime 上，Runtime 对象可以为 tools 和 middleware 提供这些运行期信息：`context` 可用于传入 user id、数据库连接等依赖；`store` 用于长期记忆；`stream_writer` 用于通过 `custom` stream mode 输出自定义流式信息；`execution_info` 则提供 thread ID、run ID 和 attempt number 等执行身份。[^langchain-runtime]

Durable Execution 和 Interrupt 则来自 LangGraph 更底层的持久化与中断机制。LangGraph durable execution 文档说明，workflow 可以在关键点保存进度，并在系统失败或 HITL 中断后，从最后记录的状态继续执行。LangGraph interrupts 文档则说明，interrupts 可以在特定位置暂停 graph execution，等待外部输入后再继续；当 interrupt 触发时，LangGraph 会通过 persistence layer 保存 graph state，并等待 resume。[^langgraph-durable][^langgraph-interrupts]

可以粗略整理为：

| 能力 | 含义 | 主要出处 |
| --- | --- | --- |
| Runtime Context | 为工具和中间件提供运行时依赖，例如 user_id、数据库连接、API client 等 | LangChain Runtime |
| Store | 提供长期记忆或跨运行状态的保存接口 | LangChain Runtime |
| Stream Writer | 允许工具或中间件通过 `custom` stream mode 输出自定义流式事件 | LangChain Runtime |
| Execution Info | 提供 thread ID、run ID、attempt number 等执行身份 | LangChain Runtime |
| Durable Execution | 在关键点保存进度，使任务能够暂停和恢复 | LangGraph Durable Execution |
| HITL / Interrupt | 在需要人工审批或外部输入时暂停执行，并在用户决策后继续 | LangGraph Interrupts |

- 这里最容易混淆的是 Runtime Context 和传入 LLM 的 Model Context。
  ```text
  Model Context:
    给模型看的上下文
    例如 messages、system prompt、tool schema、summary

  Runtime Context:
    给工具和 Runtime 用的上下文
    例如 user_id、db connection、permission scope、API client
  ```
  这个区分很重要。模型不一定需要知道数据库连接对象，也不应该直接接触某些运行时凭据；但工具执行时确实需要这些依赖。Runtime Context 的作用，就是把执行依赖以受控方式提供给工具和中间件，而不是把它们硬编码在工具内部。

- Store、Durable Execution 和 Interrupt 共同解决状态与恢复问题。
  
  Store 更偏向长期保存，例如用户偏好、项目背景、跨会话记忆和任务摘要；Durable Execution 更偏向运行过程保存，例如在模型调用、工具调用、人类审批等关键点保存进度；Interrupt 则让 Runtime 能够在需要外部输入时暂停执行，并在用户决策后继续。

  可以简单理解为：
  ```text
  Store:
    保存长期信息

  Durable Execution:
    保存运行进度

  Interrupt:
    暂停执行并等待外部输入
  ```

- Stream Writer 和 Execution Info 则让 Agent 的执行过程更容易被观察和追踪。

  在普通 Chatbot 中，streaming 更多是打字机效果。但在 Agent 中，streaming 还可以展示当前步骤、工具调用、命令输出、文件读写和用户确认点。Execution Info 则让每一次执行都有明确身份：这是哪个 thread、哪次 run、第几次 attempt。这些信息会进一步支撑日志、恢复、权限判断和多租户隔离。

  所以，Runtime 提供的并不是某一个单独功能，而是一组让 Agent Loop 变成工程系统的基础能力：
  ```text
  依赖如何传入？
  状态如何保存？
  执行如何暂停？
  中断如何恢复？
  事件如何流出？
  运行如何追踪？
  ```
这些问题，在简单的 Chatbot 中并不突出；但一旦 Agent 开始执行长任务、调用外部工具、等待用户审批，它们就会变成 Runtime 必须处理的核心问题。

### 2.3 小结：LangGraph 是 DeepAgents 的底层基础
```text
LangChain Agent:
  提供基础工具调用循环

LangGraph Runtime:
  提供运行时状态、持久化、流式输出、中断恢复和人工介入能力

DeepAgents:
  在此基础上封装复杂任务 Harness
```
LangGraph 更像是连接 LangChain Agent 和 DeepAgents 的中间层。

没有 LangGraph，Agent Loop 可以成立，但很容易停留在内存中的短程执行；有了 LangGraph，Agent Loop 才开始具备长任务所需的运行时属性。

也就是说，LangGraph 让基础 Agent Loop 具备了成为复杂 Agent Runtime 的条件。

但它仍然主要提供底层运行能力。至于复杂任务中的计划管理、中间产物、文件系统、上下文压缩、子任务隔离、沙箱和技能加载，仍然需要更上层的 Harness 来组织。

这就进入了 DeepAgents 的范围。

## 3. 基础 Agent Loop 的边界

前两节整理了 LangChain Agent 和 LangGraph Runtime 的关系。

LangChain Agent 提供基础工具调用循环：模型读取 messages 和 tools，判断下一步是否需要调用工具；Runtime 执行工具，并把 tool result 写回 messages；模型再根据新的观察结果继续判断。LangChain 官方文档也明确说明，create_agent 构建的是基于 LangGraph 的 graph-based agent runtime，Agent 会在 model node、tools node、middleware 等节点之间移动。[^langchain-agents]

LangGraph 则提供更底层的运行时能力。它把 Agent Loop 放进一个可以持久化、流式输出、中断、恢复和接入 human-in-the-loop 的 orchestration runtime 中。LangGraph 官方概览也将它定位为 orchestration runtime，强调 durable execution、streaming、human-in-the-loop 和 persistence 等能力。[^langgraph-overview]

到这里为止，我们已经有了两层基础：

```text
LangChain Agent:
  让模型能够进入工具调用循环

LangGraph Runtime:
  让这个循环具备运行时状态、持久化、中断恢复和流式输出能力
```

但是这在应付复杂任务时仍显捉襟见肘：
- 一个简单 Agent 只需要围绕当前 messages 和 tools 做几轮循环即可。例如查询资料、调用一个 API、读取少量文件、生成一段回答。这类任务的主要难点在于模型是否选对工具、工具执行是否成功、最终回答是否正确。
- 复杂任务则不同。它往往会持续更久，跨越更多步骤，产生更多中间产物，并且需要在执行过程中不断调整策略。比如：
  ```text
  调研一个技术主题并写成长文；
  阅读一个代码库并提出重构方案；
  修复一个复杂 bug 并运行测试；
  生成一份报告并根据反馈多轮修改；
  拆分任务给不同子 Agent 执行；
  在文件系统中持续保存草稿、摘要和结果。
  ```
  这时，基础 Agent Loop 会遇到几个明显边界：

  - 第一，计划不能只存在于模型上下文中。

    在短任务中，模型可以临时判断下一步。但长任务通常需要一个持续更新的计划：当前目标是什么，已经完成哪些步骤，下一步要做什么，哪些问题还没有解决，哪些发现已经被验证。若这些计划只停留在模型上下文里，一旦上下文被压缩、连接中断、任务暂停或运行恢复，计划就可能丢失或变形。

    这也是为什么复杂 Agent(`Claude Code / Codex`) 往往需要把计划外部化为可保存、可展示、可更新的 Task State。DeepAgents 内置的 `write_todos` 就服务于这个目标：让 Agent 把复杂任务拆成离散步骤，追踪进度，并根据新信息调整计划。[^deepagents-overview]

  - 第二，messages 不适合作为所有中间产物的唯一容器。

    在基础 Agent Loop 中，messages 承载用户输入、模型输出、tool call 和 tool result。这个设计足以支撑工具调用循环，但复杂任务会产生大量工作产物，比如：

    ```text
    阶段性摘要
    调研笔记
    文件草稿
    代码修改方案
    测试输出
    网页摘录
    数据分析结果
    最终报告的多个版本
    ```
    如果这些内容全部进入 messages，模型上下文会迅速膨胀，噪音也会增加。更合理的方式是把中间产物保存到外部工作区中，再由模型按需读取。DeepAgents 的 filesystem 能力正是为此服务：它让 Agent 拥有一个可读、可写、可持久化甚至是可替换 backend 的工作空间。[^deepagents-overview]

  - 第三，上下文不能只靠 _不断追加历史_ 来管理。

    长任务的关键不是让模型看到全部内容，而是让模型在当前步骤看到最有用的信息。复杂 Agent 需要判断：
    ```text
    哪些历史消息需要保留？
    哪些工具结果需要摘要？
    哪些文件只需要保留路径？
    哪些中间产物需要按需读取？
    哪些用户偏好需要从 memory 中注入？
    哪些工具应该在当前步骤暴露？
    哪些执行依赖只应该留在 Runtime Context 中？
    ```
    这已经进入 Context Engineering 的范围。DeepAgents 文档也将 context engineering 作为核心能力之一，讨论 input context、runtime context、context compression、context isolation 和 long-term memory 等问题。[^deepagents-context]

  - 第四，复杂子任务需要上下文隔离。

    如果主 Agent 亲自执行所有子任务，它的上下文会不断吸收各种细节：搜索过程、失败尝试、文件片段、临时假设、工具输出和局部结论。时间一长，主上下文会变得非常拥挤，模型也更容易被不相关信息干扰。

    因此，复杂 Agent 需要一种子任务委派机制。主 Agent 可以把某个局部任务交给 subagent，让 subagent 在自己的上下文中深入执行，再把压缩后的结论返回主 Agent。DeepAgents 内置的 task tool 就用于 subagent spawning，其重点之一是 context isolation：让主 Agent 的上下文保持干净，同时允许子任务深入展开。[^deepagents-overview]

  - 第五，工具行动需要更明确的持久化边界和安全边界。

    当 Agent 只是调用搜索工具或普通 API 时，风险相对有限。但一旦它开始读写文件、运行 Shell、访问外部网络、执行代码、写入长期记忆，Runtime 就必须回答更多工程问题：

    ```text
    文件保存在哪里？
    哪些文件可以跨线程复用？
    哪些内容只保存在当前任务中？
    Shell 命令在哪里执行？
    是否允许访问宿主机文件？
    是否允许访问网络？
    是否需要用户确认？
    执行失败后如何恢复？
    ```

    LangGraph 提供 durable execution、persistence 和 interrupt 等底层能力，但复杂任务仍然需要更上层的组织方式：文件系统 backend 如何选择，sandbox 如何接入，哪些工具需要 HITL，哪些 memory 可以跨任务保存。DeepAgents 的 pluggable filesystem backends、sandbox backend、long-term memory 和 HITL 机制，正是围绕这些问题展开。[^deepagents-overview][^deepagents-hitl]

所以，基础 Agent Loop 的边界并不在于“模型能不能继续调用工具”。真正的边界在于：
```
计划如何被保存？
产物如何被管理？
上下文如何被压缩？
子任务如何被隔离？
工具如何被约束？
状态如何跨运行恢复？
用户如何在关键节点介入？
```

这也解释了为什么 DeepAgents 的定位会是一个 agent harness。它没有替换掉基础 tool-calling loop，而是在这个基础之上，增加复杂任务所需的执行环境以及执行规则：
```text
Agent Loop:
  让模型能够行动

Runtime:
  让行动能够被保存、恢复和观察

Harness:
  让复杂任务能够被规划、拆分、隔离、持久化和安全执行
```
从这一点看，DeepAgents 的意义不只是提供更多工具。它更像是在基础 Agent Runtime 之上，进一步补齐复杂任务执行所需的任务状态、文件系统、上下文管理、子任务隔离、持久化 backend、sandbox、memory 和 HITL。

## 4. DeepAgents：复杂任务的 Agent Harness

上一节讨论了基础 Agent Loop 的边界。

LangChain Agent 已经让模型进入工具调用循环，LangGraph Runtime 也让这个循环具备状态、持久化、流式输出、中断恢复和 human-in-the-loop 等运行时能力。但对于复杂任务来说，只有 Loop 和 Runtime 仍然不够。因为复杂任务需要的不只是“能调用工具”，还包括一套能够组织任务、保存产物、隔离上下文、委派子任务、约束执行环境的上层结构。

这正是 DeepAgents 负责的功能。

官方文档把 DeepAgents 称为一个 agent harness。它使用的仍然是和其他 Agent 框架相同的核心 tool-calling loop，但额外内置了 planning、filesystem、subagent spawning、long-term memory、pluggable filesystem backends 等能力。[^deepagents-overview]

这里的关键点是，DeepAgents 没有推翻基础 Agent Loop。它的意义在于：在同一套 tool-calling loop 之上，预装复杂任务需要的运行设施。

可以这样理解：
```text
基础 Agent Loop:
  Model + Tools + Messages + Runtime

DeepAgents Harness:
  Model + Tools + Messages + Runtime
  + Planning
  + Filesystem
  + Context Engineering
  + Subagents
  + Backends
  + Sandbox
  + Memory
  + Skills
  + HITL
```
LangChain Agent 解决的是“模型如何行动”，LangGraph Runtime 解决的是“行动过程如何保存、恢复和观察”，DeepAgents 进一步解决的是：
```text
复杂任务如何被规划？
中间产物如何被保存？
上下文如何被压缩和隔离？
子任务如何被委派？
文件系统如何持久化？
Shell 和代码执行如何隔离？
长期信息如何跨任务复用？
专门能力如何按需加载？
高风险动作如何交给用户确认？
```

因此，DeepAgents 更适合被看作一个面向复杂任务的工作台。它把一组复杂 Agent 常见的工程能力封装成默认 Harness，使开发者不必从零开始搭建 planning、filesystem、subagents、context management 和 memory 等基础设施。LangChain 官方对 Deep Agents 的产品介绍也将其描述为面向 long-running tasks 的 open source agent harness，强调 planning、context management 和 multi-agent orchestration 等能力。[^deepagents-product]

### 4.1 Harness——任务执行结构的革新
如果只从表面看，DeepAgents 似乎只是比普通 Agent 多了一些工具：
```text
write_todos
ls
read_file
write_file
edit_file
task
execute
memory tools
skill tools
```
但这类理解仍然偏浅。DeepAgents 的核心不只是多了几个工具，而是通过这些工具共同组成了一个复杂任务执行结构。

以 `write_todos` 为例，它表面上只是让 Agent 写任务列表；但系统含义是把模型的隐式计划外部化为 Task State。Harness 文档中也明确说明，`write_todos` 可以维护结构化任务列表，追踪 `pending`、`in_progress`、`completed` 等状态，并持久化在 agent state 中。[^deepagents-harness]

再看 filesystem。文件工具表面上只是 `ls`、`read_file`、`write_file`、`edit_file` 等操作；但系统含义是为 Agent 提供一个外部工作区。Agent 可以把长内容、中间结果、草稿、报告和代码改动保存到文件系统中，而不是全部塞进 messages。DeepAgents overview 也提到，文件系统工具可以帮助 Agent 将大上下文卸载到内存或文件系统存储中，避免 context window overflow，并处理长度不固定的工具结果。[^deepagents-overview]

Subagent 也是类似。`task` 工具表面上是创建子 Agent；系统含义则是 context isolation。主 Agent 可以把局部任务交给 subagent，让 subagent 在隔离上下文中深入执行，然后只把压缩后的结论返回主 Agent。官方文档也明确提到，内置 `task tool` 可以生成 specialized subagents，用于 context isolation，使 main agent 的 context 保持干净。[^deepagents-overview]

综上所述，DeepAgents 的机制最好不要写成工具列表，而应该写成一组系统结构：

| DeepAgents 机制            | 系统含义                              |
| ------------------------ | --------------------------------- |
| Planning / `write_todos` | 把隐式任务计划转化为可见、可保存、可更新的 Task State  |
| Filesystem               | 为 Agent 提供外部工作区，用于保存草稿、报告、代码和中间结果 |
| Context Engineering      | 决定哪些信息进入当前模型调用，哪些信息保留在外部工作区       |
| Subagents                | 通过子任务委派实现上下文隔离，减少主 Agent 上下文污染    |
| Backends                 | 决定文件系统和记忆落在哪里，形成持久化边界             |
| Sandbox                  | 限制文件操作和 Shell 执行的影响范围             |
| Memory                   | 保存跨线程、跨任务的长期信息                    |
| Skills                   | 按需加载专门任务说明和操作规范                   |
| HITL                     | 在敏感工具调用前暂停执行，等待用户审批               |

这张表也说明，DeepAgents 的设计目标并不是把 Agent Loop 变复杂，而是把复杂任务中反复出现的工程问题抽象成默认能力。

### 4.2 DeepAgents 与 LangGraph
DeepAgents 构建在 LangGraph 之上。

这一点很重要。因为 DeepAgents 的 planning、filesystem、subagents、memory 和 HITL，都需要依赖底层 Runtime 提供的状态管理、持久化、中断、恢复和流式输出能力。

LangGraph 官方文档将 LangGraph 定位为用于构建 long-running、stateful agents 的 low-level orchestration framework，强调 durable execution、streaming、human-in-the-loop、persistence 等基础能力。[^langgraph-overview] DeepAgents 则站在更上层，提供面向复杂任务的默认组织方式。

可以粗略分层为：
```text
LangChain:
  提供模型、工具、消息和 agent loop 抽象

LangGraph:
  提供 stateful orchestration runtime
  负责 durable execution、persistence、streaming、interrupt、HITL 等

DeepAgents:
  在 LangGraph 之上提供 agent harness
  负责 planning、filesystem、subagents、context management、memory、sandbox 等
```
可以看到，基础 Loop 已经由 LangChain / LangGraph 提供；DeepAgents 要做的是把长任务所需的任务状态、工作区、上下文隔离和持久化策略组织起来。


到这里，这篇文章的主线可以先收住。

LangChain Agent 说明了基础工具调用循环如何成立；LangGraph Runtime 说明了这个循环如何具备状态、持久化、流式输出、中断恢复和人工介入能力；DeepAgents 则进一步说明，复杂任务并不只是需要更多工具，而是需要一套围绕 planning、filesystem、context management、subagents、memory、sandbox 和 HITL 组织起来的 Harness。

也就是说，这一篇主要回答的是：
```text
一个基础 Agent Loop 如何逐步长成复杂任务型 Agent 的运行结构？
```

下一篇再继续拆开 DeepAgents Harness 的具体机制：任务计划如何外部化，中间产物如何进入文件系统，上下文如何压缩和隔离，子 Agent 如何承担局部任务，以及权限、沙箱、Memory、Skills 和 HITL 如何共同构成执行边界。
